\chapter{ケーススタディ：API adapter と自然性}
\label{chap:ch31-api-adapter-naturality}
\BookIndex{API adapter}{API adapter}
\BookIndex{しぜんせい}{自然性}
\BookIndex{かかんずしき}{可換図式}

\begin{chapterabstract}
本章では、API レスポンス形式の変更を題材に、adapter の正しさを自然性として表します。
旧形式のレスポンスに業務ロジックを適用してから新形式へ変換しても、先に新形式へ変換してから業務ロジックを適用しても、結果が一致します。
この性質を、自然性の四角形と Lean 4 の等式として確認します。
本章の狙いは、adapter を「変換関数があるから安全」と見るのではなく、「業務ロジックと干渉しないことを証明できる層」として扱うことです。
\end{chapterabstract}

\begin{learninggoals}
\item API レスポンス wrapper の変更を、型に依存しない変換として整理できます。
\item 自然性の四角形を、adapter と \texttt{map} の順序交換として読めます。
\item \texttt{OldResponse A} と \texttt{NewResponse A} の小さな Lean モデルを書けます。
\item adapter を通す位置を変えても結果が同じであることを、Lean の定理として表せます。
\item 実務で自然性を使うべき範囲と、使ってはいけない範囲を区別できます。
\end{learninggoals}

\section{問題設定}

API のバージョンアップでは、レスポンスの中身そのものは同じでも、包み方が変わることがあります。
たとえば、旧 API では次のような形で返していたとします。

\begin{itemize}
\item ステータスコードを \texttt{status} に入れます。
\item 実データを \texttt{body} に入れます。
\item トレース用の ID を \texttt{requestId} に入れます。
\end{itemize}

新 API では、名前や追加メタデータを変えて、次のような形にします。

\begin{itemize}
\item ステータスコードを \texttt{code} に入れます。
\item 実データを \texttt{data} に入れます。
\item \texttt{requestId} は引き継ぎます。
\item キャッシュ由来かどうかを \texttt{cached} として追加します。
\end{itemize}

このとき、旧レスポンスを新レスポンスへ変換する adapter を書くこと自体は難しくありません。
しかし、本章で確認したいのは「変換関数が存在するか」ではありません。
本当に確認したいのは、adapter がレスポンスの中の業務データに余計な影響を与えないかという点です。

図\ref{fig:ch31-response-shape-change}は、payload を保ったままフィールド名を変え、定数の metadata を追加する adapter を示します。

\section{\texttt{OldResponse A} と \texttt{NewResponse A}}

まず、payload の型を \texttt{A} として抽象化します。
\texttt{A} は、ユーザーでも注文でも商品でも構いません。
重要なのは、レスポンス wrapper が payload の型に対して一様に振る舞うことです。

\begin{definitionbox}
\texttt{OldResponse A} は、型 \texttt{A} の値を旧形式で包むレスポンス型です。
\texttt{NewResponse A} は、同じ型 \texttt{A} の値を新形式で包むレスポンス型です。
ここで \texttt{A} を固定しないのは、adapter が特定の payload 型だけでなく、任意の payload 型に対して同じ規則で動くかを確認するためです。
\end{definitionbox}

Lean では、次のように最小モデルを作ります。
ここでは HTTP の詳細、JSON エンコード、エラー詳細、認証情報などは扱いません。
自然性を確認するために必要な情報だけを残します。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure OldResponse (A : Type) where
  status : Nat
  body : Option A
  requestId : String

structure NewResponse (A : Type) where
  code : Nat
  data : Option A
  requestId : String
  cached : Bool
\end{minted}
\caption{\texttt{OldResponse} と \texttt{NewResponse}}
\label{lst:ch31-response-types}
\end{listing}

\texttt{OldResponse} と \texttt{NewResponse} は、どちらも \texttt{Type -> Type} という形を持ちます。
つまり、payload の型 \texttt{A} を受け取ると、レスポンス型を返す型レベルの関数として読めます。
このような形は、圏論では関手を考える入口になります。

\subsection{payloadだけを変換する \texttt{map}}

業務ロジックは、多くの場合、レスポンス全体ではなく payload に対して働きます。
たとえば、内部ユーザー型を公開用ユーザー型へ変換する関数は、レスポンス metadata ではなく \texttt{body} や \texttt{data} の中身に作用します。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace OldResponse

def map {A B : Type} (f : A -> B)
    (r : OldResponse A) : OldResponse B :=
  { status := r.status,
    body := Option.map f r.body,
    requestId := r.requestId }

end OldResponse

namespace NewResponse

def map {A B : Type} (f : A -> B)
    (r : NewResponse A) : NewResponse B :=
  { code := r.code,
    data := Option.map f r.data,
    requestId := r.requestId,
    cached := r.cached }

end NewResponse
\end{minted}
\caption{\texttt{OldResponse.map} と \texttt{NewResponse.map}}
\label{lst:ch31-response-map}
\end{listing}

この \texttt{map} は、metadata をそのまま保ち、payload だけに \texttt{f : A -> B} を適用します。
旧レスポンスでも新レスポンスでも、payload が存在しない \texttt{none} の場合は、存在しないままにします。

\texttt{Option.map f r.body} は、値があれば \texttt{f} を適用し、\texttt{none} ならそのままにします。

標準的な圏論の自然変換と呼ぶには、\texttt{OldResponse} と \texttt{NewResponse} の各 \texttt{map} が恒等関数と合成を保存し、関手を成すことも必要です。
本章のレコード定義ではそれらも証明できますが、コードでは自然性等式だけを取り出しており、関手構造までは一つの型にまとめていません。

\section{adapter が業務ロジックと可換であること}

次に、旧レスポンスから新レスポンスへの adapter を定義します。
この adapter は、\texttt{status} を \texttt{code} へ、\texttt{body} を \texttt{data} へ写し、\texttt{requestId} を保存します。
新しく追加された \texttt{cached} には、ここでは既定値 \texttt{false} を入れます。

\begin{listing}[htbp]
\begin{minted}{lean4}
def adapt {A : Type} (r : OldResponse A) : NewResponse A :=
  { code := r.status,
    data := r.body,
    requestId := r.requestId,
    cached := false }
\end{minted}
\caption{\texttt{adapt}}
\label{lst:ch31-adapter}
\end{listing}

この \texttt{adapt} は、\texttt{A} の中身を見ていません。
ユーザー ID が偶数かどうか、注文金額が高いかどうか、文字列が空かどうかといった payload 固有の判断はしません。
これは自然性を期待する理由にはなりますが、正式な判定は、次の等式をすべての型、関数、レスポンスについて証明することです。

\begin{definitionbox}
本章でいう「可換」とは、二つの処理の順序を入れ替えても最終結果が同じであるという意味です。
ここでは、次の二つの経路が同じであることを示します。
\begin{align*}
&\text{経路1：旧レスポンス内で payload を変換してから、新レスポンスへ adapt します。}\\
&\text{経路2：先に新レスポンスへ adapt してから、新レスポンス内で payload を変換します。}
\end{align*}
\end{definitionbox}

payload 変換を \(f : A \to B\)、旧レスポンスを \(r : \texttt{OldResponse}\ A\) とすると、証明したい等式は次のとおりです。

\[
\mathsf{adapt}\,(\mathsf{OldResponse.map}\ f\ r)
=
\mathsf{NewResponse.map}\ f\,(\mathsf{adapt}\ r).
\]

左辺は「先に旧形式の中で業務ロジックを適用し、その後 adapter を通す」経路です。
右辺は「先に adapter を通し、その後新形式の中で業務ロジックを適用する」経路です。

本章では、この四角形を API adapter の検証図として読みます。

左右の辺は、payload 型 \texttt{A} と \texttt{B} における adapter です。

\subsection{自然性が言っていること}

自然性は、単に「あるサンプル入力で同じ結果になる」という主張ではありません。
任意の payload 型 \texttt{A}、任意の変換先型 \texttt{B}、任意の業務ロジック \(f : A \to B\)、任意の旧レスポンス \(r\) について、同じ等式が成り立つという主張です。

\subsection{自然性が言っていないこと}

自然性は、adapter が現実の HTTP 通信や JSON パーサのすべての振る舞いを保証するわけではありません。
本章の証明が扱うのは、型で表したレスポンス wrapper と payload 変換の関係です。

\begin{warningbox}
自然性を証明しても、外部 API のステータスコード仕様、JSON フィールド名の実際の綴り、ネットワーク障害、認証、タイムアウト、ログ出力、副作用の順序までは保証されません。
Lean のモデルが保証するのは、モデルに含めた関数とデータ構造についての性質です。
実装コードとの対応は、別途レビューやテストで確認する必要があります。
\end{warningbox}

\section{Lean で自然性を証明する}

証明方針は単純です。
旧レスポンス \texttt{r} を構造体の各フィールドに分解します。
すると、左辺と右辺はどちらも同じ \texttt{NewResponse} の値になります。
したがって、Lean では \texttt{rfl} で証明できます。

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem adapt_natural {A B : Type} (f : A -> B)
    (r : OldResponse A) :
    adapt (OldResponse.map f r) =
      NewResponse.map f (adapt r) := by
  cases r
  rfl
\end{minted}
\caption{\texttt{adapt\_natural}}
\label{lst:ch31-adapt-natural}
\end{listing}

この証明は短いですが、主張は強力です。
\texttt{cases r} によって、\texttt{r} は \texttt{status}、\texttt{body}、\texttt{requestId} というフィールドへ分解されます。
分解後、左辺と右辺は同じ構造体リテラルに計算されます。
そのため、\texttt{rfl} が使えます。

\subsection{二つの経路に名前を付ける}

実務の説明では、自然性の等式をそのまま見るより、二つの経路に名前を付けたほうが読みやすくなります。
ここでは、先に業務ロジックを適用する経路を \texttt{routeBefore}、先に adapter を通す経路を \texttt{routeAfter} と呼びます。

\begin{listing}[htbp]
\begin{minted}{lean4}
def routeBefore {A B : Type} (f : A -> B)
    (r : OldResponse A) : NewResponse B :=
  adapt (OldResponse.map f r)

def routeAfter {A B : Type} (f : A -> B)
    (r : OldResponse A) : NewResponse B :=
  NewResponse.map f (adapt r)

theorem route_position_independent {A B : Type}
    (f : A -> B) (r : OldResponse A) :
    routeBefore f r = routeAfter f r := by
  exact adapt_natural f r
\end{minted}
\caption{\texttt{routeBefore} と \texttt{routeAfter}}
\label{lst:ch31-routes}
\end{listing}

\texttt{route\_position\_independent} は、実務向けには「adapter の位置独立性」と読めます。
この定理があると、リファクタリングで adapter の位置を変更しても、payload だけを見る業務ロジックについては結果が変わらないと説明できます。

\subsection{自然な adapter を構造として持つ}

adapter と自然性証明を一緒に持つ小さな構造を作ることもできます。
この構造は、Mathlib の \texttt{NatTrans} を直接使うものではありません。
本章の目的に合わせて、\texttt{OldResponse} と \texttt{NewResponse} の間だけに絞った簡易版です。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure ResponseAdapter where
  app : {A : Type} -> OldResponse A -> NewResponse A
  naturality : {A B : Type} -> (f : A -> B) ->
    (r : OldResponse A) ->
    app (OldResponse.map f r) = NewResponse.map f (app r)

def oldToNewAdapter : ResponseAdapter where
  app := fun {A} r => adapt (A := A) r
  naturality := by
    intro A B f r
    exact adapt_natural f r
\end{minted}
\caption{\texttt{ResponseAdapter}}
\label{lst:ch31-response-adapter}
\end{listing}

このようにまとめると、adapter を単なる関数ではなく「可換性の証明を持った互換層」として扱えます。
仕様レビューでは、adapter の本体だけでなく、何と可換でなければならないかを明確にできます。

\section{実務での意味：adapter の位置を変えても結果が同じ}

ここまでの定理は、実務では次のように使えます。

\begin{itemize}
\item 旧 API レスポンスに対して既存の業務ロジックを適用してから、新 API レスポンスへ変換します。
\item 先に新 API レスポンスへ変換し、新形式の上で同じ業務ロジックを適用します。
\item この二つの経路が一致することを、Lean で確認します。
\end{itemize}

たとえば、内部ユーザー型 \texttt{UserV1} を公開用の \texttt{UserView} へ変換する処理を考えます。
この処理は payload の中身だけを見て、レスポンス metadata には触れません。
そのような処理に対して、adapter は前後どちらに置いても同じ結果になります。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure UserV1 where
  id : Nat
  name : String

structure UserView where
  id : Nat

def toUserView (u : UserV1) : UserView :=
  { id := u.id }

theorem user_view_adapter_position_independent
    (r : OldResponse UserV1) :
    routeBefore toUserView r = routeAfter toUserView r := by
  exact route_position_independent toUserView r
\end{minted}
\caption{\texttt{user\_view\_adapter\_position\_independent}}
\label{lst:ch31-user-view}
\end{listing}

この証明では、\texttt{UserV1} や \texttt{UserView} に固有の難しい議論はしていません。
すでに任意の \texttt{A} と \texttt{B} について証明した自然性を、ユーザー変換に適用しているだけです。
これが、抽象化の利点です。

自然性を実務に持ち込むときは、次の問いが有効です。

\subsection{自然性が壊れる典型例}

次のような adapter には、自然性を期待しにくくなります。

\begin{itemize}
\item payload の値に応じて \texttt{code} を変更します。
\item payload の型ごとに別のフィールド変換を行います。
\item \texttt{String} payload だけを特別扱いして整形します。
\item adapter の中で DB 参照や外部 API 呼び出しを行います。
\item 変換前後で payload の欠損を補完するため、型 \texttt{A} の既定値を注入します。
\end{itemize}

これらが、必ず悪いわけではありません。
ただし、その場合は「自然な adapter」としてではなく、payload に依存する業務変換として扱うべきです。
証明すべき性質も、自然性ではなく、事前条件、事後条件、仕様弱化、または片方向互換性になります。

\begin{warningbox}
実務では、adapter と business logic が混ざりやすくなります。
自然性を証明したいなら、adapter はできるだけ「包みの変換」に限定します。
payload の意味を変更する処理は、別の関数として切り出し、\texttt{map} で扱うほうが設計を明確にできます。
\end{warningbox}

\section{本章のまとめ}

API adapter は、旧レスポンス wrapper から新レスポンス wrapper への変換としてモデル化できます。

\texttt{OldResponse.map} と \texttt{NewResponse.map} は、metadata を保ち、payload だけに業務ロジックを適用します。

自然性は、\texttt{adapt (OldResponse.map f r) = NewResponse.map f (adapt r)} という等式で表せます。

Lean では、構造体を分解し、定義を展開することで、adapter と \texttt{map} の可換性を証明できます。

この証明は、モデル化した \texttt{map} で持ち上げる payload 変換について、adapter の位置を変えても結果が同じであることを示します。

ただし、外部 API 仕様、JSON エンコード、ネットワーク、副作用、実装との同期は、Lean モデルの外側で確認する必要があります。
