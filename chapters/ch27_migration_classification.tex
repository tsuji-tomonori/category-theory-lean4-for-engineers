\chapter{マイグレーションを数学的に分類する}
\label{chap:ch27-migration-classification}
\BookIndex{まいぐれーしょん}{マイグレーション}
\BookIndex{ごかんせい}{互換性}
\BookIndex{しようじゃくか}{仕様弱化}

\begin{chapterabstract}
本章では、データや API のマイグレーションを「作業手順」ではなく「型の間の変換」として捉えます。
移行には、情報をまったく失わないもの、旧形式を新形式で扱えるだけのもの、情報を意図的に捨てるもの、条件を満たすデータだけ戻せるものがあります。
これらを同じ「互換性」という言葉でまとめると、レビューで何を確認すべきかが曖昧になります。

ここでは、完全同型、片方向互換、情報損失ありの移行、制約付き同型、仕様弱化という五つの分類を導入します。
そして Lean 4 で、移行関数、逆変換、round-trip 性質、well-formed 条件を小さく表す方法を確認します。
\end{chapterabstract}

\begin{learninggoals}
\item マイグレーションを型と関数として読めるようになります。
\item 完全同型、片方向互換、lossy migration、制約付き同型、仕様弱化を区別できます。
\item \texttt{rollback (migrate x) = x} のような round-trip 性質を、証明すべき仕様として切り出せます。
\item well-formed 条件が必要になる場面を説明できます。
\item 実務の移行レビューで「何を Lean で証明すべきか」を選べるようになります。
\end{learninggoals}

\section{問題設定}

ソフトウェア開発では、マイグレーションという言葉が広く使われます。
DB スキーマを変えます。
API レスポンスの形を変えます。
設定ファイルの形式を変えます。
ログの保存形式を変えます。
古いバージョンのデータを新しいバージョンで読み込めるようにします。
これらはすべて、ある形式のデータを別の形式へ移す作業です。

しかし、実務の会話では、次のような表現が混ざりやすくなります。

\begin{itemize}
\item 「旧データを新形式に移せる」
\item 「新旧は互換である」
\item 「ロールバックできる」
\item 「情報は失われない」
\item 「一部のデータだけなら戻せる」
\item 「表示上は同じなので問題ない」
\end{itemize}

これらは似ていますが、同じ主張ではありません。
旧データを新形式に変換できることと、新形式から旧形式へ完全に戻せることは違います。
表示が同じであることと、内部データが同じであることも違います。
ロールバック処理が実装されていることと、ロールバック後に必ず元の値に戻ることも違います。

本章の目的は、マイグレーションを一律に安全または危険と判定することではありません。
どの種類の安全性を主張しているのかを分類し、それぞれに対応する証明義務を明確にすることです。

図\ref{fig:ch27-migration-taxonomy}は、可逆性、保証する方向、対象範囲、保存する観測という軸から、代表的な保証パターンを配置しています。

\section{マイグレーションを型と関数として見る}

まず、マイグレーションを最小限の形で表します。
旧形式を \texttt{Old}、新形式を \texttt{New} とします。
旧形式から新形式への移行は、関数として次のように読めます。

\[
\texttt{migrate} : \texttt{Old} \to \texttt{New}
\]

新形式から旧形式へ戻す処理があるなら、それも関数です。

\[
\texttt{rollback} : \texttt{New} \to \texttt{Old}
\]

この二つの関数があるだけでは、まだ安全性は分かりません。
重要なのは、それらを合成したときに何が起きるかです。

ここで使う基本的な見方は、これまでの章で扱った「型と関数の世界」です。
型はデータ形式であり、関数は変換です。
マイグレーションの分類とは、変換と逆変換の組み合わせがどの法則を満たすかを調べることだといえます。

\begin{definitionbox}
本章では、旧形式を \texttt{Old}、新形式を \texttt{New} と書きます。
\texttt{migrate : Old -> New} は旧形式から新形式への変換であり、\texttt{rollback : New -> Old} は新形式から旧形式への変換です。
証明したい性質は、これらの関数を合成した結果に対する等式や条件として表します。
\end{definitionbox}

\section{完全同型：情報を失わない変更}

もっとも強い分類は、完全同型です。
これは、旧形式と新形式が見た目は違っても、持っている情報が同じであることを意味します。

たとえば、次のような変更を考えます。

\begin{itemize}
\item フィールド名を \texttt{name} から \texttt{displayName} に変えます。
\item 平坦なレコードをネストしたレコードに変えます。
\item \texttt{firstName} と \texttt{lastName} を \texttt{profile} オブジェクトの中へ移します。
\end{itemize}

これらは、形が変わっています。
しかし、すべての情報を保存しており、機械的に戻せるなら、完全同型として扱えます。

\begin{definitionbox}
\textbf{完全同型}とは、\texttt{migrate : Old -> New} と \texttt{rollback : New -> Old} があり、両方向の round-trip が成り立つ移行です。
すなわち、任意の旧データ \texttt{x} と任意の新データ \texttt{y} について、次が成り立ちます。
\[
\texttt{rollback}(\texttt{migrate}(x)) = x
\]
\[
\texttt{migrate}(\texttt{rollback}(y)) = y
\]
この二つが両方あるとき、旧形式と新形式は情報の量が同じだと見なせます。
\end{definitionbox}

\begin{warningbox}
\texttt{Old -> New} と \texttt{New -> Old} の二つの関数があるだけでは、完全同型ではありません。
必要なのは、変換して戻したときに本当に元に戻るという法則です。
実装上の「ロールバック関数がある」と、数学的な「ロールバックが正しい」は別の主張です。
\end{warningbox}

Lean では、完全同型の形を小さな構造体として表せます。
ここでは Mathlib の高度な圏論ライブラリは使わず、最小限の構造を自分で定義します。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure EquivLike (A B : Type) where
  toFun : A -> B
  invFun : B -> A
  left_inv : (a : A) -> invFun (toFun a) = a
  right_inv : (b : B) -> toFun (invFun b) = b
\end{minted}
\caption{\texttt{EquivLike}}
\label{lst:ch27-equivlike}
\end{listing}

次の例では、ユーザー名のフィールド名を変えるだけの移行を考えます。
これは情報を落としていないので、両方向の round-trip が成り立ちます。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure UserV1 where
  id : Nat
  name : String
  deriving DecidableEq, Repr

structure UserV2 where
  id : Nat
  profileName : String
  deriving DecidableEq, Repr

def migrateUser (u : UserV1) : UserV2 :=
  { id := u.id, profileName := u.name }

def rollbackUser (v : UserV2) : UserV1 :=
  { id := v.id, name := v.profileName }

theorem user_left_roundtrip (u : UserV1) :
    rollbackUser (migrateUser u) = u := by
  cases u
  rfl

theorem user_right_roundtrip (v : UserV2) :
    migrateUser (rollbackUser v) = v := by
  cases v
  rfl
\end{minted}
\caption{\texttt{user\_left\_roundtrip} と \texttt{user\_right\_roundtrip}}
\label{lst:ch27-user-roundtrip}
\end{listing}

この例で Lean が確認しているのは、個別のテストデータではありません。
任意の \texttt{UserV1} と任意の \texttt{UserV2} について、変換して戻すと元の値に戻るという一般的な主張です。

図\ref{fig:ch27-roundtrip-laws}の二つの三角形は、それぞれ旧形式側と新形式側の合成が恒等変換になることを表します。

\FigurePlaceholder{fig-ch27-roundtrip-laws.png}{0.88}{両方向の round-trip}{fig:ch27-roundtrip-laws}

\begin{warningbox}
以下の節は、互いに排他的な「種類」を列挙するものではありません。
可逆性、保証する方向、対象範囲、保存する観測という別々の軸から、よく使う保証パターンを説明します。
同じ移行が、旧形式については片方向 round-trip を持ち、新形式の像上では制約付き同型であり、特定の観測については仕様保存でもあるという重なりは、普通に起こります。
\end{warningbox}

\section{片方向互換：旧データを新形式で扱える}

次に、片方向互換を考えます。
これは、旧形式から新形式へは安全に移せても、新形式のすべてを旧形式へ完全には戻せない状況です。

典型例は、フィールド追加です。
たとえば、旧形式には \texttt{token} だけがあり、新形式には \texttt{token} に加えて \texttt{issuedAt} があるとします。
旧データを新形式に移すときは、\texttt{issuedAt} にデフォルト値を入れれば移行できます。
しかし、新形式の任意の \texttt{issuedAt} を旧形式に戻すと、その時刻情報は失われます。

\begin{softwarebox}
片方向互換は、後方互換性の議論でよく現れます。
旧クライアントや旧データを新システムで扱えることは重要です。
しかし、それは「新形式の全データを旧形式へ戻せる」ことまでは意味しません。
\end{softwarebox}

Lean では、旧形式から新形式へ移して戻る片方向だけを証明できます。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure OldToken where
  token : String
  deriving DecidableEq, Repr

structure NewToken where
  token : String
  issuedAt : Nat
  deriving DecidableEq, Repr

def oldToNew (o : OldToken) : NewToken :=
  { token := o.token, issuedAt := 0 }

def newToOld (n : NewToken) : OldToken :=
  { token := n.token }

theorem old_roundtrip (o : OldToken) :
    newToOld (oldToNew o) = o := by
  cases o
  rfl
\end{minted}
\caption{\texttt{old\_roundtrip}}
\label{lst:ch27-one-way}
\end{listing}

この証明は、「旧データは新形式へ移しても旧形式として回収できる」ことを示しています。
しかし、逆向きの \texttt{oldToNew (newToOld n) = n} は一般には成り立ちません。
\texttt{newToOld} が \texttt{issuedAt} を捨てるためです。

\begin{warningbox}
片方向互換を完全同型と呼んではいけません。
旧データを新形式で扱えることは、実務でとても有用です。
しかし「いつでも戻せる」「情報は失われない」と述べるには、逆向きの round-trip も必要です。
\end{warningbox}

\section{情報損失あり：戻せない変更}

三つ目は、情報損失ありの移行です。
これは、変換の途中で意図的に、または構造上避けられず、情報を捨てる移行です。

たとえば、ログを匿名化して公開ログを作る場合を考えます。
内部ログには \texttt{userId} と \texttt{message} があります。
公開ログには \texttt{message} だけを残します。
この変換は必要な場合がよくあります。
しかし、公開ログから元の \texttt{userId} を復元することはできません。

\begin{definitionbox}
\textbf{情報損失ありの移行}とは、旧形式から新形式への変換によって、旧形式に含まれていた情報の一部が新形式から観測できなくなる移行です。
情報を落とすこと自体が悪いのではありません。
問題は、情報を落としているにもかかわらず、完全復元できるかのように扱うことです。
\end{definitionbox}

Lean では、完全復元を証明するのではなく、保存される性質だけを証明します。
次の例では、匿名化後もメッセージ本文は保存されますが、ユーザー ID は保存されません。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure PrivateLog where
  userId : Nat
  message : String
  deriving DecidableEq, Repr

structure PublicLog where
  message : String
  deriving DecidableEq, Repr

def anonymize (l : PrivateLog) : PublicLog :=
  { message := l.message }

def SamePublicView (x y : PrivateLog) : Prop :=
  anonymize x = anonymize y

theorem anonymize_preserves_message (l : PrivateLog) :
    (anonymize l).message = l.message := by
  rfl

theorem different_users_can_have_same_public_view (msg : String) :
    SamePublicView { userId := 1, message := msg }
                   { userId := 2, message := msg } := by
  rfl
\end{minted}
\caption{\texttt{anonymize}}
\label{lst:ch27-lossy-log}
\end{listing}

最後の定理は、異なる \texttt{userId} を持つ二つの内部ログが、同じ公開ログになりうることだけを示しています。
これは非単射性の証拠であり、攻撃者が利用できる補助情報、メッセージ本文、頻度、外部相関まで含めた秘匿性の証明ではありません。

匿名化では、型から特定フィールドが除かれたことや表示用メッセージの保存が証明候補になります。

秘匿性を主張するには、観測者と補助情報を定めた別のセキュリティ仕様が必要です。

\section{制約付き同型：well-formed なデータだけ戻せる}

四つ目は、制約付き同型です。
これは、すべての新データについては戻せませんが、ある条件を満たす新データだけなら戻せるという状況です。

先ほどの \texttt{NewToken} の例に戻ります。
任意の \texttt{NewToken} については、旧形式へ戻して再び新形式へ移したとき、元の \texttt{issuedAt} は復元されません。
しかし、\texttt{issuedAt = 0} という条件を満たす新データに限れば、戻して移しても元に戻ります。

このような条件を、本章では \textbf{well-formed 条件} と呼びます。
well-formed とは、「この変換にとって正しい形をしている」という意味です。

\begin{definitionbox}
\textbf{制約付き同型}とは、すべての値ではなく、well-formed 条件を満たす値の範囲で round-trip が成り立つ移行です。
Lean では、条件を述語として定義し、その条件を仮定にした定理を書きます。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
def IsDefaultIssuedAt (n : NewToken) : Prop :=
  n.issuedAt = 0

theorem new_roundtrip_when_default
    (n : NewToken) (h : IsDefaultIssuedAt n) :
    oldToNew (newToOld n) = n := by
  cases n with
  | mk token issuedAt =>
      unfold IsDefaultIssuedAt at h
      simp at h
      cases h
      rfl
\end{minted}
\caption{\texttt{new\_roundtrip\_when\_default}}
\label{lst:ch27-conditional-roundtrip}
\end{listing}

この定理は、無条件の復元ではありません。
\texttt{IsDefaultIssuedAt n} という仮定があります。
つまり、\texttt{issuedAt} がデフォルト値である新データだけなら、旧形式に戻して再び新形式へ移しても元に戻ります。

\begin{softwarebox}
制約付き同型は、実務では「この移行ツールで作ったデータだけなら戻せる」「正規化済みデータだけなら復元できる」「バージョン番号が特定範囲なら互換である」といった形で現れます。
重要なのは、条件を口頭で済ませず、仕様として明示することです。
\end{softwarebox}

\section{仕様弱化：完全復元ではなく、必要な性質だけを保存する}

五つ目は、仕様弱化です。
これは、完全な復元を諦め、実務で必要な性質だけを保存する形に仕様を調整することです。

たとえば、氏名の扱いを考えます。
旧形式に \texttt{fullName : String} があり、新形式に \texttt{firstName : String} と \texttt{lastName : String} があるとします。
この移行は、文化圏、ミドルネーム、スペース、表記ゆれの問題を含みます。
すべての文字列について完全復元を証明するのは、現実的ではありません。

この場合は、証明対象を変えます。
たとえば、次のような弱い仕様にします。

\begin{itemize}
\item well-formed な入力だけを対象にします。
\item 正規化後の表示が一致することを証明します。
\item 検索キーとして使う文字列だけが保存されることを証明します。
\item 旧 API の表示互換性だけを証明します。
\end{itemize}

\begin{definitionbox}
\textbf{仕様弱化}とは、証明できない強すぎる主張を捨て、実務で必要な弱い主張へ置き換えることです。
完全な等式ではなく、表示上の一致、正規化後の一致、順序関係、安全な抽象化などを証明対象にします。
\end{definitionbox}

仕様弱化は妥協ではありますが、悪いことではありません。
証明できないことを証明済みのように扱うよりも、はるかに正確です。

\begin{warningbox}
「証明できないから Lean が役に立たない」のではありません。
Lean は、強すぎる主張が本当に強すぎることを明らかにします。
そこで、証明対象を実務で必要な性質へ落とします。
この判断そのものが、仕様レビューの価値になります。
\end{warningbox}

\section{実務で証明すべき性質の選び方}

実務では、すべての性質を Lean で証明する必要はありません。
むしろ、証明する性質を絞ることが重要です。
判断の出発点は、次の問いです。

\begin{itemize}
\item 変更前後の型は何でしょうか。
\item 移行関数は何でしょうか。
\item 逆変換は存在するでしょうか。
\item 逆変換が存在するなら、両方向の round-trip を要求するのでしょうか。
\item 情報損失はあるでしょうか。
\item well-formed 条件は必要でしょうか。
\item 完全な等式ではなく、弱めた仕様で十分でしょうか。
\item Lean のモデルと本番実装の差分は、どこにあるでしょうか。
\end{itemize}

マイグレーションレビューでは「互換性あり」を、保証する方向、条件、保存する観測へ分解してから、証明対象を絞ります。

たとえば、DB スキーマ変更であれば、すべての SQL の意味を証明するのは重すぎる場合がよくあります。
しかし、重要なクエリ結果が保存されること、主キーが保存されること、件数が保存されることは、小さなモデルで扱えるかもしれません。

API バージョニングであれば、すべてのクライアント挙動を証明するのではなく、旧レスポンスから新レスポンスへ変換したときに特定フィールドが保存されることや、adapter が業務ロジックと干渉しないことを証明対象にできます。

ログ匿名化であれば、復元不能性そのものを厳密に扱うには追加のモデルが必要になります。
しかし、少なくとも公開ログに残すメッセージが保存されることや、内部 ID を出力しない構造になっていることは、小さな型で表せます。

\begin{warningbox}
Lean で証明した性質は、Lean 上のモデルについての性質です。
本番 DB、外部 API、文字コード、NULL、時刻、並行実行、運用手順まで自動的に保証されるわけではありません。
設計文書では、証明した範囲と証明していない範囲を分けて書く必要があります。
\end{warningbox}

\section{本章のまとめ}

マイグレーションは、旧形式から新形式への単なる作業ではなく、型の間の変換です。
完全同型では、両方向の round-trip を証明します。
片方向互換では、旧データを新形式で扱い、必要なら旧形式へ戻せることだけを証明します。
情報損失ありの移行では、完全復元を主張せず、保存される性質を証明します。
制約付き同型では、well-formed 条件を明示して、その範囲だけで round-trip を証明します。
仕様弱化では、強すぎる等式を諦め、実務で必要な弱い性質へ落とします。
