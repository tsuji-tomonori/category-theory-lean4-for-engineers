\chapter{圏論用語チートシート}
\label{app:category-cheatsheet}
\BookIndex{けんろんようご}{圏論用語}

\begin{chapterabstract}
この付録は、本文で使う圏論用語を、ソフトウェアエンジニアが設計、仕様、変更管理で使いやすい形にまとめた早見表です。
用語の厳密な一般論を網羅することよりも、「この言葉はコードや設計レビューでは何を指すのか」「Lean 4 ではどのような形で小さく表せるのか」をすぐに確認できることを目的とします。
\end{chapterabstract}

\begin{learninggoals}
\item 圏論の基本語彙を、型、関数、データ変換、API adapter、DB join などの実務例へ読み替えられるようになります。
\item 本文に現れた用語について、どの章または証明パターンに戻ればよいかを判断できるようになります。
\item Lean 4 で概念を小さくモデル化するときの最小単位を確認できます。
\end{learninggoals}

\section{使い方}

\begin{intuitionbox}
この付録は、圏論用語を暗記するための辞書ではありません。
実務上の問いへ戻すための索引です。
たとえば、「このマイグレーションは戻せるのか」と考えているなら、同型を確認します。
「この adapter は業務ロジックと干渉しないか」と考えているなら、自然変換を確認します。
「二つのデータソースを同じ ID で整合させたい」と考えているなら、Pullback を確認します。
\end{intuitionbox}

\begin{softwarebox}
読み方の基本は一つです。
対象は型やデータ構造、射は関数や変換、合成はパイプライン、法則はリファクタリングしても意味が変わらないための条件として読みます。
抽象語を見たら、まず「何を変換しているのか」「何が保存されるべきか」「どの順序で実行しても同じか」という問いへ戻します。
\end{softwarebox}

\section{用語別ミニカード}

\subsection{対象}

\begin{definitionbox}
対象とは、圏の中で矢印の出発点または到着点になるものです。
型と関数の世界で考えるなら、対象は \texttt{Nat}、\texttt{String}、\texttt{UserV1}、\texttt{UserV2} のような型です。
\end{definitionbox}

\begin{softwarebox}
対象は「値そのもの」ではなく、「その値が属する形」と読むとよいでしょう。
たとえば、一人のユーザー値ではなく、\texttt{User} というデータ構造全体が対象です。
状態遷移を扱うなら、状態空間も対象として読めます。
\end{softwarebox}

\begin{leanbox}
Lean では、対象はまず \texttt{Type} として現れます。
ドメインモデルであれば、\texttt{structure User where ...} のように定義します。
\end{leanbox}

\subsection{射}

\begin{definitionbox}
射とは、ある対象から別の対象へ向かう矢印です。
型と関数の世界では、射は関数です。
\texttt{A -> B} は、対象 \texttt{A} から対象 \texttt{B} への射として読めます。
\end{definitionbox}

\begin{softwarebox}
一般の圏における射は、関数に限られません。
マイグレーション、正規化、DTO 変換、encoder、decoder、adapter、validator なども、対象、合成、恒等射、法則を定めたモデルの中では射として見られます。
ただし、副作用を含む任意の処理を、モデルを定めずに自動的に射と呼べるわけではありません。
\end{softwarebox}

\begin{warningbox}
実務コードには、副作用を持つ処理も多くあります。
本書の初期モデルでは、まず純粋関数として切り出せる部分を射として扱います。
失敗や状態を含む処理は、後で \texttt{Option}、\texttt{Except}、\texttt{State} などの文脈付き処理として整理します。
\end{warningbox}

\subsection{合成}

\begin{definitionbox}
合成とは、\(f : A \to B\) と \(g : B \to C\) をつなぎ、\(g \circ f : A \to C\) を作る操作です。
先に \(f\) を実行し、その結果に \(g\) を適用します。
\end{definitionbox}

\begin{softwarebox}
合成はパイプラインです。
入力正規化、検証、変換、保存用 DTO への変換を順につなぐとき、各ステージの出力型と次の入力型が合っていれば合成できます。
\end{softwarebox}

\begin{leanbox}
Lean では、\texttt{fun x => g (f x)} が最小の合成表現です。
関数合成記法を使わなくても、まずこの形で読むと分かりやすいでしょう。
\end{leanbox}

\subsection{恒等射}

\begin{definitionbox}
恒等射とは、対象 \(A\) から \(A\) 自身への「何もしない」射です。
通常は \(\mathrm{id}_A\) と書きます。
\end{definitionbox}

\begin{softwarebox}
恒等射は no-op です。
何もしない adapter、変換しない migration、空の wrapper として読めます。
パイプラインの前後に置いても、結果は変わりません。
\end{softwarebox}

\begin{definitionbox}
恒等律は、任意の \(f : A \to B\) に対して次の二つが成り立つという法則です。
\[
\mathrm{id}_B \circ f = f, \qquad f \circ \mathrm{id}_A = f.
\]
\end{definitionbox}

\subsection{圏}

\begin{definitionbox}
圏とは、対象、射、射の合成、恒等射を持ち、恒等律と結合律を満たす世界です。
\end{definitionbox}

\begin{softwarebox}
圏は「合成できる処理の設計空間」と読みます。
パイプラインを安全に組み替えるには、各処理をつなげられること、何もしない処理が自然に振る舞うこと、括弧の付け方を変えても意味が変わらないことが重要です。
\end{softwarebox}

\begin{warningbox}
本書の前半では、一般の圏を最初から Mathlib の \texttt{CategoryTheory} で扱いません。
まず型と関数の世界で、対象、射、合成、恒等射の感覚を作ります。
\end{warningbox}

\subsection{双対}

\begin{definitionbox}
双対とは、すべての矢印の向きを逆にして得られる見方です。
ある概念に対して、矢印を逆にした対応概念が現れる場合が多くあります。
\end{definitionbox}

\begin{softwarebox}
双対は、作る側と使う側、入力側と出力側、分解と合成を反転して見るための視点です。
積と余積、極限と余極限のように、双対な概念は組にして理解すると見通しがよくなります。
\end{softwarebox}

\begin{warningbox}
双対は、「逆関数が存在する」という意味ではありません。
矢印を逆向きにして理論全体を見直すという、メタな操作です。
同型の逆変換とは区別します。
\end{warningbox}

\subsection{同型}

\begin{definitionbox}
同型とは、二つの対象が情報として同じであり、相互変換しても元に戻る関係です。
\(A\) と \(B\) が同型であるとは、\(to : A \to B\) と \(from : B \to A\) があり、両方向の round-trip が成り立つことです。
\[
from \circ to = \mathrm{id}_A, \qquad to \circ from = \mathrm{id}_B.
\]
\end{definitionbox}

\begin{softwarebox}
同型は、lossless migration を表す言葉です。
\texttt{UserV1} と \texttt{UserV2} の形が異なっても、\texttt{migrate} と \texttt{rollback} の両方向で完全に戻せるなら、情報は失われていないといえます。
\end{softwarebox}

\begin{warningbox}
\texttt{A -> B} と \texttt{B -> A} が存在するだけでは、同型ではありません。
戻したときに本当に元に戻ることを証明しなければなりません。
\end{warningbox}

\subsection{積}

\begin{definitionbox}
積とは、「両方を持つ」構造です。
型の世界では、\texttt{Prod A B} や二つのフィールドを持つ構造体として現れます。
\end{definitionbox}

\begin{softwarebox}
積は、ペア、レコード、DTO、複数のフィールドをまとめたレスポンスとして読みます。
ユーザー情報と権限情報を同時に持つ値や、注文と請求先情報を同時に持つ値は、積の直感で理解できます。
\end{softwarebox}

\begin{leanbox}
Lean では、\texttt{Prod A B}、\texttt{(A × B)}、または \texttt{structure} を使います。
取り出しは、第一射影または第二射影として読めます。
\end{leanbox}

\subsection{余積}

\begin{definitionbox}
余積とは、「どちらか一方」を表す構造です。
型の世界では、\texttt{Sum A B} として現れます。
\end{definitionbox}

\begin{softwarebox}
余積は、成功または失敗、旧形式または新形式、同期結果または非同期受付結果のような分岐を表します。
\texttt{Either}、\texttt{Result}、\texttt{Option} は、余積として読めます。
\end{softwarebox}

\begin{warningbox}
積は「両方ある」構造であり、余積は「どちらかである」構造です。
レコード設計とエラー分岐設計では、この違いを混同しないことが重要です。
\end{warningbox}

\subsection{関手}

\begin{definitionbox}
関手とは、対象を対象へ、射を射へ移し、恒等射と合成を保存する対応です。
\end{definitionbox}

\begin{softwarebox}
関手は、「構造を壊さない一括変換」と読みます。
\texttt{List.map migrate} は、リストという構造を保ったまま各要素を移行します。
\texttt{Option.map f} は、値があるかないかという構造を保ったまま、中身だけを変換します。
\end{softwarebox}

\begin{definitionbox}
関手則は、次の二つとして読むとよいでしょう。
\[
map\;\mathrm{id} = \mathrm{id}, \qquad map\;(g \circ f) = map\;g \circ map\;f.
\]
これは、何もしない変換と二段階変換が、構造の中へ持ち上げても自然に振る舞うという条件です。
\end{definitionbox}

\subsection{自然変換}

\begin{definitionbox}
自然変換とは、二つの関手の間の一貫した変換です。
型 \(A\) ごとに変換 \(\eta_A : F A \to G A\) があり、任意の関数 \(f : A \to B\) と整合します。
\end{definitionbox}

\begin{softwarebox}
自然変換は、型に依存しない adapter です。
たとえば、\texttt{Option A} を \texttt{List A} にする処理は、\texttt{A} が \texttt{User} でも \texttt{Order} でも同じ規則で動きます。
先に中身を変換してから adapter を通した場合と、先に adapter を通してから中身を変換した場合とで結果が一致することが、自然性です。
\end{softwarebox}

\begin{definitionbox}
自然性は、次の可換性として読みます。
\[
G(f) \circ \eta_A = \eta_B \circ F(f).
\]
実務モデルでは、ここで射 \(f\) として表した処理について、「adapter の前後で同じ結果になる」と読みます。
\end{definitionbox}

\subsection{モノイド}

\begin{definitionbox}
モノイドとは、結合演算と単位元を持つ構造です。
演算は結合律を満たし、単位元は左右のどちらから結合しても値を変えません。
\end{definitionbox}

\begin{softwarebox}
モノイドは、ログ、文字列、リスト、設定、集計結果などを安全にまとめるための言葉です。
空ログ、空文字列、空リスト、デフォルト設定は、単位元として読めます。
\end{softwarebox}

\begin{warningbox}
モノイドに必要なのは結合律であり、必ずしも交換律ではありません。
すなわち、\(a \diamond b = b \diamond a\) までは要求しません。
ログや文字列では順序が重要であるため、交換できるとは限りません。
\end{warningbox}

\subsection{モナド}

\begin{definitionbox}
モナドとは、文脈付きの値を扱い、その文脈を保ったまま処理を合成するための構造です。
中心となるのは、\texttt{pure} と \texttt{bind} です。
\end{definitionbox}

\begin{softwarebox}
モナドは、失敗する処理、例外を返す処理、状態を更新する処理、検証ログを伴う処理を合成するための形として読みます。
\texttt{Option} は失敗するかもしれない処理、\texttt{Except E} はエラー情報を返す処理の代表例です。
\end{softwarebox}

\begin{warningbox}
モナドは、「副作用そのもの」ではありません。
文脈付き処理を、一定の法則に従って合成するためのインターフェースです。
実務では、法則をリファクタリング規則として読むと役立ちます。
\end{warningbox}

\subsection{随伴}

\begin{definitionbox}
随伴とは、二つの構成が互いに最も自然に対応している関係です。
本書では、まず「自由に作る」操作と「構造を忘れる」操作の対応として読みます。
\end{definitionbox}

\begin{softwarebox}
随伴は、余計な関係式を加えずに構造を自由に生成する設計と、その構造を忘れて基礎データだけを見る設計とを結び付けます。
たとえば、要素型からリストを作ることを free monoid の例として読むと、要素への関数がリスト全体の準同型へ一意に拡張される理由が見えてきます。
\end{softwarebox}

\begin{warningbox}
この付録では、随伴の一般定義や単位・余単位の証明までは扱いません。
\ChapterRef{chap:ch37_adjunction}で直感を作り、必要になったら標準教材や Mathlib の \texttt{CategoryTheory} へ進みます。
\end{warningbox}

\subsection{極限}

\begin{definitionbox}
極限とは、ある図式へ整合する射を持つ錐のうち、ほかの任意の錐から一意の射を受け取る普遍的な錐です。
単に条件を満たす値が一つ存在するだけでは、極限になりません。
\end{definitionbox}

\begin{softwarebox}
極限は、複数の設定、複数の API の結果、複数の制約を同時に満たすデータを作る場面に現れます。
単に値を並べるだけでなく、整合条件を満たす必要があるときに重要になります。
\end{softwarebox}

\begin{warningbox}
初学者向けには、一般の極限を最初から形式化しません。
まずは構造体と述語で「条件を満たす値」を表し、その後に Pullback などの具体例へ進みます。
\end{warningbox}

\subsection{Pullback}

\begin{definitionbox}
Pullback、または引き戻しとは、二つの射が共通の対象へ向かうとき、その二つの射を可換にする普遍的な対象です。
型と関数の圏では、その結果が一致するペアの型として表せます。
\end{definitionbox}

\begin{softwarebox}
Pullback は、整合条件を満たす join と読めます。
たとえば、\texttt{User} と \texttt{Order} を単にペアにするのではなく、\texttt{user.id = order.userId} を満たすペアだけを作るときに、Pullback の直感が働いています。
\end{softwarebox}

\begin{leanbox}
Lean では、まず次のように小さく表します。
二つの値を持つ構造体を作り、その中に \texttt{left.id = right.userId} のような証明フィールドを入れます。
または、\texttt{Subtype} で「整合するペア」だけを取り出します。
\end{leanbox}

\section{Lean 4 で見る最小対応}

\begin{leanbox}
次のコードは、圏論用語のうち、対象、射、合成、恒等射、同型、自然性を Lean で最小限に表したものです。
Mathlib の高度な \texttt{CategoryTheory} API は使わず、型と関数だけで確認できる形にしています。
\end{leanbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace AppendixB

def idFn (A : Type) : A -> A := fun x => x

def comp {A B C : Type} (g : B -> C) (f : A -> B) : A -> C :=
  fun x => g (f x)

theorem comp_assoc {A B C D : Type}
    (h : C -> D) (g : B -> C) (f : A -> B) :
    comp h (comp g f) = comp (comp h g) f := by
  rfl

structure Iso (A B : Type) where
  toFun : A -> B
  invFun : B -> A
  left_inv : forall a : A, invFun (toFun a) = a
  right_inv : forall b : B, toFun (invFun b) = b

def optionToList {A : Type} : Option A -> List A
  | none => []
  | some a => [a]

theorem optionToList_natural {A B : Type}
    (f : A -> B) (x : Option A) :
    optionToList (Option.map f x) = List.map f (optionToList x) := by
  cases x <;> rfl

end AppendixB
\end{minted}
\caption{\texttt{idFn}・\texttt{comp}・\texttt{Iso}・\texttt{optionToList\_natural}}
\label{lst:appB_category_minicode}
\end{listing}

このコードは、付録Bで扱う用語を Lean の小さな定義へ対応させた一覧です。
\texttt{idFn}と\texttt{comp}は恒等射と合成を表し、\texttt{Iso}は二方向の関数と round-trip 性質を表します。
\texttt{optionToList\_natural}は、adapter が\texttt{map}と干渉しないことを表しています。

\section{本付録のまとめ}

対象、射、合成、恒等射は、型、関数、パイプライン、no-op として読みます。

同型は、情報を失わない相互変換であり、round-trip 証明が必要です。

関手は恒等射と合成を保存する変換であり、自然変換は対象ごとの成分が自然性を満たす adapter です。

モノイドは集約、モナドは文脈付き処理の合成、Pullback は整合条件付き join として読みます。

Lean では、まず型、関数、構造体、述語、等式を使って小さくモデル化し、必要になってから Mathlib の \texttt{CategoryTheory} へ進みます。
