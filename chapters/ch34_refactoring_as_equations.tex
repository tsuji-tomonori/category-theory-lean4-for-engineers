\begin{chapterabstract}
本章では、リファクタリングを「意味を変えない書き換え」として扱います。
関数名の変更、関数の切り出し、共通関数への置き換え、処理順序の変更、失敗を含む処理の括弧変更を考えます。
変更の前後で外から見た振る舞いが同じなら、その変更は Lean の等式で表せます。
本章の目的は、すべての実装を Lean へ移すことではありません。
変更で壊れてほしくない小さな意味保存性を選び、変更前後の関数が同じ結果を返すことを検査可能な形にします。
\end{chapterabstract}

\begin{learninggoals}
\item リファクタリングを「名前の変更」ではなく「観測可能な意味の保存」として説明できます。
\item 関数抽出、処理順序の変更、共通化、wrapper 削除を等式として読み替えられます。
\item \texttt{Option} のような失敗を含む処理について、\texttt{bind} の括弧付け変更を意味保存として理解できます。
\item Lean 4 で小さな変更前後の関数を定義し、\texttt{rfl}、\texttt{rw}、\texttt{calc}、\texttt{cases} を使って安全性を証明できます。
\item 実務コードと Lean モデルの差分を明示し、コードレビューで使える証明の粒度を判断できます。
\end{learninggoals}

\section{リファクタリングを等式で捉える}

リファクタリングは、実装の内部構造を変える作業です。
関数を分けます。
重複した計算を共通関数へ寄せます。
処理をパイプラインにします。
不要になった adapter を削除します。
失敗を含む処理を読みやすい \texttt{bind} の連鎖へ整えます。

しかし、実務で本当に確認したいのは「コードの形がきれいになったか」だけではありません。
確認したい中心は、「外から見た意味が変わっていないか」です。
入力が同じなら、出力も同じでしょうか。
成功と失敗の分岐は同じでしょうか。
保存すべき ID、件数、金額、状態、不変条件は変わっていないでしょうか。

リファクタリングを等式として見るとは、変更前の処理を左辺、変更後の処理を右辺に置き、両者が同じ結果を返すことを示すという意味です。
つまり、コードレビューでよく述べる「意味は変えていません」を、Lean が検査できる命題に変えます。

\FigurePlaceholder{fig-ch34-refactoring-equation-map.png}{0.90}{リファクタリングと意味保存}{fig:ch34-refactoring-equation-map}

図\ref{fig:ch34-refactoring-equation-map}は、変更前後の実装を等式の両辺に置き、その間を証明で結ぶ見方を示しています。
前の章までに、合成可能な設計、等式推論、関数外延性、モナドの直感を見てきました。
本章では、それらを実務のリファクタリングへ戻します。
圏論の言葉では、処理は射であり、リファクタリングは射の表現を変える作業です。
変えるべきでないものは、射としての意味、つまり入力から出力への対応です。

\section{名前変更ではなく意味保存を見る}

まず、リファクタリングで保存したい性質を明確にします。
関数名、変数名、ファイル分割、ディレクトリ構成は、直接には意味そのものではありません。
もちろん名前は読みやすさに影響しますが、Lean で証明したいのは、名前の良し悪しではなく、観測できる振る舞いです。

本章では、二つの処理 \(f\) と \(g\) が意味保存的に同じであるとは、扱うモデルの範囲で、すべての入力 \(x\) について \(f(x) = g(x)\) が成り立つことをいいます。
状態や失敗を含む処理では、返される状態、成功値、失敗値まで含めて等しいことを確認します。

たとえば、税込金額を計算する処理を考えます。
変更前は式を直接書いていました。
変更後は、基礎金額の計算と税額計算を補助関数へ切り出しました。
この変更が安全であるとは、任意の小計と送料に対して、変更前後の合計が同じであるという意味です。

\begin{listing}[htbp]
\begin{minted}{lean4}
def basePrice (subtotal shipping : Nat) : Nat :=
  subtotal + shipping

def tax10 (n : Nat) : Nat :=
  n / 10

def totalBefore (subtotal shipping : Nat) : Nat :=
  (subtotal + shipping) + ((subtotal + shipping) / 10)

def totalAfter (subtotal shipping : Nat) : Nat :=
  let base := basePrice subtotal shipping
  base + tax10 base

theorem extract_base_price_preserves (subtotal shipping : Nat) :
    totalAfter subtotal shipping = totalBefore subtotal shipping := by
  rfl
\end{minted}
\caption{\texttt{extract\_base\_price\_preserves}}
\label{lst:ch34-extract-base-price}
\end{listing}

この証明で \texttt{rfl} が通るのは、\texttt{totalAfter} を定義展開すると \texttt{totalBefore} と同じ式になるためです。
ここでは、高度な定理を使っていません。
Lean に「補助関数を展開すれば同じ形になる」と確認させているだけです。

このような証明は、小さすぎるように見えるかもしれません。
しかし、実務上の価値は「リファクタリングの単位を小さく切り出す」ことにあります。
複雑な関数全体ではなく、壊れると困る計算の核だけを Lean へ移します。
そこで等式が通れば、少なくともその核については、変更が意味を変えていないと説明できます。

\section{関数抽出は補助定義の導入である}

関数抽出は、リファクタリングの中でも最も扱いやすい変更です。
多くの場合、変更前に式の中へ直接書かれていた部分式に名前を付けるだけだからです。
数学的には、共通する部分式を補助定義にする操作と読めます。

実務で関数抽出を証明したいときは、まず「抽出した関数が何を計算しているか」を短い仕様にします。
次に、抽出前の式と抽出後の式を並べ、補助関数を展開すれば同じ形になるかを確認します。

関数抽出が安全になりやすいのは、抽出した関数が純粋な関数である場合です。
つまり、同じ入力なら同じ出力を返し、外部状態、時刻、乱数、DB、ネットワーク、ログ出力などに依存しない場合です。
この条件が崩れると、同じ式に見えても意味が変わる可能性があります。

「同じコードを関数へ切り出しただけ」に見えても、評価タイミングが変わると意味が変わることがあります。
たとえば、現在時刻を読む処理、乱数を生成する処理、DB から値を読む処理、副作用を持つログ出力は、単純な等式として扱えないことがあります。
その場合は、時刻や外部入力を明示的な引数にするなど、証明しやすいモデルへ切り出す必要があります。

この注意は、圏論の言葉では「同じ射として扱える範囲を決める」といえます。
副作用を隠したままでは、射の入力と出力が足りません。
Lean モデルでは、観測したい外部情報を型に入れ、処理を明示的な関数として表します。

\section{処理順序の変更は交換法則を要求する}

次に、処理順序の変更を考えます。
パイプラインの順序を入れ替えるリファクタリングは、見た目以上に危険です。
二つの処理 \(f\) と \(g\) について、\texttt{g (f x)} と \texttt{f (g x)} が常に等しいとは限らないためです。

安全に順序を入れ替えられるかどうかは、少なくとも観測対象について等式
\[
  g(f(x)) = f(g(x))
\]
がすべての入力 \(x\) で成り立つかによって判断します。
この形は、自然性や可換図式の章で見た「順序を変えても結果が同じ」という考え方と同じです。

\FigurePlaceholder{fig-ch34-order-change.png}{0.88}{処理順序と可換性}{fig:ch34-order-change}

図\ref{fig:ch34-order-change}の二つの経路が同じ出力へ到達するとき、二処理の順序はこのモデル上で交換できます。
単純な例として、金額に二種類の固定手数料を加える処理を考えます。
どちらも単に自然数を足すだけなら、順序を入れ替えても結果は同じです。
ここで使う道具は、足し算の結合律と交換律です。

\begin{listing}[htbp]
\begin{minted}{lean4}
def addServiceFee (fee amount : Nat) : Nat :=
  amount + fee

def addHandlingFee (fee amount : Nat) : Nat :=
  amount + fee

theorem fixed_fee_order (amount service handling : Nat) :
    addHandlingFee handling (addServiceFee service amount) =
    addServiceFee service (addHandlingFee handling amount) := by
  unfold addHandlingFee addServiceFee
  calc
    (amount + service) + handling = amount + (service + handling) := by
      rw [Nat.add_assoc]
    _ = amount + (handling + service) := by
      rw [Nat.add_comm service handling]
    _ = (amount + handling) + service := by
      rw [← Nat.add_assoc]
\end{minted}
\caption{\texttt{fixed\_fee\_order}}
\label{lst:ch34-order-preserves}
\end{listing}

この証明では、\texttt{unfold} で二つの補助関数を展開し、\texttt{calc} で式変形を段階的に示しています。
\texttt{Nat.add\_assoc} は結合律、\texttt{Nat.add\_comm} は交換律です。
定理は暗記するものではなく、「この順序変更が安全である理由」を表す道具として使います。

この定理から分かるのは、二処理が固定値の加算としてモデル化されているため可換だということだけです。
課金規則として互いに独立であることや、本番処理に別の副作用がないことまでは証明していません。

一方、順序を入れ替えてはいけない処理も多くあります。
たとえば、「1を足してから2倍する」処理と「2倍してから1を足す」処理は一致しません。

\begin{listing}[htbp]
\begin{minted}{lean4}
def double (n : Nat) : Nat := n * 2

def addOne (n : Nat) : Nat := n + 1

example : double (addOne 3) ≠ addOne (double 3) := by
  decide
\end{minted}
\caption{\texttt{double}・\texttt{addOne}}
\label{lst:ch34-order-not-always-safe}
\end{listing}

この小例は、処理順序変更のレビューで見るべき点をよく表しています。
すべての順序変更が危険なのではありません。
必要な法則がある順序変更だけが安全です。

\section{共通化は分岐の境界を明示する}

共通化も、よくあるリファクタリングです。
似たような関数が二つあり、重複している部分を一つの共通関数へ寄せます。
このとき注意すべきなのは、共通化により「今まで別々だった仕様」が暗黙に混ざることです。

たとえば、クーポンなしの計算とクーポンありの計算を共通関数へ寄せるとします。
クーポンなしの旧関数が、共通関数の \texttt{false} 分岐と一致することを示せれば、その分岐については意味保存を確認できます。

\begin{listing}[htbp]
\begin{minted}{lean4}
def discountCore (useCoupon : Bool) (price : Nat) : Nat :=
  if useCoupon then price - 10 else price

def oldWithoutCoupon (price : Nat) : Nat :=
  price

def newWithoutCoupon (price : Nat) : Nat :=
  discountCore false price

theorem commonize_without_coupon_preserves (price : Nat) :
    newWithoutCoupon price = oldWithoutCoupon price := by
  rfl
\end{minted}
\caption{\texttt{commonize\_without\_coupon\_preserves}}
\label{lst:ch34-commonize-preserves}
\end{listing}

この例では、\texttt{false} 分岐が旧実装と同じであることだけを示しています。
クーポンありの場合の仕様は、別に検討する必要があります。
共通化の証明では、すべてを一つの大きな定理に押し込めるより、分岐ごとに小さな定理へ分けるほうがレビューしやすくなります。

共通化のレビューでは、「共通関数が正しいか」だけでなく、「旧関数 A は共通関数のどの引数、どの分岐に対応するか」「旧関数 B はどの分岐に対応するか」を明示するとよいでしょう。
Lean では、それぞれを別々の定理にすることで、対応関係が読みやすくなります。

\section{不要なwrapperの削除は恒等処理の除去である}

adapter や wrapper を削除するリファクタリングも、等式として扱えます。
特に、その wrapper が実質的に何もしない処理であるなら、圏論でいう恒等射、すなわち恒等関数に近いものです。

\begin{listing}[htbp]
\begin{minted}{lean4}
def identityAdapter (x : Nat) : Nat := x

def wrappedNormalize (x : Nat) : Nat :=
  identityAdapter x

def directNormalize (x : Nat) : Nat :=
  x

theorem remove_identity_wrapper_preserves (x : Nat) :
    wrappedNormalize x = directNormalize x := by
  rfl
\end{minted}
\caption{\texttt{remove\_identity\_wrapper\_preserves}}
\label{lst:ch34-remove-wrapper}
\end{listing}

ここでのポイントは、wrapper が本当に恒等的かどうかです。
ログを出す、メトリクスを送る、例外を捕捉する、型変換時に正規化するといった処理が入っているなら、それは「何もしない」処理ではありません。
削除してよいかどうかは、何を観測対象にするかによって変わります。

「戻り値が同じ」だけでは十分でないことがあります。
ログ、監査証跡、メトリクス、キャッシュ、DB 更新などが仕様の一部なら、それらも出力や状態としてモデルに入れる必要があります。
Lean モデルで観測しないものは、Lean の証明では保証されません。

この注意は、チーム開発で特に重要です。
証明は強力ですが、証明したモデルの範囲を超えては語れません。
wrapper 削除の安全性を説明するときは、「戻り値については等しい」「ログについてはこのモデルでは扱っていない」のように、保証範囲を明示します。

\section{モナド則による effectful code の整理}

失敗を含む処理では、普通の関数合成だけでは足りません。
\texttt{Option}、\texttt{Except}、\texttt{Result} のような型を使うと、成功した場合だけ次の処理へ進み、失敗したらそこで止まるという形になります。
このような処理の合成には、\texttt{bind} が現れます。

実務コードでは、次のようなリファクタリングが行われます。

\begin{itemize}
\item ネストした失敗処理を、平坦な \texttt{bind} の連鎖にします。
\item 途中の一時変数を消して、処理をパイプライン化します。
\item 不要な \texttt{pure} や \texttt{some} を取り除きます。
\item 括弧の位置を変えて読みやすくします。
\end{itemize}

これらを安全に行うための考え方が、モナド則です。
本章では一般のモナドを形式化せず、\texttt{Option} だけを使って、括弧の付け替えが意味を変えないことを見ます。

\FigurePlaceholder{fig-ch34-option-bind-assoc.png}{0.88}{\texttt{Option.bind} の結合律}{fig:ch34-option-bind-assoc}

図\ref{fig:ch34-option-bind-assoc}の二つの経路は、失敗を伝播させながら同じ三処理を異なる括弧付けで実行します。
次の例では、正の値か確認し、偶数か確認し、最後に値を増やします。
変更前は、最初の \texttt{bind} の中に次の \texttt{bind} がネストしています。
変更後は、先に二つの検査を合成してから、最後の処理をつないでいます。

\begin{listing}[htbp]
\begin{minted}{lean4}
def parsePositive (n : Nat) : Option Nat :=
  if n = 0 then none else some n

def checkEven (n : Nat) : Option Nat :=
  if n % 2 = 0 then some n else none

def enrich (n : Nat) : Option Nat :=
  some (n + 1)

def flowBefore (n : Nat) : Option Nat :=
  Option.bind (parsePositive n) (fun x =>
    Option.bind (checkEven x) enrich)

def flowAfter (n : Nat) : Option Nat :=
  Option.bind (Option.bind (parsePositive n) checkEven) enrich
\end{minted}
\caption{\texttt{flowBefore}・\texttt{flowAfter}}
\label{lst:ch34-option-assoc}
\end{listing}

この時点では、二つの定義が同じかどうかはまだ主張していません。
次に、\texttt{parsePositive n} が \texttt{none} か \texttt{some x} かで場合分けし、さらに \texttt{checkEven x} も場合分けします。
すべてのケースで同じ結果になることを確認します。

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem option_assoc_refactor_preserves (n : Nat) :
    flowBefore n = flowAfter n := by
  unfold flowBefore flowAfter
  cases parsePositive n with
  | none => rfl
  | some x =>
      cases checkEven x with
      | none => rfl
      | some y => rfl
\end{minted}
\caption{\texttt{option\_assoc\_refactor\_preserves}}
\label{lst:ch34-option-assoc-proof}
\end{listing}

この証明は、\texttt{Option} の結合律を一般定理として呼び出しているわけではありません。
失敗する場合と成功する場合を展開し、どちらの書き方でも同じ結果になることをケースごとに確認しています。
初学者向けには、抽象的なモナド則よりも、このような具体的な場合分けから読むほうが理解しやすくなります。

\texttt{pure} に相当する \texttt{some} についても、小さな等式として確認できます。
値を \texttt{some} に入れてすぐ \texttt{bind} で次の関数へ渡すことは、直接その関数を呼ぶことと同じです。
また、\texttt{Option} 値を \texttt{some} へ渡すだけの \texttt{bind} は、元の値を変えません。

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem option_bind_pure_left (x : Nat) (f : Nat -> Option Nat) :
    Option.bind (some x) f = f x := by
  rfl

theorem option_bind_pure_right (mx : Option Nat) :
    Option.bind mx some = mx := by
  cases mx with
  | none => rfl
  | some x => rfl
\end{minted}
\caption{\texttt{option\_bind\_pure\_left}・\texttt{option\_bind\_pure\_right}}
\label{lst:ch34-option-pure-laws}
\end{listing}

これらの等式は、effectful code のリファクタリング規則として読めます。
不要な \texttt{some} を消す、\texttt{bind} の括弧を変える、失敗処理を読みやすく平坦化するといった変更が該当します。
こうした変更は、法則が成り立つ範囲では意味を変えません。

\section{Lean 4で小さなリファクタリング証明を書く手順}

実務のリファクタリングを Lean で扱うときは、最初から本番コード全体を移植しません。
まず、意味保存を確認したい核だけを小さなモデルにします。

推奨する手順は、次のとおりです。
\begin{enumerate}
\item 変更前の処理を小さな関数として書きます。
\item 変更後の処理を別の小さな関数として書きます。
\item 保存したい性質を、戻り値の等式、関数の等式、状態の等式、または成功結果と失敗結果の等式として選びます。
\item まず \texttt{rfl} や \texttt{simp} で通る形を探します。
\item 通らない場合は、必要な補題を作り、\texttt{rw} や \texttt{calc} で段階的に示します。
\item 分岐を含む場合は、\texttt{cases} で全ケースを明示します。
\end{enumerate}

次の表は、よくあるリファクタリングと Lean での見方を対応させています。

\small
\setlength{\tabcolsep}{3pt}
\begin{longtable}{@{}p{0.22\linewidth}p{0.36\linewidth}p{0.36\linewidth}@{}}
\toprule
リファクタリング & Leanでの表現 & 主な証明道具 \\
\midrule
関数抽出 & 補助関数を展開すると同じ式になる & \texttt{rfl}, \texttt{unfold}, \texttt{simp} \\
処理順序の変更 & \texttt{g (f x) = f (g x)} を示す & \texttt{calc}, \texttt{rw}, 結合律, 交換律 \\
共通化 & 旧関数が共通関数の特定分岐と一致する & \texttt{rfl}, \texttt{cases}, 分岐ごとの補題 \\
wrapper削除 & 恒等的な処理を消す & \texttt{rfl}, 関数外延性 \\
失敗処理の整理 & \texttt{bind} の括弧や \texttt{pure} を整理する & \texttt{cases}, モナド則 \\
\bottomrule
\end{longtable}

ここで重要なのは、証明対象を「変更全体」ではなく「保存したい意味」にすることです。
たとえば、巨大な注文処理全体を Lean で再実装する必要はありません。
合計金額の計算だけ、ID 保存だけ、失敗時に状態が変わらないことだけを切り出しても構いません。

\section{証明結果を実装へ適用する}

Lean で証明したことは、モデルの範囲では強い保証になります。
しかし、モデルと本番実装の間には必ず差分があります。
そこを曖昧にすると、証明の価値も危うくなります。

「Lean で証明した」と述べるときは、何を証明したのかを明記します。
たとえば、「自然数モデル上の合計金額について、関数抽出前後の戻り値が等しい」「\texttt{Option} でモデル化した失敗処理について、\texttt{bind} の括弧付け変更が結果を変えない」のように書きます。
現実の DB、例外、並行性、時刻、I/O、丸め誤差まで自動的に保証されるわけではありません。

実務では、次の問いを設計レビューに入れるとよいでしょう。

\begin{itemize}
\item 変更前後で保存したい観測結果は何でしょうか。
\item その観測結果は、戻り値だけでしょうか、それともログや状態も含むのでしょうか。
\item Lean モデルでは、外部依存をどの引数として表すのでしょうか。
\item 証明した性質と通常テストで見る性質を、どのように分担するのでしょうか。
\item 証明が壊れたとき、それは仕様変更でしょうか、それとも実装ミスでしょうか。
\end{itemize}

\section{本章のまとめ}

リファクタリングの安全性は、変更前後の意味保存として等式で表せます。

関数抽出は、補助定義を展開すると元の式へ戻ることとして確認できます。

処理順序の変更には、交換できる理由が必要です。
足し算のように法則がある場合は、\texttt{calc} と \texttt{rw} で根拠を示せます。

共通化では、旧関数が共通関数のどの分岐に対応するかを明示します。

wrapper 削除は、wrapper が本当に恒等的である範囲でだけ安全です。

\texttt{Option.bind} の括弧付け変更や \texttt{some} の整理は、effectful code のリファクタリング規則として読めます。

Lean で証明するのは、本番コード全体ではなく、壊れると困る小さな意味保存性で十分です。
