\begin{chapterabstract}
本章から圏論に入ります。
最初に扱うのは、圏という最も基本的な概念です。
圏とは、対象と射があり、射を合成でき、何もしない射がある世界です。
本章では、一般圏論のすべてを一度に形式化せず、まず「型と関数の世界」を使って、対象、射、恒等射、合成、恒等律、結合律を理解します。
Lean 4 では、恒等関数と関数合成を小さく定義し、パイプラインの括弧の付け方を変えても結果が変わらないことを確認します。
\end{chapterabstract}

\begin{learninggoals}
\item 圏を「合成できる処理の世界」として説明できます。
\item 対象、射、恒等射、合成を、型、関数、何もしない処理、パイプラインとして読み替えられます。
\item 恒等律と結合律を、リファクタリングやパイプライン設計の安全性として理解できます。
\item Lean 4 で恒等関数、関数合成、合成の法則を小さな定理として書けます。
\end{learninggoals}

\section{処理の合成に注目する}

ソフトウェアの処理が、一つの操作だけで完結することは多くありません。
たとえば、入力を検証し、正規化し、変換し、保存してから、表示用に整えます。
API gateway では request を受け取り、認証し、ログを付けて、内部サービスへ流します。
ETL では extract、transform、load を順に並べます。
これらはすべて「処理をつなぐ」作業です。

圏論は、この「つなげる」という行為を中心に据えます。
ここで注目するのは、個々の処理の内部を細かく調べることではありません。
どの処理をつなげられるか、つないだ結果が何になるか、何もしない処理を挟んでも意味が変わらないか、三つ以上をつなぐときに括弧の付け方で意味が変わらないかを調べます。

図\ref{fig:ch11-category-overview}では、型を点、関数を矢印として配置し、恒等関数と、型が接続する関数合成を同じネットワーク上に示します。

\FigurePlaceholder{fig-ch11-category-overview.png}{0.90}{型と関数の圏}{fig:ch11-category-overview}

本章では、最も身近な例として「型と関数の世界」を使います。
たとえば、\texttt{String}、\texttt{Nat}、\texttt{Bool} は対象の例です。
\texttt{String} から \texttt{Nat} への関数や、\texttt{Nat} から \texttt{String} への関数は射の例です。
二つの関数の出力型と入力型が合えば、それらを合成できます。

\section{ソフトウェア上の困りごと：処理を分けると意味が壊れるのか}

次のような処理を考えます。
文字列で届いたユーザーIDを解釈し、ユーザー名を取り出して、挨拶文を作ります。
実際のシステムでは DB や外部 API が関わるかもしれませんが、ここでは単純な関数として扱います。

処理を分割すると、レビューでは次の問いが生じます。
「処理を三つに分けたが、元の意味は保たれているのか」
「先に前半をまとめるか、後半をまとめるかで結果は変わらないのか」
「何もしない adapter や middleware を挟んでも本当に影響はないのか」
圏の法則は、これらの問いを整理するための最小限の語彙になります。

たとえば、次の三つの処理を考えます。

\begin{itemize}
\item \texttt{parseUserId : String -> Nat}
\item \texttt{fetchUserName : Nat -> String}
\item \texttt{renderGreeting : String -> String}
\end{itemize}

これらは型が接続するため、順に合成できます。
\texttt{parseUserId} の出力と \texttt{fetchUserName} の入力は、どちらも \texttt{Nat} です。
同様に、\texttt{fetchUserName} の出力と \texttt{renderGreeting} の入力は、どちらも \texttt{String} です。

ここで、前半と後半のどちらを先にまとめるかを考えます。

\[
  (\texttt{renderGreeting} \circ \texttt{fetchUserName}) \circ \texttt{parseUserId}
\]

\[
  \texttt{renderGreeting} \circ (\texttt{fetchUserName} \circ \texttt{parseUserId})
\]

この二つは、どちらも「IDを解釈し、名前を取り出し、挨拶文を作る」という順序で実行されます。
括弧は、どこを先に一まとまりの部品として見るかを表すだけであり、データが流れる順序は変えません。
これが結合律のソフトウェア上の意味です。

\section{対象を型、射を関数として見る}

圏には対象と射があります。
対象は点のようなものであり、射は対象から対象へ向かう矢印です。
ただし、圏論では、対象そのものの中身をのぞき込むことよりも、対象どうしを結ぶ射と、その射をどう合成するかに注目します。

圏の基本要素は次のとおりです。
\begin{itemize}
\item \textbf{対象}：矢印の始点や終点になるものです。
\item \textbf{射}：ある対象から別の対象へ向かう矢印です。
\item \textbf{恒等射}：各対象にある、何もしない射です。
\item \textbf{合成}：接続できる二つの射から、新しい射を作る操作です。
\end{itemize}
本章では、対象を型、射を関数として読みます。

型と関数の世界では、対象は \texttt{String} や \texttt{Nat} のような型です。
射は、\texttt{String -> Nat} のような関数です。
関数 \(f : A \to B\) は、対象 \(A\) から対象 \(B\) への射として読めます。
このとき、\(A\) を始域、\(B\) を終域と呼ぶことがあります。

\begin{table}[htbp]

普段扱っている型、関数、変換、パイプラインを、同じ構造として見られるようにすることです。
同型、関手、自然変換も、対象、射、合成を基礎として定義されます。

\section{恒等射は何もしない処理}

各対象には恒等射があります。
対象 \(A\) の恒等射は、\(A\) から \(A\) へ向かい、何も変えない射です。
型と関数の世界では、恒等関数に相当します。

\[
  \mathrm{id}_A : A \to A
\]

任意の値 \(x : A\) に対して、\(\mathrm{id}_A(x) = x\) が成り立ちます。
つまり、入力をそのまま返します。

恒等射は「何も書かない」ことではない点に注意してください。
これは、「入力をそのまま返す」ことを明示した射です。
プログラム上では、\texttt{id} や \texttt{fun x => x} に相当します。
入力をそのまま返す middleware や、無効化された adapter も、このモデルでは恒等射として読めます。

\section{合成はパイプライン}

射 \(f : A \to B\) と \(g : B \to C\) があるとします。
\(f\) の終域と \(g\) の始域が同じなので、\(f\) の後に \(g\) を実行できます。
この合成を \(g \circ f : A \to C\) と書きます。

\[
  (g \circ f)(x) = g(f(x))
\]

右辺では、\(f\) が先に実行され、その結果に \(g\) が適用されます。
表記上は \(g \circ f\) と \(g\) が左に来ますが、処理順序は \(f\) から \(g\) です。
これは最初につまずきやすい点です。

\(g \circ f\) は、「先に \(f\)、次に \(g\)」と読みます。
日本語で左から右にパイプラインを読む感覚とは逆に見えることがあります。
本書では、式には標準的な \(g \circ f\) を使い、説明では常に「先に \(f\)、次に \(g\)」と読みます。

図\ref{fig:ch11-composition-order}では、入力がまず (f) を通り、その出力が (g) の入力になることを矢印の向きで確認できます。

\FigurePlaceholder{fig-ch11-composition-order.png}{0.90}{合成の順序}{fig:ch11-composition-order}

合成を使うと、小さな部品を大きな部品として扱えます。
\(f : A \to B\) と \(g : B \to C\) を合成した \(g \circ f : A \to C\) も、一つの射です。
この「合成したものも同じ種類の部品になる」という性質が、圏論をソフトウェア設計に結び付けます。
パイプラインの一部をまとめて名前を付け、さらに別の処理と合成できるためです。

\section{圏の二つの法則}

圏には、恒等射と合成があるだけでは不十分です。
それらが自然に振る舞うための法則も必要です。
本章では、恒等律と結合律という二つの法則を使います。

\subsection{恒等律}

射 \(f : A \to B\) に、\(A\) 側または \(B\) 側の恒等射を合成しても、\(f\) は変わりません。

\[
  f \circ \mathrm{id}_A = f
\]

\[
  \mathrm{id}_B \circ f = f
\]

ソフトウェア上では、入力側または出力側に no-op を挟んでも、全体の振る舞いは変わらないという意味です。
この法則は、不要な adapter や wrapper を取り除くリファクタリングの基礎になります。

\subsection{結合律}

射 \(f : A \to B\)、\(g : B \to C\)、\(h : C \to D\) があるとします。
このとき、三つの射の合成には、括弧の付け方が二通りあります。

\[
  (h \circ g) \circ f
\]

\[
  h \circ (g \circ f)
\]

結合律は、この二つが等しいことを主張します。

\[
  (h \circ g) \circ f = h \circ (g \circ f)
\]

実行順序は、どちらも \(f\)、\(g\)、\(h\) の順です。
異なるのは、どこを先に一つの部品としてまとめるかだけです。

図\ref{fig:ch11-associativity}では、三つの射の順序を (f,g,h) のまま保ち、前半と後半のどちらを先に一つの射としてまとめるかだけを変えます。

\FigurePlaceholder{fig-ch11-associativity.png}{0.90}{合成の結合律}{fig:ch11-associativity}

結合律は「処理の順序を入れ替えてよい」とは主張していません。
\(f\)、\(g\)、\(h\) の順序を保ったまま、括弧だけを変えてよいと主張しています。
順序を入れ替えてよいかどうかは別の性質であり、一般には成り立ちません。

\section{Lean 4 で関数合成の法則を確認する}

Lean では、型と関数をそのまま扱えます。
本章では Mathlib の高度な圏論ライブラリを使いません。
恒等関数と合成を自分で小さく定義し、何が起きているかを見える形にします。

ここでの Lean コードは、一般の圏を定義するものではありません。
まず、型を対象、関数を射とする特別な世界で、恒等射と合成がどのように見えるかを確認します。
これを厳密に一つの圏として定式化するには、対象にする型の universe level を管理する必要があります。
一方、本章のコードはその構造を包装せず、関数合成の法則だけを点ごとに検査します。
一般の圏や Mathlib の \texttt{CategoryTheory} API は、発展章で扱います。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter11

-- 対象を型、射を関数として見る。
-- idFn は「何もしない処理」である。
def idFn {A : Type} (x : A) : A :=
  x

-- comp g f は「先に f、次に g」を実行する合成である。
def comp {A B C : Type} (g : B -> C) (f : A -> B) : A -> C :=
  fun x => g (f x)

end Chapter11
\end{minted}
\caption{\texttt{idFn} と \texttt{comp}}
\label{lst:ch11_id_comp}
\end{listing}

\texttt{comp g f} は、数学の \(g \circ f\) に対応します。
入力 \texttt{x} に対して、まず \texttt{f x} を計算し、その結果に \texttt{g} を適用します。
したがって、\texttt{comp g f x} は \texttt{g (f x)} と同じです。

次に、恒等律と結合律を「任意の入力 \texttt{x} に対して結果が同じ」という形で書きます。
恒等律と結合律を、各入力で出力が一致する点ごとの等式として確認します。
関数そのものの等式は、関数外延性によってこの点ごとの等式から得られます。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter11

-- 左から恒等処理を合成しても、結果は変わらない。
theorem id_left_pointwise {A B : Type} (f : A -> B) (x : A) :
    comp idFn f x = f x := by
  rfl

-- 右から恒等処理を合成しても、結果は変わらない。
theorem id_right_pointwise {A B : Type} (f : A -> B) (x : A) :
    comp f idFn x = f x := by
  rfl

-- 三つの関数を合成するとき、括弧の付け方は結果に影響しない。
theorem comp_assoc_pointwise {A B C D : Type}
    (h : C -> D) (g : B -> C) (f : A -> B) (x : A) :
    comp (comp h g) f x = comp h (comp g f) x := by
  rfl

end Chapter11
\end{minted}
\caption{\texttt{id\_left\_pointwise}・\texttt{id\_right\_pointwise}・\texttt{comp\_assoc\_pointwise}}
\label{lst:ch11_laws}
\end{listing}

ここで使う \texttt{rfl} は、両辺を定義に従って展開すると同じ形になることを Lean に確認させる道具です。
たとえば、\texttt{comp idFn f x} を展開すると \texttt{idFn (f x)} になり、さらに \texttt{f x} になります。
したがって、証明は \texttt{rfl} で終わります。

\section{ミニケース：パイプラインの括弧を変えても意味は変わらない}

ここからは、ソフトウェアに近い小さなパイプラインを Lean で書きます。
次の例では、本物の DB や API を使いません。
証明の対象が外部システムの挙動ではなく、関数合成としてのパイプライン構造だからです。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter11

-- 例を小さくするため、文字列の長さをユーザーIDの代わりに使う。
def parseUserId (s : String) : Nat :=
  s.length

def fetchUserName (id : Nat) : String :=
  "user-" ++ toString id

def renderGreeting (name : String) : String :=
  "hello, " ++ name

-- 前半を後でまとめる見方。
def pipelineLeft : String -> String :=
  comp (comp renderGreeting fetchUserName) parseUserId

-- 後半を先にまとめる見方。
def pipelineRight : String -> String :=
  comp renderGreeting (comp fetchUserName parseUserId)

#eval pipelineLeft "abc"  -- => "hello, user-3"

end Chapter11
\end{minted}
\caption{\texttt{pipelineLeft} と \texttt{pipelineRight}}
\label{lst:ch11_pipeline}
\end{listing}

\texttt{pipelineLeft} と \texttt{pipelineRight} の違いは、括弧の付け方だけです。
どちらも、先に \texttt{parseUserId}、次に \texttt{fetchUserName}、最後に \texttt{renderGreeting} を実行します。
そのため、任意の入力文字列 \texttt{s} に対して結果が一致します。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter11

-- 点ごとに見ると、両方のパイプラインは同じ式へ展開される。
theorem pipeline_assoc_pointwise (s : String) :
    pipelineLeft s = pipelineRight s := by
  rfl

-- calc を使うと、レビュー用の変形ログとして読める。
theorem pipeline_review_log (s : String) :
    pipelineLeft s = pipelineRight s := by
  calc
    pipelineLeft s
        = renderGreeting (fetchUserName (parseUserId s)) := by rfl
    _   = pipelineRight s := by rfl

end Chapter11
\end{minted}
\caption{\texttt{pipeline\_assoc\_pointwise} と \texttt{pipeline\_review\_log}}
\label{lst:ch11_pipeline_proof}
\end{listing}

\texttt{pipeline\_review\_log} は、数学的には短い証明です。
実務上は、「変更前後のパイプラインは、どちらも同じ中間式に展開される」というレビュー記録として読めます。
このような小さな証明は、リファクタリングの説明や設計変更の説明に適しています。

\section{恒等処理の除去を証明する}

恒等律は、no-op を取り除くときに使えます。
たとえば、何もしない文字列変換 \texttt{noOpString} をパイプラインの末尾に挟んだとします。
この処理は入力をそのまま返すため、取り除いても結果は変わりません。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter11

-- 何もしない文字列処理。
def noOpString : String -> String :=
  idFn

-- no-op を最後に挟んだパイプライン。
def pipelineWithNoOp : String -> String :=
  comp noOpString pipelineLeft

-- 任意の入力で、no-op ありと no-op なしの結果は一致する。
theorem remove_noop_pointwise (s : String) :
    pipelineWithNoOp s = pipelineLeft s := by
  rfl

end Chapter11
\end{minted}
\caption{\texttt{remove\_noop\_pointwise}}
\label{lst:ch11_noop}
\end{listing}

この例は小さいものの、設計上は重要です。
処理の入口や出口に、将来の拡張用 adapter を置くことがあります。
その adapter が本当に何もしないなら、全体の意味は変わりません。
Lean では、その主張を「すべての入力で結果が一致する」という形で記述できます。

図\ref{fig:ch11-noop-removal}では、\texttt{pipelineLeft} の後ろに恒等処理を合成した経路と、合成しない経路が同じ出力へ至ることを示します。

\section{圏を定義として見る}

圏、恒等射、合成、結合律の標準的な定義と記法については、Leinster~\cite{leinster-basic-category-theory}およびRiehl~\cite{riehl-category-theory-in-context}を基準とします。
本章の型と関数の例はその特殊例であり、一般の圏の射が関数であるとは限りません。

ここまでは、型と関数の世界で対象、射、合成、恒等射を見てきました。
ここで、圏の定義を少しだけ一般化して眺めます。

圏とは、次のデータと法則を持つ構造です。
\begin{itemize}
\item 対象の集まり
\item 各対象 \(A, B\) に対する、\(A\) から \(B\) への射の集まり
\item 各対象 \(A\) に対する恒等射 \(\mathrm{id}_A : A \to A\)
\item 射 \(f : A \to B\) と \(g : B \to C\) から射 \(g \circ f : A \to C\) を作る合成
\item 恒等律と結合律
\end{itemize}

この定義は、最初は抽象的に見えるかもしれません。
しかし、型と関数の世界を思い出すと、各項目を自然に読めます。
型が対象、関数が射、恒等関数が恒等射、関数合成が合成に対応します。
恒等律と結合律は、関数合成について成り立つ基本的な法則です。

本章では、一般の圏を Lean の構造体として完全に定義するところまでは進みません。
理由は二つあります。
第一に、依存型、universe、型クラスが同時に現れると、初学者の負担が大きくなります。
第二に、本書では、まず実務上の変換と合成を読めるようにすることを目指します。
そのため、当面は「型と関数の圏」を手で触りながら理解します。

「圏」は「型と関数」だけを意味するわけではありません。
順序、状態遷移、データベーススキーマ、仕様の強弱、証明の変換など、さまざまなものを圏として見られます。
ただし、本章では最初の足場として、型と関数に限定します。
この限定により、対象、射、合成、恒等射を具体的に確認できます。

\section{実務への読み替え}

圏の法則は、実務上の設計判断に直接対応します。
本章では、特に三つの読み替えを押さえます。

\subsection{処理の境界を型で見る}

射を合成できるかどうかは、終域と始域が合うかどうかで決まります。
ソフトウェアでは、出力型と入力型が合うかどうかに対応します。
型が合わなければ、adapter が必要です。
adapter を入れる場合は、その adapter が何を保存するのかを明確にする必要があります。

\subsection{no-op を設計上の部品として扱う}

何もしない処理が、設計上も無意味とは限りません。
ログをまだ入れていない middleware、将来の互換層、設定で無効化されたフィルタなど、構造上の場所を確保するために no-op を使うことがあります。
それらが本当に no-op である限り、前後に合成しても意味が変わらないことを恒等律が保証します。

\subsection{括弧の変更と順序変更を区別する}

結合律は括弧の変更を許しますが、順序の変更は許しません。
\texttt{validate -> normalize -> save} と \texttt{normalize -> validate -> save} は、一般には同じではありません。
一方、\texttt{(validate -> normalize) -> save} と \texttt{validate -> (normalize -> save)} は、処理順序が同じなので、同じ全体処理として扱えます。

\section{よくある誤解}

\subsection{誤解1：圏論は難しい対象の中身を調べる理論である}

圏論は、対象の中身だけを見る理論ではありません。
対象どうしの関係、すなわち射と合成に注目します。
型の内部表現をすべて知ることよりも、その型からどこへ変換できるか、どの変換を合成できるかを重視します。

\subsection{誤解2：結合律があるなら処理を自由に並べ替えられる}

結合律は、順序を変えてよいとは主張していません。
\((h \circ g) \circ f\) と \(h \circ (g \circ f)\) は同じですが、\(g \circ f\) と \(f \circ g\) は型が合わないことすらあります。
型が合ったとしても、結果が同じとは限りません。

\subsection{誤解3：恒等射は実務では役に立たない}

恒等射は何もしないため、軽く見られがちです。
しかし、何もしない処理が「本当に何もしない」と言えることには意味があります。
互換層、デフォルト設定、無効化されたフィルタを安全に取り除けるかどうかは、恒等律として考えられます。

\section{本章のまとめ}

対象、射、恒等射、合成と二つの法則が、圏の最小語彙です。
\(g \circ f\) は、先に \(f\)、次に \(g\) と読みます。
型と純粋関数は重要な例ですが、一般の圏の対象を型、射を関数と無条件に同一視してはいけません。
