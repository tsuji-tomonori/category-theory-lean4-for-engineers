\chapter{Lean 4 最小チートシート}
\label{chap:appA-lean4-min-cheatsheet}
\BookIndex{Lean 4 ちーとしーと}{Lean 4 チートシート}

\begin{chapterabstract}
この付録は、本書の本文で繰り返し使う Lean 4 の最小語彙を、仕様、変更、証明の文脈から参照できるように整理したチートシートです。
目的は、Lean の全機能を網羅することではありません。
読者が本文のコードを読み返すときに、「これは何をしている構文か」「ソフトウェア上の何に対応するか」「どの証明で使うか」をすぐに確認できるようにします。

本書では Lean 4 を、実装そのものを置き換える言語ではなく、仕様の核を小さく切り出して検査するための言語として使います。
そのため、この付録でも、型、関数、構造体、命題、証明、場合分け、帰納法という最小限の道具に集中します。
\end{chapterabstract}

\begin{learninggoals}
\item \texttt{def}、\texttt{structure}、\texttt{inductive} の役割を、仕様モデルを書くための道具として説明できるようになります。
\item \texttt{example}、\texttt{theorem}、\texttt{by} を使った最小の証明の形を読めるようになります。
\item \texttt{rfl}、\texttt{simp}、\texttt{rw}、\texttt{calc} を、等式による意味保存の道具として使い分けられるようになります。
\item \texttt{cases} と \texttt{induction} を、全パターンの検査と任意長データの検査として読めるようになります。
\item \texttt{constructor}、\texttt{exact}、\texttt{apply} を、証明を組み立てる基本操作として理解できます。
\item \texttt{intro}、\texttt{unfold}、\texttt{decide}、\texttt{funext}、\texttt{<;>} の役割を、短い証明から確認できます。
\end{learninggoals}

\section{使い方}

この付録は、最初から順番に暗記するための表ではありません。
本文を読んでいて Lean コードで行き詰まったときに、戻ってくるための場所です。
特に、次の三つの読み方を意識するとよいでしょう。

\begin{intuitionbox}
Lean の構文は、単なる記号ではなく、「何を定義しているか」「何を主張しているか」「どの根拠で主張を閉じているか」を分けるための印です。
\end{intuitionbox}

\begin{softwarebox}
ソフトウェア設計に置き換えると、\texttt{def} は関数や変換、\texttt{structure} は DTO やドメインモデル、\texttt{theorem} は設計上守りたい仕様、\texttt{by} 以下はレビュー可能な根拠に対応します。
\end{softwarebox}

図\ref{fig:appA-lean4-cheatsheet-map}では、この付録の語彙を四つの領域に分けています。
値や関数を作る語彙、仕様を主張する語彙、等式を閉じる語彙、データの形を分解する語彙です。

\section{値・型・関数を書く}

\subsection{\texttt{def}：名前を付けて値や関数を定義する}

\begin{definitionbox}
\texttt{def} は、値や関数に名前を付ける構文です。
本書では、仕様モデル上の処理、データ変換、計算ロジックを小さく定義するときに使います。
\end{definitionbox}

たとえば、価格に手数料を足す関数や、二つの処理をつないだパイプラインは、次のように書けます。

\begin{listing}[htbp]
\begin{minted}{lean4}
def serviceFee : Nat := 300

def addFee (price : Nat) : Nat :=
  price + serviceFee

def inc (n : Nat) : Nat :=
  n + 1

def double (n : Nat) : Nat :=
  n * 2

def pipeline (n : Nat) : Nat :=
  double (inc n)

#eval addFee 1000  -- => 1300
#eval pipeline 10  -- => 22
\end{minted}
\caption{\texttt{serviceFee}・\texttt{addFee}・\texttt{pipeline}}
\label{lst:appA_def}
\end{listing}

\begin{leanbox}
\texttt{Nat} は自然数の型です。
\texttt{price : Nat} は、「\texttt{price} という名前の入力は \texttt{Nat} 型である」と読みます。
\texttt{pipeline} は \texttt{inc} の結果を \texttt{double} に渡す関数であり、本文で扱うパイプライン合成の最小例です。
\end{leanbox}

\subsection{\texttt{\#eval}：小さく動かして確認する}

\texttt{\#eval} は、定義した値や関数をその場で評価するための命令です。
証明ではありませんが、モデルを作っている途中で意図を確認するのに役立ちます。

\begin{warningbox}
\texttt{\#eval} はテストに近い確認であり、すべての入力に対する保証ではありません。
すべての場合について保証したい性質は、後で \texttt{theorem} として書きます。
\end{warningbox}

\section{データモデルを書く}

\subsection{\texttt{structure}：フィールドを持つ型を作る}

\begin{definitionbox}
\texttt{structure} は、複数のフィールドを持つデータ型を定義する構文です。
ソフトウェアでは、レコード、DTO、設定オブジェクト、ドメインモデルに対応します。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
structure UserV1 where
  id : Nat
  name : String

structure UserV2 where
  userId : Nat
  displayName : String

def migrateUser (u : UserV1) : UserV2 :=
  { userId := u.id, displayName := u.name }

def rollbackUser (u : UserV2) : UserV1 :=
  { id := u.userId, name := u.displayName }
\end{minted}
\caption{\texttt{UserV1}・\texttt{UserV2}・\texttt{migrateUser}・\texttt{rollbackUser}}
\label{lst:appA_structure}
\end{listing}

\begin{softwarebox}
\texttt{UserV1} から \texttt{UserV2} への変換は、スキーマ移行の最小モデルです。
フィールド名は変わっていますが、保持している情報が同じなら、後で round-trip 性質を証明できます。
\end{softwarebox}

\subsection{フィールドアクセス}

構造体の値 \texttt{u : UserV1} から \texttt{id} フィールドを取り出すには、\texttt{u.id} と書きます。
これは、多くのプログラミング言語のドットアクセスと同じ感覚で読めます。

\begin{listing}[htbp]
\begin{minted}{lean4}
def sameIdAfterMigrate (u : UserV1) : Nat :=
  (migrateUser u).userId
\end{minted}
\caption{\texttt{sameIdAfterMigrate}}
\label{lst:appA_field_access}
\end{listing}

このコードでは、\texttt{migrateUser u}で作った新形式の値から、\texttt{.userId}でフィールドを取り出しています。
構造体の変換を読むときは、どのフィールドがどの名前へ移ったかを、このような小さな関数で確認するとよいでしょう。

\section{分岐を持つ型を書く}

\subsection{\texttt{inductive}：値の形を列挙する}

\begin{definitionbox}
\texttt{inductive} は、値が取り得る形を列挙して、新しい型を作る構文です。
成功か失敗か、左右のどちらか、空か追加かといった分岐を表すときに使います。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
inductive Result (A : Type) where
  | ok : A -> Result A
  | err : String -> Result A

def validatePositive (n : Nat) : Result Nat :=
  if n = 0 then
    Result.err "zero is not positive"
  else
    Result.ok n
\end{minted}
\caption{\texttt{Result}・\texttt{validatePositive}}
\label{lst:appA_inductive}
\end{listing}

\begin{leanbox}
\texttt{Result.ok} と \texttt{Result.err} は、この型の値を作る方法です。
後で \texttt{cases} を使うと、成功ケースと失敗ケースを漏れなく分解できます。
\end{leanbox}

\section{命題と証明を書く}

\subsection{\texttt{example} と \texttt{theorem}：性質を主張する}

\begin{definitionbox}
\texttt{example} と \texttt{theorem} は、Lean で命題を主張するための構文です。
\texttt{example} は名前を付けない一時的な例であり、\texttt{theorem} は名前を付けて再利用できる定理です。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
example : inc 0 = 1 := by
  rfl

theorem inc_eq_add_one (n : Nat) :
    inc n = n + 1 := by
  rfl

theorem rollback_migrate_user (u : UserV1) :
    rollbackUser (migrateUser u) = u := by
  rfl
\end{minted}
\caption{\texttt{inc\_eq\_add\_one}・\texttt{rollback\_migrate\_user}}
\label{lst:appA_example_theorem}
\end{listing}

\begin{softwarebox}
\texttt{rollback\_migrate\_user} は、「移行してから戻すと元のユーザーに戻る」という仕様です。
テストケースではなく、任意の \texttt{u : UserV1} について成り立つ主張として書いています。
\end{softwarebox}

\subsection{\texttt{by}：ここから証明を書く}

\texttt{by} は、命題の証明を始める合図です。
\texttt{by} の後ろには、\texttt{rfl}、\texttt{simp}、\texttt{rw} などの証明手順を書きます。

\begin{leanbox}
\texttt{theorem 名前 : 命題 := by ...} という形は、「この命題を、\texttt{by} 以下の手順で証明する」と読みます。
本文では、この形が仕様証明の基本単位になります。
\end{leanbox}

\section{等式を閉じる}

\subsection{\texttt{rfl}：定義どおりなら閉じる}

\begin{definitionbox}
\texttt{rfl} は反射律のタクティクであり、両辺が定義を展開すれば同じ形になるときに証明を閉じます。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem pipeline_ten :
    pipeline 10 = 22 := by
  rfl
\end{minted}
\caption{\texttt{pipeline\_ten}}
\label{lst:appA_rfl}
\end{listing}

\begin{warningbox}
\texttt{rfl} は、「何となく明らか」であれば何でも解決する道具ではありません。
定義を展開して同じ形になるときだけ使えます。
\end{warningbox}

\subsection{\texttt{simp}：明らかな整理を任せる}

\begin{definitionbox}
\texttt{simp} は、Lean が知っている単純化規則を使って式を整理するタクティクです。
空リスト、恒等的な演算、不要な \texttt{if} の分岐などを整理するときによく使います。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem append_nil_right (xs : List Nat) :
    xs ++ [] = xs := by
  simp

theorem map_nil_inc :
    List.map inc [] = [] := by
  simp
\end{minted}
\caption{\texttt{append\_nil\_right}・\texttt{map\_nil\_inc}}
\label{lst:appA_simp}
\end{listing}

\begin{softwarebox}
\texttt{simp} は、レビューに置き換えると「定義上すぐに消えるノイズを整理する」操作に近いものです。
証明の本筋を読みやすくするために使います。
\end{softwarebox}

\subsection{\texttt{rw}：既知の等式で書き換える}

\begin{definitionbox}
\texttt{rw} は、すでに持っている等式を使って、ゴールや仮定の中の式を書き換えるタクティクです。
リファクタリング前後の式を同じ形へ近付けるときによく使います。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem rewrite_price
    (base fee total : Nat)
    (h : base + fee = total) :
    (base + fee) + 0 = total := by
  rw [h]
  simp
\end{minted}
\caption{\texttt{rewrite\_price}}
\label{lst:appA_rw}
\end{listing}

\begin{warningbox}
\texttt{rw [h]} は、等式 \texttt{h} の左辺を右辺へ書き換えます。
逆向きに使う場合は、\texttt{rw [← h]} と書きます。
\end{warningbox}

\subsection{\texttt{calc}：段階的な式変形を書く}

\begin{definitionbox}
\texttt{calc} は、式変形を複数行に分けて書く構文です。
証明をレビューしやすい変更ログとして残す場合に適しています。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem add_assoc_calc (a b c : Nat) :
    (a + b) + c = a + (b + c) := by
  calc
    (a + b) + c = a + (b + c) := by
      rw [Nat.add_assoc]
\end{minted}
\caption{\texttt{add\_assoc\_calc}}
\label{lst:appA_calc}
\end{listing}

\begin{softwarebox}
\texttt{calc} は、計算式や変換パイプラインのリファクタリングを説明するのに適しています。
各行が、「この理由で次の形に変える」という根拠になります。
\end{softwarebox}

\section{全パターンを分解する}

\subsection{\texttt{cases}：値の形ごとに場合分けする}

\begin{definitionbox}
\texttt{cases} は、値が取り得る形に応じてゴールを分解するタクティクです。
\texttt{Option}、\texttt{Bool}、\texttt{Sum}、自作の \texttt{inductive} 型でよく使います。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
def isSomeNat (x : Option Nat) : Bool :=
  match x with
  | none => false
  | some _ => true

theorem isSomeNat_cases (x : Option Nat) :
    isSomeNat x = true \/ isSomeNat x = false := by
  cases x with
  | none =>
      exact Or.inr rfl
  | some n =>
      exact Or.inl rfl
\end{minted}
\caption{\texttt{isSomeNat\_cases}}
\label{lst:appA_cases}
\end{listing}

\begin{softwarebox}
\texttt{cases} は、成功ケースだけを見るレビューを防ぎます。
失敗ケース、空ケース、例外ケースを Lean に明示させることで、仕様漏れに気付きやすくなります。
\end{softwarebox}

\section{任意の長さのデータを扱う}

\subsection{\texttt{induction}：空ケースと追加ケースで証明する}

\begin{definitionbox}
\texttt{induction} は、自然数、リスト、木のような再帰的データ構造について、すべての場合に成り立つことを示すタクティクです。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem map_length_inc (xs : List Nat) :
    (List.map inc xs).length = xs.length := by
  induction xs with
  | nil =>
      rfl
  | cons x xs ih =>
      simp [List.map, ih]
\end{minted}
\caption{\texttt{map\_length\_inc}}
\label{lst:appA_induction}
\end{listing}

\begin{intuitionbox}
リストの帰納法は、「空リストで成り立つ」「先頭に一つ追加しても成り立つ」ことを示すことで、任意の長さのリストについて成り立つことを確認する方法です。
\end{intuitionbox}

\begin{softwarebox}
1件の移行を \texttt{List.map} でバッチ全体へ持ち上げるとき、件数の保存や全要素の性質の保存は、この形の証明になります。
\end{softwarebox}

\section{証明を組み立てる基本操作}

\subsection{\texttt{constructor}：かつ条件や構造体を組み立てる}

\begin{definitionbox}
\texttt{constructor} は、ゴールが「両方を示せ」または「この構造を作れ」という形のときに、必要な部品ごとにゴールを分けるタクティクです。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem pair_spec (n : Nat) :
    n = n /\ inc n = n + 1 := by
  constructor
  · exact rfl
  · exact rfl
\end{minted}
\caption{\texttt{pair\_spec}}
\label{lst:appA_constructor_exact}
\end{listing}

このコードでは、ゴールが二つの条件の組であるため、\texttt{constructor}で左右のゴールへ分けています。
それぞれのゴールは定義どおりの等式であるため、手元の証拠として\texttt{rfl}を\texttt{exact}に渡して閉じています。

\subsection{\texttt{exact}：手元の証拠でゴールを閉じる}

\begin{definitionbox}
\texttt{exact} は、現在のゴールに一致する証拠を渡して証明を閉じるタクティクです。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem use_existing_evidence
    (P : Prop)
    (hp : P) : P := by
  exact hp
\end{minted}
\caption{\texttt{use\_existing\_evidence}}
\label{lst:appA_exact}
\end{listing}

このコードでは、仮定\texttt{hp : P}が、そのままゴール\texttt{P}の証拠になっています。
\texttt{exact hp}は、新しい証明を作るのではなく、すでに手元にある証拠を提出するものとして読みます。

\subsection{\texttt{apply}：使いたい定理にゴールを合わせる}

\begin{definitionbox}
\texttt{apply} は、現在のゴールを、使いたい定理や仮定の前提へ変換するタクティクです。
\end{definitionbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem use_implication
    (P Q : Prop)
    (h : P -> Q)
    (hp : P) : Q := by
  apply h
  exact hp
\end{minted}
\caption{\texttt{use\_implication}}
\label{lst:appA_apply}
\end{listing}

\texttt{apply h} は、「\texttt{Q} を示すには \texttt{h : P -> Q} が使えるため、残る前提 \texttt{P} を示せばよい」という形へゴールを変えます。
\texttt{exact} が完成済みの証拠を渡すのに対し、\texttt{apply} は使う証明の形を先に決めます。

\section{ゴールの導入・展開・一括処理}

\subsection{\texttt{intro}：任意の値や仮定を受け取る}

\texttt{intro} は、全称量化された値や含意の前提を、現在の証明で使える名前として受け取ります。
たとえば \texttt{forall n, n = n} の証明では、任意の \texttt{n} を受け取ってから等式を示します。

\subsection{\texttt{unfold}：定義を明示的に展開する}

\texttt{unfold 名前} は、指定した定義の本体をゴール内で展開します。
定義の内部を一段見せてから \texttt{rfl} や \texttt{simp} を使いたいときに適しています。

\subsection{\texttt{decide}：決定可能な具体命題を計算で閉じる}

\texttt{decide} は、真偽を計算できる具体的な命題について、その判定結果から証明を作ります。
一般の数学的主張を自動で発見する道具ではなく、決定可能性が用意された命題に使います。

\subsection{\texttt{funext}：関数等式を点ごとの等式へ変える}

\texttt{funext x} は、二つの関数が等しいというゴールを、任意の入力 \texttt{x} で出力が等しいというゴールへ変えます。
wrapper や adapter の実装が異なっても観測結果が一致することを示すときに使います。

\subsection{\texttt{<;>}：生じたすべてのゴールへ続けて適用する}

\texttt{tactic1 <;> tactic2} は、\texttt{tactic1} が作ったすべてのゴールに \texttt{tactic2} を適用します。
場合分けした各ゴールを同じ手順で閉じられるときに、重複を減らせます。

\begin{listing}[htbp]
\begin{minted}{lean4}
def addTwo (n : Nat) : Nat :=
  n + 1 + 1

theorem every_nat_self : forall n : Nat, n = n := by
  intro n
  rfl

theorem addTwo_zero : addTwo 0 = 2 := by
  unfold addTwo
  rfl

example : (3 : Nat) < 5 := by
  decide

theorem inc_as_function : inc = (fun n => n + 1) := by
  funext n
  rfl

def optionToList {A : Type} : Option A -> List A
  | none => []
  | some a => [a]

theorem optionToList_length (x : Option Nat) :
    (optionToList x).length <= 1 := by
  cases x <;> simp [optionToList]
\end{minted}
\caption{\texttt{intro}・\texttt{unfold}・\texttt{decide}・\texttt{funext}・\texttt{<;>}}
\label{lst:appA_goal_operations}
\end{listing}

コード\ref{lst:appA_goal_operations}では、各タクティクを一つの役割に絞っています。
\texttt{every\_nat\_self} は任意の値を導入し、\texttt{addTwo\_zero} は定義を展開し、具体的な大小比較は計算で閉じています。
関数等式では入力ごとの等式へ移り、\texttt{Option} の場合分けでは二つのゴールへ同じ単純化を適用しています。

\section{よくある読み方ミス}

\begin{warningbox}
\textbf{\texttt{Bool} と \texttt{Prop} を混同しない。}
\texttt{Bool} は計算される真偽値であり、\texttt{Prop} は証明対象となる命題です。
仕様として「すべての場合に成り立つ」と述べる場合は、通常 \texttt{Prop} 側で書きます。
\end{warningbox}

\begin{warningbox}
\textbf{\texttt{simp} を魔法だと思わない。}
\texttt{simp} は、既知の単純化規則を使って整理する道具です。
モデルが曖昧なままでも仕様を発見するわけではありません。
\end{warningbox}

\begin{warningbox}
\textbf{\texttt{rw} の向きに注意する。}
\texttt{rw [h]} で進まないときは、等式の向きが逆かもしれません。
その場合は、\texttt{rw [← h]} を検討します。
\end{warningbox}

\begin{warningbox}
\textbf{\texttt{cases} と \texttt{induction} を混同しない。}
\texttt{cases} は、値の現在の形を分けます。
\texttt{induction} は、再帰的データについて、より小さい場合の仮定を使って全体を示します。
\end{warningbox}

\section{最小コマンド一覧}

\section{本付録のまとめ}

Lean 4 の最小語彙は、役割に分けて読むと整理しやすくなります。
\texttt{def}、\texttt{structure}、\texttt{inductive} は、モデルを作るための語彙です。
\texttt{example}、\texttt{theorem}、\texttt{by} は、仕様を主張して証明を始めるための語彙です。
\texttt{rfl}、\texttt{simp}、\texttt{rw}、\texttt{calc} は、等式を扱うための語彙です。
\texttt{cases} と \texttt{induction} は、分岐や任意長のデータを漏れなく扱うための語彙です。
最後に、\texttt{constructor}、\texttt{exact}、\texttt{apply} は、証明を小さな部品から組み立てるための語彙です。
\texttt{intro}、\texttt{unfold}、\texttt{decide}、\texttt{funext}、\texttt{<;>} は、ゴールの形を読みやすく変え、共通の手順を適用するための語彙です。

本書の本文では、これらを組み合わせて、リファクタリング前後の意味保存、データ移行の round-trip、不変条件の保存、API adapter の一貫性を表します。
分からない構文に出会ったら、まずこの一覧に戻り、その構文が「モデルを書く」「仕様を書く」「等式を閉じる」「ケースを分ける」のどれに当たるかを確認するとよいでしょう。
