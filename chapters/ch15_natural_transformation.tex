\begin{chapterabstract}
本章では、自然変換を「関手どうしを一貫してつなぐ adapter」として導入します。
前章では、\texttt{List.map} や \texttt{Option.map} を通じて、関手を「構造を保ったまま中身を変える仕組み」として見ました。
本章では、その関手どうしをつなぐ変換を考えます。
典型例は、\texttt{Option A} を \texttt{List A} に変換する処理です。
\texttt{A} が \texttt{Nat}、\texttt{String}、ユーザー型のいずれであっても、\texttt{none} は空リストへ、\texttt{some a} は1要素リストへ変換するという同じ規則で動きます。
この変換と \texttt{map} の整合性を、自然性という等式で表します。

実務上、自然変換は API response wrapper、SDK adapter、nullable 値からリストへの変換、旧形式から新形式への wrapper 変換を考えるときに役立ちます。
adapter を業務ロジックの前に置いても後に置いても結果が同じなら、その adapter は業務ロジックと干渉していないと言えます。
本章では、まず \texttt{Option} から \texttt{List} への小さな例で自然性を Lean 4 の等式として証明し、次に API response wrapper のミニケースへ接続します。
\end{chapterabstract}

\begin{learninggoals}
\item 自然変換を「関手どうしを一貫してつなぐ adapter」として説明できます。
\item 自然性の四角形を「adapter と \texttt{map} で持ち上げた関数の適用順を交換できる」という等式として読めます。
\item \texttt{Option A -> List A} のような型に依存しない変換を Lean 4 で定義できます。
\item API response wrapper の変更に対して、adapter と \texttt{map} が可換であることを仕様として書けます。
\end{learninggoals}

\section{adapter と変換順序の一貫性}

API のレスポンス形式を変える場面を考えます。
旧 API では、1件あるかもしれない値を \texttt{Option A} のように返していました。
新 API では、0件以上の値を \texttt{List A} のように返す設計へ変えます。
このとき、旧形式を新形式へ変換する adapter は簡単に書けます。
\texttt{none} を空リスト、\texttt{some a} を1要素リストにするだけです。

しかし、adapter があるだけでは不十分です。
多くの実務コードでは、レスポンスの中身に業務ロジックを適用します。
たとえば、ユーザー ID のマスク、価格の税込み変換、内部DTOから公開DTOへの変換などです。
このとき、次の二つの順序が考えられます。

\begin{itemize}
\item 先に旧形式の中身を変換してから、新形式へ adapter を通します。
\item 先に新形式へ adapter を通してから、新形式の中身を変換します。
\end{itemize}

この二つが常に同じ結果になるなら、adapter は中身の業務ロジックと干渉していません。
自然変換は、この性質を圏論の言葉で整理する概念です。

図\ref{fig:ch15-overview}は、二つの wrapper と、それぞれの \texttt{map}、wrapper 間の adapter の関係を示します。
先に中身を変えても、先に wrapper を変えても結果が一致することが、本章で確かめる自然性です。

\section{ソフトウェア上の問題：adapter は業務ロジックと干渉していないか}

\subsection{wrapper 変更は単なる型変換ではない}

レスポンス wrapper の変更は、単なるフィールド名の変更より複雑です。
中身の値 \texttt{A} だけでなく、その値を包む構造も変わるためです。
たとえば、次のような変更を考えます。

\begin{center}
\begin{tabular}{lll}
\toprule
観点 & 旧形式 & 新形式 \\
\midrule
値がない場合 & \texttt{none} & \texttt{[]} \\
値が1件ある場合 & \texttt{some a} & \texttt{[a]} \\
中身の変換 & \texttt{Option.map f} & \texttt{List.map f} \\
adapter & \texttt{Option A -> List A} & wrapper の形を変える処理 \\
\bottomrule
\end{tabular}
\end{center}

ここで使う変換は、\texttt{A} が具体的に何であるかを見ません。
\texttt{Nat}、\texttt{String}、\texttt{User} のいずれでも、規則は同じです。
ただし、実装が payload を調べないことは自然性を理解する助けにはなりますが、自然変換の定義そのものではありません。
正式な判定条件は、対象ごとの成分を持ち、定義域の圏のすべての射について \texttt{map} と可換になることです。

\subsection{よくある危険：値の中身を勝手に見る adapter}

本章の \texttt{Option} から \texttt{List} への adapter は、中身の型 \texttt{A} の具体的な性質に依存しません。
\texttt{Option A -> List A} の例では、\texttt{none} か \texttt{some} かだけを見ます。
\texttt{some a} の中身 \texttt{a} が偶数か、名前が空文字か、価格が高いか、といった業務判断はしません。

adapter が中身を見て条件分岐すると、業務ロジックとの順序を入れ替えたときに結果が変わる可能性があります。
そのような処理は、単なる wrapper 変換ではなく、業務変換を混ぜ込んだ処理です。
このような混入は自然性を壊すことがあるため、可換性の証明を設計上の検査に使えます。
ただし、何を射として許すかを制限した別の圏では、自然性の条件も変わります。

\section{自然変換を定義する}

自然変換の標準定義では、二つの関手の各対象に成分を与え、定義域の圏の任意の射について自然性の四角形を可換にします~\cite{riehl-category-theory-in-context}。

\subsection{関手どうしをつなぐ変換}

前章では、\texttt{Option} や \texttt{List} を「中身を変える操作」を持つものとして見ました。
\texttt{Option.map f} は、値があれば中身に \texttt{f} を適用し、値がなければ \texttt{none} のままにします。
\texttt{List.map f} は、リストの各要素に \texttt{f} を適用し、リストという構造を保ちます。

自然変換は、このような二つの構造の間に、型ごとの変換を与えます。
\texttt{Option} から \texttt{List} への場合は、各型 \texttt{A} について次の変換を持ちます。

\[
  \eta_A : \mathrm{Option}\ A \to \mathrm{List}\ A
\]

ここで \(\eta_A\) は、\texttt{none} を \texttt{[]} に、\texttt{some a} を \texttt{[a]} に変換します。
添字 \(A\) は「型 \texttt{A} に対する成分」を表します。
成分とは、実務上は「各 payload 型に対して用意される adapter 関数」と読めます。

本書の範囲では、自然変換を次のように読みます。
二つの「中身を \texttt{map} できる構造」\(F\) と \(G\) があるとします。
自然変換 \(\eta : F \Rightarrow G\) は、各型 \(A\) について変換
\[
  \eta_A : F A \to G A
\]
を与え、任意の関数 \(f : A \to B\) について次の等式を満たすものと考えます。
\[
  \eta_B(\mathrm{map}_F(f)(x))
  =
  \mathrm{map}_G(f)(\eta_A(x))
\]
左辺は「中身を変えてから adapter」、右辺は「adapter してから中身を変える」という経路です。

\subsection{自然性の四角形}

自然性は、図で見ると理解しやすくなります。
左上に \(F A\)、右上に \(F B\)、左下に \(G A\)、右下に \(G B\) を置きます。
横方向の矢印は中身の変換 \(f : A \to B\) を \texttt{map} したものであり、縦方向の矢印は adapter \(\eta\) です。

\FigurePlaceholder{fig-ch15-naturality-square.png}{0.86}{自然性の四角形}{fig:ch15-naturality-square}

この四角形が可換であるとは、左上から右下へ進む二つの経路が同じ結果になるという意味です。

\begin{itemize}
\item 上段を進んでから下へ進みます：\(F A\) の中身を \(f\) で変換し、その後 \(G B\) へ adapter します。
\item 下へ進んでから下段を進みます：先に \(F A\) を \(G A\) へ adapter し、その後 \(G\) 側で中身を \(f\) で変換します。
\end{itemize}

この二つが一致することを、次の式で表します。

\[
  \eta_B(\mathrm{map}_F(f)(x))
  =
  \mathrm{map}_G(f)(\eta_A(x))
\]

自然性の四角形は、実務の言葉では「adapter と業務ロジックの順序を入れ替えてもよい」という仕様です。

\section{例：\texttt{Option A -> List A}}

\subsection{型に依存しない変換}

最初の例として、\texttt{Option A} を \texttt{List A} に変換します。
値がなければ空リスト、値があれば1要素リストにします。
この変換は、\texttt{A} の中身には触れません。
\texttt{A} が何であっても、\texttt{Option} の外側の形だけを見て変換します。

Lean では、自然性の式を直接読めるように、\texttt{optionMap} と \texttt{optionToList} を小さく自作します。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter15

def optionMap {A B : Type} (f : A -> B) : Option A -> Option B
  | none => none
  | some a => some (f a)

def optionToList {A : Type} : Option A -> List A
  | none => []
  | some a => [a]

#eval optionToList (some 10)  -- => [10]
#eval optionToList (none : Option Nat)  -- => []

theorem optionToList_naturality {A B : Type}
    (f : A -> B) (x : Option A) :
    optionToList (optionMap f x) =
      List.map f (optionToList x) := by
  cases x <;> rfl
\end{minted}
\caption{\texttt{optionToList\_naturality}}
\label{lst:ch15_option_to_list}
\end{listing}

証明には \texttt{cases x} を使います。
\texttt{x} は \texttt{none} か \texttt{some a} のどちらかです。
\texttt{<;>} は、直前のタクティクで生じたすべてのゴールに、その後ろのタクティクを適用します。
したがって \texttt{cases x <;> rfl} は、二つの場合へ分けた後、両方を \texttt{rfl} で閉じるという意味です。
\texttt{none} の場合は、左辺も右辺も空リストになります。
\texttt{some a} の場合は、左辺も右辺も \texttt{[f a]} になります。
したがって、どちらのケースでも、定義を展開すると同じ形になるため \texttt{rfl} で証明できます。

\subsection{この証明が言っていること}

この定理は、すべての型 \texttt{A} と \texttt{B}、すべての関数 \texttt{f : A -> B}、すべての \texttt{x : Option A} に対して成り立ちます。
つまり、特定の入力例だけを確認したものではありません。
\texttt{Nat}、\texttt{String}、\texttt{User} の例をいくつかテストしたという主張よりも一般的です。

したがって、\texttt{optionToList} については、\texttt{Option} 内で \texttt{f} を適用してから \texttt{List} へ移す処理と、先に \texttt{List} へ移してから \texttt{f} を適用する処理を置き換えられます。

\section{Lean 4で自然変換を小さな構造として表す}

\subsection{成分と自然性を持つ構造}

ここまでの例では、\texttt{optionToList\_naturality} という定理を単独で書きました。
少しだけ抽象化すると、「各型に対する adapter」と「自然性の証明」を一つの構造体にまとめられます。

本格的な圏論では、一般の関手 \(F\) と \(G\) の間の自然変換を扱います。
しかし本章では、初学者向けに \texttt{Option} から \texttt{List} への自然変換だけを小さく表します。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure NatTransOptionList where
  app : {A : Type} -> Option A -> List A
  naturality :
    {A B : Type} -> (f : A -> B) -> (x : Option A) ->
      app (optionMap f x) = List.map f (app x)

def optionToListNT : NatTransOptionList where
  app := fun x => optionToList x
  naturality := by
    intro A B f x
    cases x <;> rfl
\end{minted}
\caption{\texttt{NatTransOptionList}}
\label{lst:ch15_nattrans_structure}
\end{listing}

\texttt{app} は、各型 \texttt{A} に対する adapter の本体です。
名前は application、すなわち「適用」を短くしたものです。
\texttt{naturality} は、その adapter が \texttt{map} と可換であることの証明です。

ここで定義した \texttt{NatTransOptionList} は、学習用の最小モデルです。
Mathlib の \texttt{CategoryTheory.NatTrans} を直接使っているわけではありません。
本書のこの段階では、抽象的な category theory API を覚えることより、自然性がどの等式を意味するかを理解することを優先します。

\subsection{自然変換は型ごとの関数の寄せ集めではない}

自然変換は、各型 \texttt{A} について関数を持ちます。
しかし、単に関数を多数集めただけでは自然変換とは言えません。
それらの関数が \texttt{map} と整合している必要があります。

たとえば、\texttt{Nat} のときだけ特別な値を足す、\texttt{String} のときだけ空文字を消す、\texttt{User} のときだけ管理者を除外する、といった処理は、型に依存した業務ロジックです。
そのような処理が有用な場合もありますが、自然変換としては扱いにくくなります。
少なくとも本章のように \texttt{Type} のすべての関数を射として扱うなら、自然性はそのような型固有の処理を許さない強い条件になります。

\section{自然性を実務の仕様として読む}

\subsection{自然性は順序交換の仕様である}

自然性の等式をもう一度見ます。

\[
  \eta_B(\mathrm{map}_F(f)(x))
  =
  \mathrm{map}_G(f)(\eta_A(x))
\]

この式に現れる \(f\) は、任意の業務ロジックとして読めます。
たとえば、次のような変換です。

\begin{itemize}
\item 内部ユーザーを公開ユーザーへ変換します。
\item 税抜価格を税込価格へ変換します。
\item DB レコードを API DTO へ変換します。
\item 機微情報をマスクします。
\end{itemize}

自然性が成り立つとき、adapter はこれらの変換と衝突しません。
つまり、adapter の導入位置を変えても、最終結果は変わりません。
これは、レイヤー分離や互換層の設計で重要な性質です。

「adapter は箱の形を変え、\texttt{map} される処理とは干渉しない」という説明は、この等式の実務的な読み替えです。
形式的な根拠は、選んだ関手と射に対して自然性の等式が成り立つことです。

\subsection{テストではなく仕様として切り出す}

自然性は、具体例をいくつかテストするだけでは確認しきれません。
\texttt{Nat}、\texttt{String}、特定の \texttt{User} の変換で成功しても、すべての型、すべての関数、すべての値で成り立つとは限りません。

Lean で定理として書くと、この性質を一般的な仕様として扱えます。
ただし、保証はLean 上でモデル化した範囲に限られます。
adapter の核となる変換規則が小さければ、自然性を証明する価値があります。

\section{ミニケース：API response wrapper の変更}

\subsection{旧レスポンスと新レスポンス}

次に、少し実務に近い例へ進みます。
旧レスポンスは、trace ID と、存在するかもしれない payload を持つとします。
新レスポンスは、trace ID、状態、payload のリストを持つとします。
ここでは、値がない場合は \texttt{empty}、値がある場合は \texttt{ok} という単純な状態を使います。

図\ref{fig:ch15-api-adapter}では、上段が旧 wrapper 内で変換してから adapter を適用する経路、下段が先に新 wrapper へ移してから変換する経路です。
次のモデルでは、この二つの経路が一致することを定理にします。

\FigurePlaceholder{fig-ch15-api-adapter.png}{0.90}{API response adapter}{fig:ch15-api-adapter}

\begin{listing}[htbp]
\begin{minted}{lean4}
inductive Status where
  | ok
  | empty
  deriving Repr, DecidableEq

structure OldResponse (A : Type) where
  traceId : Nat
  payload : Option A
  deriving Repr

structure NewResponse (A : Type) where
  traceId : Nat
  status : Status
  items : List A
  deriving Repr

def statusOf {A : Type} : Option A -> Status
  | none => Status.empty
  | some _ => Status.ok
\end{minted}
\caption{\texttt{OldResponse} と \texttt{NewResponse}}
\label{lst:ch15_api_model}
\end{listing}

ここで \texttt{OldResponse} と \texttt{NewResponse} は、それぞれ型引数 \texttt{A} を持ちます。
これは、payload の中身が何であるかを抽象化しているという意味です。
\texttt{A} がユーザー、注文、価格のいずれであっても、wrapper の形は同じです。

\subsection{map と adapter を定義する}

次に、旧レスポンスと新レスポンスのそれぞれに、\texttt{map} 相当の操作を定義します。
\texttt{oldMap} は \texttt{payload} の中身だけを変換します。
\texttt{newMap} は \texttt{items} の各要素だけを変換し、\texttt{traceId} と \texttt{status} をそのまま保ちます。

\begin{listing}[htbp]
\begin{minted}{lean4}
def oldMap {A B : Type} (f : A -> B)
    (r : OldResponse A) : OldResponse B :=
  { traceId := r.traceId,
    payload := optionMap f r.payload }

def newMap {A B : Type} (f : A -> B)
    (r : NewResponse A) : NewResponse B :=
  { traceId := r.traceId,
    status := r.status,
    items := List.map f r.items }

def adapt {A : Type} (r : OldResponse A) : NewResponse A :=
  { traceId := r.traceId,
    status := statusOf r.payload,
    items := optionToList r.payload }

#eval adapt ({ traceId := 7, payload := some 42 } : OldResponse Nat)  -- => { traceId := 7, status := Chapter15.Status.ok, items := [42] }
\end{minted}
\caption{\texttt{oldMap}・\texttt{newMap}・\texttt{adapt}}
\label{lst:ch15_api_map_adapt}
\end{listing}

\texttt{adapt} は、旧形式から新形式への adapter です。
この adapter は、payload の中身そのものには触れません。
値が存在するかどうかだけを見て、\texttt{status} と \texttt{items} を作ります。

\subsection{adapter と map が可換であることを証明する}

証明したいことは、次の等式です。

\[
  \mathrm{adapt}(\mathrm{oldMap}(f)(r))
  =
  \mathrm{newMap}(f)(\mathrm{adapt}(r))
\]

左辺は、旧レスポンスの中で業務ロジック \(f\) を適用してから新レスポンスへ変換する経路です。
右辺は、先に新レスポンスへ変換してから、新レスポンスの中で業務ロジック \(f\) を適用する経路です。

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem adapt_naturality {A B : Type}
    (f : A -> B) (r : OldResponse A) :
    adapt (oldMap f r) = newMap f (adapt r) := by
  cases r with
  | mk trace payload =>
    cases payload <;> rfl

theorem adapt_naturality_as_function_equality {A B : Type}
    (f : A -> B) :
    (fun r : OldResponse A => adapt (oldMap f r)) =
      (fun r : OldResponse A => newMap f (adapt r)) := by
  funext r
  exact adapt_naturality f r

end Chapter15
\end{minted}
\caption{\texttt{adapt\_naturality}}
\label{lst:ch15_api_naturality}
\end{listing}

\texttt{funext r} は、二つの関数そのものの等式を、任意の入力 \texttt{r} における出力の等式へ変えます。
ここでは、得られた点ごとの等式が \texttt{adapt\_naturality f r} と一致するため、\texttt{exact} で証明を閉じています。

最初の定理 \texttt{adapt\_naturality} は、任意のレスポンス \texttt{r} について二つの経路が同じであることを述べています。
証明では、まず \texttt{r} を \texttt{trace} と \texttt{payload} に分解します。
次に、\texttt{payload} が \texttt{none} か \texttt{some} かで場合分けします。
どちらの場合も、定義を展開すると左右は同じ形になります。

二つ目の定理 \texttt{adapt\_naturality\_as\_function\_equality} は、同じ事実を「関数どうしが等しい」という形で書き直しています。
二つの関数がすべての入力 \texttt{r} で同じ結果を返すなら、関数として同じだと扱えます。

\section{実務上の読み替え：adapter が業務ロジックと干渉しないこと}

\subsection{adapter の責務を小さく保つ}

自然性の証明から直接分かるのは、adapter が指定した \texttt{map} と可換であることまでです。
本章のように wrapper の構造変換へ責務を絞ると、その条件を満たす設計にしやすくなります。
その結果、次のような判断もしやすくなります。

\begin{itemize}
\item 業務ロジックを旧 wrapper 側に残すか、新 wrapper 側へ移すかを選びやすくなります。
\item adapter の導入によって payload の意味が変わらないと説明しやすくなります。
\item API バージョニングや SDK 互換層のレビューで、確認すべき性質を明確にできます。
\item API adapter のケーススタディで、自然性を具体的なレビュー条件として使えます。
\end{itemize}

\subsection{自然性が壊れる典型例}

自然性が壊れる典型例は、adapter が中身を見て別の値へ変える場合です。
たとえば、\texttt{some a} を \texttt{[a]} にするだけでなく、特定条件の \texttt{a} を捨てたり、別の値を補ったりする adapter を考えます。
その条件が業務ロジック \(f\) の前後で変わるなら、自然性の四角形は可換になりません。

もう一つの典型例は、型ごとに ad hoc な変換を差し込むことです。
\texttt{Nat} のときだけ \texttt{0} を追加する、\texttt{String} のときだけ空文字を消す、といった処理は、payload 型の意味を知っています。
それは wrapper 変換ではなく、業務ルールです。
このような処理を adapter と呼ぶと、設計上の責務境界が曖昧になります。

ただし、これらが自然でないという結論は、本章と同じく、全型と全関数を対象にした場合の話です。
型クラスを備えた対象と、その構造を保つ射だけに範囲を変えれば、対応する自然性を改めて判定する必要があります。

\section{本章のまとめ}

自然変換は、二つの関手の各対象に成分を持ち、すべての射について自然性を満たす変換です。

自然性の四角形は、「中身を変えてから adapter」と「adapter してから中身を変える」が一致するという仕様です。

\texttt{Option A -> List A} の例では、\texttt{none} を空リストへ、\texttt{some a} を1要素リストへ変換する adapter が自然性を満たします。

Lean では、自然性を通常の等式として書けます。
\texttt{cases} による場合分けで、\texttt{none} と \texttt{some} の両方を確認できます。

API response wrapper の変更では、adapter と \texttt{map} が可換であることを示すと、モデル化した変換の範囲で両者の適用順を交換できると説明できます。
