\chapter{等式推論とリファクタリング}
\label{chap:ch09_equality_refactoring}

\begin{chapterabstract}
本章では、リファクタリングを「意味を保存する等式」として扱う。ソフトウェアでは、関数を分割する、共通処理を抽出する、計算式を整理する、処理順序を入れ替える、といった変更が日常的に行われる。これらの変更が安全であるとは、少なくとも観測したい範囲で、変更前と変更後が同じ結果を返すということである。Lean 4 では、この主張を等式として書き、\texttt{rfl}、\texttt{simp}、\texttt{rw}、\texttt{calc} で確認できる。

この章の目的は、Leanで大きなプログラム全体を検証することではない。変更で壊したくない性質を小さな式として切り出し、その式がすべての入力で成り立つことを確認する感覚を身につけることである。
\end{chapterabstract}

\begin{learninggoals}
\item リファクタリングを「変更前後の意味保存」として説明できる。
\item Leanで、二つの関数が同じ結果を返すことを等式として書ける。
\item \texttt{simp}、\texttt{rw}、\texttt{calc} を、式の整理・書き換え・レビュー可能な変形ログとして読める。
\item 処理順序を変えてよい場合と、変えてはいけない場合を、証明可能な等式として区別できる。
\end{learninggoals}

\section{この章を読む理由}

リファクタリングは、テストを書いていれば安全になるとは限らない。テストは、選んだ入力に対して結果が変わらないことを確認する。しかし、リファクタリングで本当に言いたいことは、たいてい次の形である。

\begin{quote}
すべての入力について、変更前の処理と変更後の処理は同じ結果を返す。
\end{quote}

これは自然言語のレビューコメントとしても書ける。しかし、Leanではこの主張をそのまま命題として書ける。たとえば、\texttt{old x = new x} という等式を、任意の \texttt{x} について証明する。

\begin{intuitionbox}
等式推論とは、「同じ意味を持つ形へ少しずつ変形すること」である。コードレビューで「この式はこのように整理できる」と説明するとき、私たちはすでに非形式的な等式推論をしている。Leanは、その変形が本当に正しいかを機械的に確認する。
\end{intuitionbox}

\begin{softwarebox}
本章では、実務コード全体ではなく、仕様の核だけを小さく取り出す。たとえば「価格計算の共通処理を抽出しても結果が変わらない」「二つの加算処理の順序を入れ替えても合計は同じ」といった性質を、Lean上の小さな関数で表す。
\end{softwarebox}

\section{等しいとは何か}

\subsection{見た目が同じことと、意味が同じこと}

ソフトウェアでは、二つのコードが同じであるという言い方に複数の意味がある。文字列として同じ、ASTとして同じ、実行速度が同じ、外部から見た結果が同じ、ログや副作用まで同じ、などである。

本章で扱うのは、最初の一歩としての「戻り値の等しさ」である。つまり、同じ入力を与えたときに同じ値を返す、という性質である。これは、純粋な関数を小さくモデル化するときに最も扱いやすい。

\begin{definitionbox}
本章でいう意味保存とは、変更前の関数 \texttt{old} と変更後の関数 \texttt{new} について、すべての入力 \texttt{x} で \texttt{old x = new x} が成り立つことである。
\end{definitionbox}

注意したいのは、これは現実のリファクタリング安全性のすべてではない、という点である。実システムでは、例外、ログ、DB更新、時刻、乱数、外部API呼び出し、パフォーマンスなども観測されることがある。本章では、まず戻り値に集中する。

\subsection{Leanにおける等式}

Leanでは、等式 \texttt{a = b} は命題である。命題なので、証明の対象になる。

次の例では、\texttt{rawCalc} は少し冗長な計算であり、\texttt{refactoredCalc} はそれを整理したものだと考える。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Ch03EqualityRefactoring

-- 何もしない整理は、simp で消せることが多い。
def rawCalc (x : Nat) : Nat :=
  (x + 0) * 1

def refactoredCalc (x : Nat) : Nat :=
  x

theorem rawCalc_eq_refactoredCalc (x : Nat) :
    rawCalc x = refactoredCalc x := by
  unfold rawCalc refactoredCalc
  simp
\end{minted}
\caption{冗長な計算と整理後の計算}
\label{lst:ch09_raw_refactored}
\end{listing}

\begin{syntaxbox}
\texttt{namespace} は、名前の衝突を避けるための囲いである。
\begin{itemize}
\item \texttt{namespace X} から \texttt{end X} までの定義は、\texttt{X} というまとまりの中に置かれる。
\item 同じ \texttt{def name} が別章や別例にあっても、名前空間が違えば衝突しにくくなる。
\item \texttt{unfold f} は、定義 \texttt{f} の中身を証明中で展開するタクティクである。
\end{itemize}
\end{syntaxbox}


ここで証明しているのは、\texttt{rawCalc} と \texttt{refactoredCalc} が、任意の自然数 \texttt{x} について同じ結果を返す、ということだ。\texttt{unfold} は定義を展開し、\texttt{simp} は \texttt{x + 0} や \texttt{x * 1} のような明らかな整理を行う。

\begin{leanbox}
\texttt{simp} は、Leanが知っている単純化規則を使って式を整理する道具である。ここでは「0を足しても変わらない」「1を掛けても変わらない」という規則により、\texttt{(x + 0) * 1} が \texttt{x} へ整理される。
\end{leanbox}

\section{関数の結果が同じであることを証明する}

\subsection{関数そのものではなく、観測結果を比較する}

実務で知りたいのは、しばしば「実装が同じか」ではなく「外から見た結果が同じか」である。内部で一時変数を使っても、補助関数を抽出しても、同じ入力に対して同じ出力を返すなら、その観測範囲では安全な変更だと言える。

この考え方をLeanで表すには、「すべての入力について結果が等しい」という命題を書く。

\begin{listing}[htbp]
\begin{minted}{lean4}
-- 関数の結果が同じ、という仕様を全入力についての命題にする。
def sameOnNat (f g : Nat → Nat) : Prop :=
  ∀ x, f x = g x

theorem raw_and_refactored_same :
    sameOnNat rawCalc refactoredCalc := by
  intro x
  unfold rawCalc refactoredCalc
  simp
\end{minted}
\caption{すべての入力で同じ結果を返すという仕様}
\label{lst:ch09_same_on_nat}
\end{listing}

\texttt{sameOnNat} は、二つの \texttt{Nat → Nat} 関数が同じ結果を返すことを表す述語である。\texttt{intro x} は、「任意の \texttt{x} について示す」ために、入力 \texttt{x} を一つ仮に取る操作である。その \texttt{x} について等式が証明できれば、任意の \texttt{x} について証明したことになる。

\begin{softwarebox}
これは、テストケースを一つ増やすこととは違う。\texttt{intro x} で扱っている \texttt{x} は特定の値ではない。任意の入力を代表する記号である。そのため、証明が通れば、モデル化した範囲のすべての入力について性質が成り立つ。
\end{softwarebox}

\subsection{この章では関数外延性を深追いしない}

数学的には、二つの関数が同じであることは「すべての入力で同じ出力を返すこと」として扱う。この考え方を関数外延性という。Leanで関数そのものの等式を証明する話は、第21章で改めて扱う。

本章では、まず \texttt{∀ x, old x = new x} という形で十分である。これは実務上も読みやすい。「すべての入力 \texttt{x} で結果が一致する」とそのまま読めるからである。

\section{\texttt{calc} による式変形}

\subsection{証明をレビュー可能なログにする}

\texttt{simp} は便利だが、何が起きたのかを読者に説明したい場合は、変形を段階的に書いたほうがよい。そこで使うのが \texttt{calc} である。

\texttt{calc} は、式を一行ずつ変形するための構文である。コードレビューで「まず括弧を付け替え、次に補助変数の形に合わせる」と説明するのに近い。

ここでは、料金計算を例にする。変更前は式をそのまま書いている。変更後は、小計 \texttt{subtotal} と追加費用 \texttt{extra} を一時変数として抽出している。

\begin{listing}[htbp]
\begin{minted}{lean4}
-- calc は、レビューしやすい段階的な式変形である。
def totalOriginal (price qty shipping handling : Nat) : Nat :=
  ((price * qty) + shipping) + handling

def totalExtracted (price qty shipping handling : Nat) : Nat :=
  let subtotal := price * qty
  let extra := shipping + handling
  subtotal + extra

theorem total_extracted_preserves
    (price qty shipping handling : Nat) :
    totalOriginal price qty shipping handling =
      totalExtracted price qty shipping handling := by
  unfold totalOriginal totalExtracted
  calc
    ((price * qty) + shipping) + handling
        = (price * qty) + (shipping + handling) := by
          rw [Nat.add_assoc]
\end{minted}
\caption{一時変数の抽出を等式で確認する}
\label{lst:ch09_calc_extract}
\end{listing}

\begin{syntaxbox}
ここでは局所定義と計算列が出てくる。
\begin{itemize}
\item \texttt{let x := v} は、その場だけで使う名前 \texttt{x} に値 \texttt{v} を束縛する構文である。
\item \texttt{calc} は、等式変形を一行ずつ並べて読むための構文である。
\item \texttt{\_ = \_ := by ...} の各行が、前の式から次の式へ進む根拠になる。
\item \texttt{rw [← h]} は、等式 \texttt{h} を通常と逆向きに使って書き換える。
\end{itemize}
\end{syntaxbox}


この証明で使っているのは、足し算の結合律である。結合律とは、\texttt{(a + b) + c} と \texttt{a + (b + c)} が等しいという法則である。

\begin{definitionbox}
結合律は、三つ以上の処理や値をまとめるとき、括弧の付け方を変えても意味が変わらないことを表す法則である。ここでは、\texttt{shipping} と \texttt{handling} を先にまとめても合計が変わらないことを示している。
\end{definitionbox}

\subsection{小さな法則を、リファクタリング規則として読む}

数学の定理名を見ると難しく感じるかもしれない。しかし、実務上の読み替えは単純である。

\begin{itemize}
\item \texttt{Nat.add\_assoc}: 足し算の括弧を付け替えてよい。
\item \texttt{Nat.add\_comm}: 足し算の左右を入れ替えてよい。
\item \texttt{simp}: 0を足す、1を掛ける、定義上明らかな分岐を整理する。
\end{itemize}

Leanの定理は、暗記する対象というより、リファクタリングで使える安全な書き換え規則である。

\begin{warningbox}
結合律や交換律は、どの処理にも成り立つわけではない。足し算では順序を入れ替えられるが、文字列連結、丸めを含む料金計算、ログ出力、DB更新、エラーを返す処理では、順序を変えると結果や観測が変わることがある。
\end{warningbox}

\section{書き換えとしてのリファクタリング}

\subsection{\texttt{rw} は仕様に基づく置換である}

\texttt{rw} は rewrite の略として読める。既に証明済みの等式を使って、現在のゴールを書き換えるための道具である。

次の例では、\texttt{price + fee} を \texttt{fee + price} に入れ替える。これは、自然数の足し算については安全である。Leanでは、その根拠として \texttt{Nat.add\_comm} を使う。

\begin{listing}[htbp]
\begin{minted}{lean4}
-- rw は、既知の等式を使った置換として読める。
def addFeeOriginal (price fee : Nat) : Nat :=
  price + fee

def addFeeRefactored (price fee : Nat) : Nat :=
  fee + price

theorem addFee_refactor_preserves (price fee : Nat) :
    addFeeOriginal price fee = addFeeRefactored price fee := by
  unfold addFeeOriginal addFeeRefactored
  rw [Nat.add_comm]
\end{minted}
\caption{\texttt{rw} による書き換え}
\label{lst:ch09_rw_comm}
\end{listing}

この証明は短い。しかし、短いから価値が低いわけではない。むしろ、根拠が非常に明確である。

\begin{softwarebox}
この変更のレビューコメントは、「足し算なので順序を入れ替えても同じです」と書ける。Leanでは、それを \texttt{rw [Nat.add\_comm]} として明示する。レビューコメントが、機械的に検査できる証明に変わっている。
\end{softwarebox}

\subsection{補題を使って証明を短くする}

実務の証明では、毎回大きな式を直接変形すると読みにくくなる。その場合は、小さな補題を先に作る。

補題とは、後で使うための小さな定理である。たとえば、「手数料Aと手数料Bの加算順序は入れ替えてよい」という補題を作っておけば、より大きな料金計算の証明でその補題を使える。

この章では大きな補題設計には深入りしない。大事なのは、リファクタリングの根拠を、名前付きの小さな証明として切り出せるという点である。

\section{サンプル：処理順序を変えたときの同値性}

\subsection{順序変更は、いつでも安全ではない}

処理順序を変えるリファクタリングは危険である。特に、丸め、上限・下限、失敗、状態更新、副作用を含む場合は、順序が意味を持つ。

ただし、一定の条件では順序を変えてもよい。たとえば、二つの処理がどちらも単なる加算であれば、順序を入れ替えても最終的な合計は変わらない。この事実をLeanで証明してみる。

\begin{listing}[htbp]
\begin{minted}{lean4}
-- 処理順序を変えてよいのは、必要な等式が証明できる場合だけである。
def addShipping (n : Nat) : Nat :=
  n + 500

def addHandling (n : Nat) : Nat :=
  n + 100

theorem add_steps_commute (n : Nat) :
    addHandling (addShipping n) = addShipping (addHandling n) := by
  unfold addHandling addShipping
  calc
    (n + 500) + 100 = n + (500 + 100) := by
      rw [Nat.add_assoc]
    _ = n + (100 + 500) := by
      rw [Nat.add_comm 500 100]
    _ = (n + 100) + 500 := by
      rw [← Nat.add_assoc]
\end{minted}
\caption{二つの加算処理の順序を入れ替える}
\label{lst:ch09_order_commutes}
\end{listing}

この証明は、次の三段階で読める。

\begin{enumerate}
\item まず、\texttt{(n + 500) + 100} の括弧を付け替える。
\item 次に、\texttt{500 + 100} を \texttt{100 + 500} に入れ替える。
\item 最後に、括弧を戻して \texttt{(n + 100) + 500} の形にする。
\end{enumerate}

これは、処理順序を入れ替えるためのレビュー可能な変形ログである。

\subsection{順序を変えると壊れる例}

一方で、割引のような処理は順序によって結果が変わることがある。次の例では、\texttt{Nat} の引き算を使っている。Leanの \texttt{Nat} では、負の値にはならず、0で止まる。この単純化されたモデルでも、順序の違いが見える。

\begin{listing}[htbp]
\begin{minted}{lean4}
-- 反例の雰囲気: Nat の引き算は 0 で止まるため、順序で結果が変わる。
def discount100 (n : Nat) : Nat :=
  n - 100

#eval discount100 (addShipping 50)  -- => 450
#eval addShipping (discount100 50)  -- => 500
\end{minted}
\caption{順序を変えると結果が変わる例}
\label{lst:ch09_order_not_safe}
\end{listing}

最初の式は、送料を足してから割引する。二つ目の式は、割引してから送料を足す。\texttt{50} という入力では、結果が変わる。つまり、この二つの処理は、一般には入れ替えられない。

\begin{warningbox}
「似たような処理だから順序を変えてよい」と考えてはいけない。順序変更を安全だと言いたいなら、対応する等式を証明する必要がある。証明できない場合は、反例があるか、モデルに必要な前提条件が足りない可能性がある。
\end{warningbox}

\section{実務での切り出し方}

Leanに実務コードをそのまま持ち込む必要はない。むしろ、最初は持ち込まないほうがよい。重要なのは、変更で壊したくない性質を小さく切り出すことである。

たとえば、料金計算の大きな実装があるとする。その全体には、通貨、丸め、税率、キャンペーン、ユーザー種別、外部設定などが絡むかもしれない。最初からすべてを形式化すると、学習コストも保守コストも高くなる。

最初に狙うべきなのは、次のような小さな核である。

\begin{itemize}
\item 0を足す処理を消しても結果が変わらない。
\item 同じ種類の加算処理をまとめても結果が変わらない。
\item 補助関数を抽出しても、外から見た戻り値が変わらない。
\item ある二つの処理は、特定の前提のもとで順序を入れ替えられる。
\item ある二つの処理は、順序を入れ替えてはいけない。
\end{itemize}

\begin{leanbox}
Leanで証明する対象は、実装全体ではなくてもよい。むしろ、仕様の中心にある小さな等式を証明するだけでも、レビューの質は上がる。証明が通ると、少なくともそのモデル化された範囲では、変更前後の意味保存が確認できる。
\end{leanbox}

\section{よくある誤解}

\begin{warningbox}
等式で証明したからといって、実システム全体が安全になるわけではない。Leanで証明したのは、Lean上に書いたモデルと命題である。本番コード、外部API、DB、時刻、乱数、並行処理などが関わる場合は、どこまでをモデル化したのかを明示する必要がある。
\end{warningbox}

\subsection{誤解1：テストは不要になる}

不要にはならない。Leanの証明は、モデル化した範囲での全称的な保証を与える。一方、テストは、本番コード、I/O、外部サービス、パフォーマンス、設定ファイル、環境依存の挙動を確認するのに向いている。

本章で扱った証明は、テストの代替ではなく、テストでは網羅しにくい性質を補うものとして使う。

\subsection{誤解2：すべてのリファクタリングを証明すべきである}

そうではない。小さすぎる変更や、壊れても影響が限定的な変更まで証明すると、形式化のコストが価値を上回る。証明すべきなのは、壊れると困る仕様の核である。

\subsection{誤解3：\texttt{simp} が通れば意味を理解しなくてよい}

\texttt{simp} は便利だが、何が単純化されたのかを人間が理解していないと、証明はレビュー資料になりにくい。初学者のうちは、\texttt{calc} を使って変形を明示する練習も重要である。

\section{本章のまとめ}

リファクタリングは、外から見た振る舞いを変えずに内部構造を変えることである。

Leanでは、変更前後の意味保存を \texttt{old x = new x} のような等式として書ける。

\texttt{simp} は明らかな整理、\texttt{rw} は既知の等式による書き換え、\texttt{calc} は段階的な変形ログとして使える。

処理順序を変えてよいのは、必要な等式が証明できる場合である。

実務では、すべての実装ではなく、壊したくない小さな仕様の核をLeanで表すところから始める。

\begin{summarybox}
この章で新しく出た記法
\begin{itemize}
\item \texttt{calc} \quad 等式や不等式の変形を段階的に並べるブロック。
\item \texttt{rw [<- h]} \quad 等式 \texttt{h} を逆向きに使って書き換える。
\end{itemize}
\end{summarybox}
