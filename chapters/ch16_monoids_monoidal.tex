\chapter{モノイドとモノイダルな見方：結合できる処理}
\label{chap:monoids-monoidal}
\BookIndex{ものいど}{モノイド}
\BookIndex{ものいだるけん}{モノイダル圏}

\begin{chapterabstract}
本章では、モノイドを「空の値」と「結合できる操作」を持つ構造として導入します。
文字列結合、ログ集約、リスト連結、設定マージ、数値集計など、多くの処理を同じ形で考えられます。
ここでは、結合できることだけでなく、結合の括弧を変えても意味が変わらないことと、空の入力を自然に扱えることが重要です。
Lean 4 では、これらを等式として表し、結合律と単位元の法則が実務上どのような安全性を与えるかを確認します。
最後に、モノイダル圏を「逐次合成に加えて、対象と射をテンソルで並置する言語」として概観します。
\end{chapterabstract}

\begin{learninggoals}
\item モノイドを「結合演算」と「単位元」を持つ構造として説明できます。
\item 結合律と単位元が、ログ集約や設定マージの安全性にどう関係するかを説明できます。
\item Lean 4 で、リスト結合や小さな設定マージの法則を等式として書けます。
\item モノイダル圏を、テンソルによる並置や資源合成を考えるための発展的な見方として位置付けられます。
\end{learninggoals}

\section{繰り返し結合できる処理}

ログを集め、設定をマージし、複数のバッチ結果を一つにまとめます。
このような処理では、「二つを一つに結合する」操作が繰り返し現れます。
一見すると単純ですが、実務では次のような問いが生じます。

\begin{itemize}
\item 空のログを混ぜても結果は変わりませんか。
\item バッチ結果を分割して集約しても、最後の結果は同じですか。
\item 並列に処理した結果を後で結合しても安全ですか。
\item 設定マージの括弧を変えても、同じ設定になりますか。
\end{itemize}

図\ref{fig:ch16-monoid-overview}は、単位元と二項演算だけでなく、左右の単位律と結合律までがモノイドの構成要素であることを示します。

\FigurePlaceholder{fig-ch16-monoid-overview.png}{0.90}{モノイドの構成要素}{fig:ch16-monoid-overview}

前章までで、圏、同型、積と余積、関手、自然変換を見てきました。
それらは主に「型の間の変換」や「構造を保った変換」を扱うための言葉でした。
本章で扱うモノイドは、一つの型の中で値どうしを安全にまとめるための言葉です。

\section{モノイドは「空」と「結合」を持つ構造}

\subsection{まずはログで考える}

ログをリストとして表すとします。
一つの処理がログを返し、別の処理もログを返します。
二つの処理をつなげたら、二つのログを連結します。

ここで、ログを一つの型 \texttt{Log} として見ます。
空ログを \texttt{emptyLog}、結合を \texttt{combineLog} と呼びます。
この二つだけであれば、単なる便利関数です。
モノイドになるためには、法則も必要です。

型 \(M\) に次の三つがあるとき、\(M\) をモノイドとして扱えます。
\begin{itemize}
\item 単位元 \(e : M\)：これは「空」や「何もない値」を表します。
\item 結合演算 \(\diamond : M \to M \to M\)：これは二つの値を一つにまとめます。
\item 単位元の法則と結合律
\end{itemize}
法則は次の形で書けます。
\[
  e \diamond x = x, \qquad
  x \diamond e = x, \qquad
  (x \diamond y) \diamond z = x \diamond (y \diamond z).
\]

\subsection{モノイドは「便利な結合関数」ではない}

結合関数があるだけでは、まだモノイドとは呼べません。
たとえば、二つの設定ファイルをマージする関数を作ったとしても、その関数がどのような法則を満たすかを確認する必要があります。

「二つをまとめる関数がある」ことと、「モノイドとして安全に使える」ことは異なります。
モノイドとして使うには、少なくとも単位元と結合律を確認する必要があります。
特に設定マージでは、競合解決の仕様によって法則が壊れることがあります。

本章では、すべての現実的な設定マージがモノイドになるとは主張しません。
どの条件を満たせばモノイドとして扱えるかを明示するために、Lean で小さなモデルを作ります。

\section{文字列結合、ログ結合、設定のマージ}

ソフトウェアでよく出会うモノイド的な構造を整理します。
次の表では、「値の型」「空の値」「結合演算」「注意点」を並べています。

\begin{small}
\begin{longtable}{@{}p{0.17\linewidth}p{0.18\linewidth}p{0.21\linewidth}p{0.32\linewidth}@{}}
\toprule
例 & 空の値 & 結合演算 & 注意点 \\
\midrule
文字列 & 空文字列 & 文字列結合 & 順序は通常入れ替えられません。 \\
リスト & 空リスト & リスト連結 & 要素の順序を保ちます。 \\
ログ & 空ログ & ログ連結 & 並列集約では時刻や順序の扱いを設計します。 \\
自然数の合計 & \(0\) & 加算 & 順序を入れ替えてもよい例です。 \\
自然数の積 & \(1\) & 乗算 & 空の積を \(1\) と見られます。 \\
設定 & 空設定 & マージ & 競合解決が結合律を満たすか確認が必要です。 \\
\bottomrule
\end{longtable}
\end{small}

各処理が部分結果を返し、それらを同じ演算で結合できれば、分割処理やツリー状の集約を設計しやすくなります。
ただし、分割後の結果をどの順序で並べるかは、結合律とは別に決める必要があります。

\subsection{順序を変えてよいとは限らない}

モノイドが保証するのは、括弧の付け方を変えてよいことです。
つまり、次の二つは同じ結果になります。

\[
  (a \diamond b) \diamond c
  =
  a \diamond (b \diamond c)
\]

しかし、\(a \diamond b = b \diamond a\) は別の法則です。
これは可換性と呼ばれ、一般のモノイドには要求されません。

ログ連結では、\texttt{frontend ++ backend} と \texttt{backend ++ frontend} は通常同じではありません。
モノイドだからといって、並列処理の結果を任意の順序で混ぜてよいわけではありません。
順序を入れ替えるには、可換性やソートなど、別の仕様が必要です。

\section{結合律と単位元}

\subsection{結合律は括弧を忘れさせる}

三つのログ \texttt{a}, \texttt{b}, \texttt{c} を考えます。
まず \texttt{a} と \texttt{b} を結合してから \texttt{c} を結合する場合と、\texttt{b} と \texttt{c} を先に結合してから \texttt{a} を結合する場合があります。
結合律は、この二つが同じ結果になるという約束です。

図\ref{fig:ch16-associativity}の二つの木は、葉の順序 \(a,b,c\) を保ったまま、結合する組だけを変えています。
結合律は、この二つの評価結果が一致することを述べます。

\FigurePlaceholder{fig-ch16-associativity.png}{0.90}{結合律}{fig:ch16-associativity}

この法則があると、実装上の分割単位を変えやすくなります。
たとえば、次の二つの集約計画を同じ意味として扱えます。

\begin{itemize}
\item \texttt{((batch1 ++ batch2) ++ batch3)}
\item \texttt{(batch1 ++ (batch2 ++ batch3))}
\end{itemize}

これにより、バッチ処理の粒度やツリー集約の形を変更できます。
ただし、要素の順序そのものを変えているわけではありません。

\subsection{単位元は空ケースを扱いやすくする}

集約処理では、入力が空の場合の扱いを決める必要があります。
ログが一件もなければ空ログ、合計なら \(0\)、積なら \(1\) を返します。

単位元があると、空入力を例外扱いする必要がありません。
初期値として単位元を置き、あとは同じ結合演算を繰り返せます。

\section{Lean 4 で小さなモノイド法則を使う}

ここでは、Mathlib の高度な代数ライブラリへは進みません。
まず、標準的なリスト連結を使い、ログ結合の法則を等式として確認します。

Lean では、\texttt{[]} と \texttt{++} について左右の単位律と結合律を定理として書き、すべてのログに成り立つことを確認します。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter16

abbrev Log := List String

def emptyLog : Log := []

def combineLog (a b : Log) : Log :=
  a ++ b

#eval combineLog ["start"] ["end"]  -- => ["start", "end"]

theorem combineLog_empty_left (xs : Log) :
    combineLog emptyLog xs = xs := by
  rfl

theorem combineLog_empty_right (xs : Log) :
    combineLog xs emptyLog = xs := by
  simp [combineLog, emptyLog]

theorem combineLog_assoc (a b c : Log) :
    combineLog (combineLog a b) c = combineLog a (combineLog b c) := by
  simp [combineLog]
\end{minted}
\caption{\texttt{combineLog} のモノイド則}
\label{lst:ch16_log_laws}
\end{listing}

このコードでは、\texttt{combineLog} の実装だけでなく、\texttt{combineLog\_assoc} が仕様として残ることが重要です。
この定理があれば、ログ集約の括弧を変えるリファクタリングを、等式として説明できます。

\subsection{自作の小さなモノイド構造}

次に、「空」「結合」「法則」をまとめた小さな構造体を作ります。
これは、Mathlib の本格的な \texttt{Monoid} を置き換えるものではありません。
本章の目的に合わせた、学習用の最小モデルです。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure SimpleMonoid (M : Type) where
  empty : M
  append : M -> M -> M
  empty_left : forall x, append empty x = x
  empty_right : forall x, append x empty = x
  append_assoc : forall x y z,
    append (append x y) z = append x (append y z)

-- List Nat は、空リストとリスト連結でモノイドになる。
def listNatMonoid : SimpleMonoid (List Nat) where
  empty := []
  append := fun xs ys => xs ++ ys
  empty_left := by
    intro xs
    rfl
  empty_right := by
    intro xs
    simp
  append_assoc := by
    intro xs ys zs
    simp
\end{minted}
\caption{\texttt{SimpleMonoid}}
\label{lst:ch16_simple_monoid}
\end{listing}

ここでは、\texttt{SimpleMonoid} のフィールドに証明を含めています。
この設計は、「この型には結合関数があります」という宣言に加えて、「この型は指定した法則を満たします」という宣言になります。

\section{サンプル：ログ集約・設定マージの順序安全性}

\subsection{ログ集約では、分割の仕方を変えたい}

ログ収集では、全ログを一度に集められるとは限りません。
サービス単位、時間帯単位、リクエスト単位で分割されることがよくあります。
後でそれらをまとめるとき、結果が括弧の付け方に依存しないなら、集約計画を変更しやすくなります。

\begin{listing}[htbp]
\begin{minted}{lean4}
def planLeft (a b c : Log) : Log :=
  combineLog (combineLog a b) c

def planRight (a b c : Log) : Log :=
  combineLog a (combineLog b c)

theorem log_plan_safe (a b c : Log) :
    planLeft a b c = planRight a b c := by
  simpa [planLeft, planRight] using combineLog_assoc a b c
\end{minted}
\caption{\texttt{log\_plan\_safe}}
\label{lst:ch16_log_plan}
\end{listing}

ここで証明しているのは、ログの順序を入れ替えてよいという性質ではありません。
同じ順序のまま、括弧の付け方だけを変えてよいという性質を証明しています。
これが、モノイドの結合律が与える安全性です。

\subsection{設定マージは法則を確認してから使う}

設定マージも、モノイド的に扱えることがあります。
ただし、現実の設定マージは複雑になりがちです。
ここでは、説明のために小さな設定型を作ります。
設定には、追加リトライ回数とタグ一覧があるとします。
マージは、リトライ回数を足し、タグを連結する操作とします。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure Config where
  retry : Nat
  tags : List String
  deriving Repr

def emptyConfig : Config :=
  { retry := 0, tags := [] }

def mergeConfig (a b : Config) : Config :=
  { retry := a.retry + b.retry,
    tags := a.tags ++ b.tags }

#eval mergeConfig { retry := 1, tags := ["api"] }
                  { retry := 2, tags := ["db"] }
-- => { retry := 3, tags := ["api", "db"] }

theorem mergeConfig_empty_left (c : Config) :
    mergeConfig emptyConfig c = c := by
  cases c
  simp [mergeConfig, emptyConfig]

theorem mergeConfig_empty_right (c : Config) :
    mergeConfig c emptyConfig = c := by
  cases c
  simp [mergeConfig, emptyConfig]
\end{minted}
\caption{\texttt{Config} と \texttt{mergeConfig}}
\label{lst:ch16_config_merge}
\end{listing}

単位元の法則は、\texttt{emptyConfig} を前後どちらからマージしても元の設定が戻ることを述べています。
Lean では、構造体を \texttt{cases} で分解し、各フィールドの等式に落とします。
\texttt{retry} については \(0\) の加算、\texttt{tags} については空リストの連結を使います。

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem mergeConfig_assoc (a b c : Config) :
    mergeConfig (mergeConfig a b) c =
    mergeConfig a (mergeConfig b c) := by
  cases a
  cases b
  cases c
  simp [mergeConfig, Nat.add_assoc, List.append_assoc]

end Chapter16
\end{minted}
\caption{\texttt{mergeConfig\_assoc}}
\label{lst:ch16_config_assoc}
\end{listing}

この小さなモデルでは、設定マージが結合律を満たします。
そのため、\texttt{(base <> env) <> service} と \texttt{base <> (env <> service)} は同じ意味になります。

図\ref{fig:ch16-config-merge}では、空設定が左右の単位元となり、三つの設定の括弧だけを変えた二つの経路が同じ設定に到達します。
これは、フィールドごとの加算とリスト連結が、それぞれ対応する法則を満たすためです。

\FigurePlaceholder{fig-ch16-config-merge.png}{0.90}{設定マージの法則}{fig:ch16-config-merge}

\subsection{このモデルが証明していないこと}

この証明は、本番システムのすべての設定マージが安全だと主張するものではありません。
証明したのは、ここで定義した \texttt{Config} と \texttt{mergeConfig} に関する性質だけです。

本番の設定マージには、キーの競合、優先順位、環境変数の展開、機密情報の扱い、外部ファイルの読み込み順序などが関わります。
Lean の小さなモデルを使うときは、「どの仕様だけをモデル化したか」を必ず明記する必要があります。

\section{発展：モノイダル圏は「テンソルによる並置」の言語}

モノイダル圏は、圏 \(\mathcal C\) 上の二変数関手
\(\otimes : \mathcal C \times \mathcal C \to \mathcal C\)、単位対象、結合子、左右の単位子を持ち、それらが整合性条件を満たす構造です~\cite{riehl-category-theory-in-context}。
この定義だけから、実行時の並列性、独立性、交換可能性は導かれません。

ここまでは、一つの型の中で値を結合するモノイドを見てきました。
圏論では、さらに一般化して、対象や射を横に並べて合成する構造を考えます。
これがモノイダル圏の入口です。

通常の圏では、たとえば \texttt{A -> B} と \texttt{B -> C} を逐次合成します。
モノイダル圏では、これに加えて、二つの射をテンソルで並置できます。
型と関数を直積で組み合わせる場合は、\texttt{A -> B} と \texttt{C -> D} から
\texttt{(A, C) -> (B, D)} を作る操作が具体例です。

ソフトウェアでは、次のような場面をモデル化するときの構造上の見方として役立ちます。

\begin{itemize}
\item 二つの入力を別々に変換し、結果をペアとしてまとめます。
\item 独立性を別途確認した複数のサービス呼び出しの入出力を組にします。
\item 画像処理やETLで、別々の列やチャネルへの変換を一つの積上の関数として表します。
\item リソースや権限を合成し、全体として使える処理を作ります。
\end{itemize}

図\ref{fig:ch16-monoidal-category}は、出力を次の入力へ渡す逐次合成と、二つの射をテンソルで組み合わせる操作を別の方向で描いています。
後者の横向き配置は、同時実行を表す記号ではありません。

\FigurePlaceholder{fig-ch16-monoidal-category.png}{0.90}{逐次合成とテンソル}{fig:ch16-monoidal-category}

本章では、モノイダル圏の厳密な定義には進みません。
ここでは、圏の合成が「処理を縦につなぐ」言語であるのに対して、モノイダルな見方は「処理を横に並べる」言語を加える、と理解してください。
本格的には、対象どうしを並べるテンソル積、単位対象、結合の自然な同一視などを扱います。

「横に並べる」は図式上、構造上の表現であり、taskを同時実行するという意味ではありません。
実行時並列に読み替えるには、共有状態がないことや、失敗、取消、順序の観測が独立であることなど、追加の仕様が必要です。
また、一般のモノイダル圏は対称とは限らないため、左右を交換できるとも限りません。

\section{よくある誤解}

\subsection{モノイドなら順序を入れ替えてよいのか}

一般には入れ替えられません。
モノイドは結合律を要求しますが、可換性は要求しません。
文字列、リスト、ログの結合は、典型的に順序を入れ替えられません。
順序を安全に入れ替えるには、可換モノイド、ソート済み表現、集合化、重複の扱いなど、追加の仕様が必要です。

\subsection{単位元はただのデフォルト値か}

単なるデフォルト値ではありません。
単位元は、結合演算と一緒に使ったときに値を変えないという法則を満たす値です。
たとえば、空文字列は文字列結合の単位元ですが、\texttt{"default"} は通常は単位元ではありません。

\subsection{設定マージは常にモノイドになるのか}

常にモノイドになるわけではありません。
設定マージの仕様によって異なります。
右優先、左優先、優先度付き、時刻付き、エラー蓄積付きなど、設計によって法則は変わります。
実務では、「モノイドとして扱えるはず」と決め付けず、空の値と結合律を明示的に確認するのが安全です。

\section{本章のまとめ}

モノイドは、単位元と結合演算を持ち、単位元の法則と結合律を満たす構造です。

ログ、リスト、文字列、合計、設定マージなど、多くの実務上の集約処理をモノイド的に設計できます。

結合律は、括弧の付け方を変えてもよいことを保証しますが、順序を入れ替えてよいことは保証しません。

単位元は、空入力や初期値を自然に扱うための仕様です。

Lean 4 では、モノイド法則を等式として書き、ログ集約や設定マージの小さな安全性を確認できます。

モノイダル圏は、逐次合成に加えてテンソルによる並置を扱う発展的な言語として理解できます。
