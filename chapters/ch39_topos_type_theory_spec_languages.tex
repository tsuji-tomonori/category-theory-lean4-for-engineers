\begin{chapterabstract}
トポス、型理論、仕様言語を、データ、変換、論理を同じ世界で語るための三つの視点として眺めます。
型、関数、命題、証明、仕様 DSL、可換性、不変条件が、これらの視点でどのように関係するかを確認します。

トポスは、非常に大まかにいえば「集合のように振る舞う一方で、その内部で論理を語れる世界」です。
この見方は、Lean の型理論、仕様言語、モデル検査、ドメイン固有言語を考える際の背景になります。
Lean で小さな仕様 DSL を作り、構文と意味論を分けることで、「世界の内部で仕様を読む」という考え方を具体化します。
\end{chapterabstract}

\begin{learninggoals}
  \item トポスを「集合のように振る舞う世界」として直感的に説明できるようになります。
  \item 内部言語を「ある世界の中で使える論理」として読めるようになります。
  \item Lean の\texttt{Type}、\texttt{Prop}、証明が、型理論と仕様記述にどのように関係するかを再確認できます。
  \item 仕様言語や DSL を、構文と意味論の組として捉えられるようになります。
  \item トポス、型理論、仕様 DSL の関係を、適用範囲を区別しながら説明できるようになります。
\end{learninggoals}

\section{データ・変換・論理を同じ世界で扱う}

「型としての対象」「関数としての射」「命題としての仕様」「証明としての保証」を、一つの世界のデータ、変換、論理として整理します。
この整理により、Lean の型理論と仕様 DSL を、圏論的な構造との対応から読めるようにします。

本章では、トポスを「データ、変換、論理を同じ世界の中で語れる場所」として捉えます。
たとえば、ある仕様 DSL の中では、通常のプログラミング言語とは異なる形で「真」「かつ」「ならば」「存在する」を解釈する場合があります。
そのような「その世界の中での論理」を考える入口が、内部言語という考え方です。

\FigurePlaceholder{fig-ch39-bridge-map.png}{0.92}{理論間の関係}{fig:ch39-bridge-map}

図\ref{fig:ch39-bridge-map}は、「世界の中で論理を語る」という視点を中心に、圏論、Lean と型理論、仕様 DSL とモデル検査の位置関係を示しています。
これらを同一視せず、共有する概念と固有の役割を区別して示しています。

\section{トポスを「集合のように振る舞う世界」として眺める}

本章で念頭に置くのは、有限極限、指数対象、部分対象分類子を持つ elementary topos です。
Grothendieck topos や内部言語の完全な意味論を、同一の短い定義で置き換えるものではありません~\cite{maclane-moerdijk-sheaves}。

集合の世界では、値の集まりと、その間の関数を考えます。
たとえば、\texttt{User} という型をユーザーの集合のように見て、\texttt{User -> Bool} をユーザーに関する判定と見ることができます。
本書の前半では、この「型と関数の世界」を圏論の入口として使ってきました。

トポスは、この集合の世界に似た性質を持ちながら、より一般的な「世界」を扱うための概念です。
そこには対象と射があり、積や余積のような構成があり、さらにその世界の内部で命題や論理を語ることができます。

本章で使う表現として、トポスを次のように読みます。

\medskip

\emph{トポスとは、集合の圏に似た構造を持ち、その内部で論理を語れる圏です。}

\medskip

これは正式な定義ではありません。
正式には、有限極限、指数対象、部分対象分類子などを用いて定義します。
有限極限、指数対象、部分対象分類子という条件は、「データの世界」と「その世界の論理」を一緒に扱うための構造を与えます。

ソフトウェアで近い感覚を探すなら、「ある DSL の中でだけ意味を持つ式」を考えるとよいでしょう。
たとえば、アクセス制御 DSL には、\texttt{isAdmin}、\texttt{hasQuota}、\texttt{sameTenant} のような式があります。
これらは通常の\texttt{Bool}式にコンパイルされるかもしれませんが、DSL の利用者はまず「その DSL の世界の中で真かどうか」を考えます。

次の表は、集合、型、仕様 DSL を並べたものです。
厳密に同一視するのではなく、読み替えるための対応表として見てください。

\begin{center}
\begin{tabular}{@{}p{0.16\linewidth}p{0.18\linewidth}p{0.18\linewidth}p{0.38\linewidth}@{}}
\toprule
視点 & 値・対象 & 変換・射 & 論理の読み方 \\
\midrule
集合の世界 & 集合 & 関数 & 部分集合や述語として命題を読む \\
Leanの型理論 & \texttt{Type}の要素 & 関数 & \texttt{Prop}と証明として仕様を読む \\
仕様DSL & 状態空間 & 状態変換・評価関数 & DSL式を意味論で命題に解釈する \\

\bottomrule
\end{tabular}
\end{center}

\section{内部言語という考え方}

内部言語とは、ある圏や世界の外側から構造を説明するのではなく、その世界の中で使える言葉として論理を読む考え方です。

たとえば、マルチテナントの SaaS を考えます。
外部から見ると、システム全体には多数のテナント、権限、ユーザー、設定があります。
しかし、あるテナントの内部で仕様を書くときには、「現在のテナントで見えるユーザー」「現在の権限で可能な操作」「現在の設定で有効な機能」だけを使って論理を組み立てる場合があります。
これは文脈に依存する仕様を理解するためのソフトウェア上の比喩であり、テナント分離そのものがトポスの内部言語であるという主張ではありません。

内部言語は、「どの世界にいるつもりで話しているか」を明確にするための考え方です。
外部からは同じデータに見えても、テナント、時刻、権限、設定、抽象化レベルが変わると、語れる命題や成り立つ命題が変わる場合があります。

\FigurePlaceholder{fig-ch39-internal-language.png}{0.90}{内部言語と文脈}{fig:ch39-internal-language}

図\ref{fig:ch39-internal-language}は、外部から区別できる複数の文脈と、それぞれの内部で解釈される仕様との関係を表しています。
ここで重要なのは、「真偽は常に一つの固定された\texttt{Bool}だけで表される」と考えないことです。
Lean でも \texttt{Bool} は計算できる二値データであり、\texttt{Prop} は証明の対象となる命題であるため、両者は同じものではありません。
最終的な実装では\texttt{Bool}やエラー値として判定することが多い一方で、仕様設計では、どの文脈でどの命題を解釈しているかを分けておく必要があります。

内部言語という言葉を聞くと、「新しいプログラミング言語を作る」という話に見えるかもしれません。
しかし、本章で重要なのは、言語の実装ではなく、仕様をどの世界の中で読むかです。
たとえば、本番環境、検証モデル、抽象化された状態空間では、同じ日本語の仕様でも意味が少しずつ異なります。

\section{トポスと型理論}

Lean では、型、値、命題、証明が密接に結び付いています。
\texttt{Nat}や\texttt{String}のような型は、値の形を表します。
\texttt{Prop}は、証明の対象となる命題を表します。
また、ある命題の証明は、その命題を満たす構成物として扱われます。

この見方は、型理論の基本的な発想につながります。
型理論では、命題を型のように扱い、その命題の証明をその型の値のように扱います。
この対応は、Lean で仕様を書くときの実務感覚にも現れています。

次の Lean コードでは、アップロードの可否を\texttt{Prop}として定義します。
ここで\texttt{CanUpload u}は、\texttt{u}に関する仕様です。
証明\texttt{canUpload\_requires\_active}は、その仕様から「ユーザーが有効である」という性質を取り出しています。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter39

structure UserCtx where
  userId : Nat
  active : Bool
  quota : Nat
deriving Repr, DecidableEq

def CanUpload (u : UserCtx) : Prop :=
  And (u.active = true) (1 <= u.quota)

theorem canUpload_requires_active (u : UserCtx) :
    CanUpload u -> u.active = true := by
  intro h
  exact h.left
\end{minted}
\caption{\texttt{CanUpload}・\texttt{canUpload\_requires\_active}}
\label{lst:ch39_user_policy}
\end{listing}

この例は、トポス理論の形式化ではありません。
しかし、「文脈\texttt{UserCtx}の上で命題を読む」という練習にはなっています。
仕様は単なる文字列ではなく、ある状態や文脈の上で真偽を持つ命題として扱われます。

実務では、\texttt{CanUpload}のような述語が、設計文書に書かれた「有効なユーザーで、残量がある場合だけアップロードできる」という仕様の核に対応します。
この核を Lean で表すと、仕様から何を導けるかを機械的に確認できます。

\section{仕様言語・DSL・モデル検査との距離感}

仕様言語や DSL を設計するとき、重要なのは「構文」と「意味論」を分けることです。
構文とは、ユーザーが書ける式の形です。
意味論とは、その式が状態や文脈に対して何を意味するかを定めるものです。

たとえば、アクセス制御 DSL には、次のような構文があるかもしれません。

\begin{itemize}
  \item 常に真である式
  \item ユーザーが有効であることを表す式
  \item 残りクォータが一定以上あることを表す式
  \item 二つの式を「かつ」で結ぶ式
\end{itemize}

Lean では、これを小さな帰納型として定義できます。
その構文を\texttt{UserCtx -> Prop}へ解釈する関数を与えると、DSL の式に意味が与えられます。

\begin{listing}[htbp]
\begin{minted}{lean4}
inductive Spec where
  | truth : Spec
  | isActive : Spec
  | hasQuota : Nat -> Spec
  | and : Spec -> Spec -> Spec
deriving Repr, DecidableEq

def holds (s : Spec) (u : UserCtx) : Prop :=
  match s with
  | Spec.truth => True
  | Spec.isActive => u.active = true
  | Spec.hasQuota n => n <= u.quota
  | Spec.and p q => And (holds p u) (holds q u)

def uploadDSL : Spec :=
  Spec.and Spec.isActive (Spec.hasQuota 1)

theorem uploadDSL_correct (u : UserCtx) :
    holds uploadDSL u <-> CanUpload u := by
  rfl
\end{minted}
\caption{\texttt{Spec}・\texttt{holds}・\texttt{uploadDSL\_correct}}
\label{lst:ch39_spec_dsl}
\end{listing}

このコードでは、\texttt{Spec}が DSL の構文であり、\texttt{holds}が意味論です。
\texttt{uploadDSL}は DSL 上の式であり、\texttt{holds uploadDSL u}は、その式が文脈\texttt{u}で成り立つという Lean の命題です。
\texttt{uploadDSL\_correct} は、この解釈結果が前節の \texttt{CanUpload} と同値であることを確認し、DSL の構文と先に定めた仕様を接続しています。

DSL の式は、それだけではまだ仕様そのものではありません。
式に意味論を与えて初めて、「この状態で成り立つ」「この状態では成り立たない」といえます。
これは、「構文に意味論を与える」という一般的な練習です。
必要な圏論的意味論を構成していないため、トポスの内部言語のミニチュアまたは形式化とは呼びません。

モデル検査との距離も、ここで整理できます。
モデル検査は、有限個または抽象化された状態を網羅的に探索し、仕様が破れる状態がないかを調べる技術です。
一方、Lean による証明は、探索というよりも、命題に対する証明を構成する作業です。

\begin{center}
\begin{tabular}{@{}p{0.16\linewidth}p{0.38\linewidth}p{0.38\linewidth}@{}}
\toprule
道具 & 得意なこと & 注意点 \\
\midrule
Lean & 任意の入力に対する一般的な性質を証明する & モデル化と証明に人間の設計判断が必要 \\
仕様DSL & ドメインの人が読める形で制約を記述する & 構文だけでは不十分で、意味論が必要 \\
モデル検査 & 有限状態や抽象状態を網羅的に探索する & 状態空間の切り方や抽象化に依存する \\
型システム & 不正な形の値や組み合わせを事前に排除する & 業務仕様のすべてを型だけで表せるとは限らない \\
\bottomrule
\end{tabular}
\end{center}

\section{仕様言語を構文と意味論の組として見る}

前節の例をもう少し一般的に書くと、仕様言語は「状態の型」「式の型」「式を状態上の命題へ解釈する関数」の組として見ることができます。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure SpecLang where
  State : Type
  Formula : Type
  Semantics : Formula -> State -> Prop

def UserSpecLang : SpecLang where
  State := UserCtx
  Formula := Spec
  Semantics := holds

def refines {A : Type} (p q : A -> Prop) : Prop :=
  forall x, p x -> q x

theorem and_refines_left {A : Type} (p q : A -> Prop) :
    refines (fun x => And (p x) (q x)) p := by
  intro x h
  exact h.left
\end{minted}
\caption{\texttt{SpecLang}・\texttt{refines}・\texttt{and\_refines\_left}}
\label{lst:ch39_spec_lang}
\end{listing}

\texttt{SpecLang}は、非常に小さなモデルです。
それでも、重要な構図が含まれています。

\begin{itemize}
  \item \texttt{State}は、仕様を評価する対象となる世界です。
  \item \texttt{Formula}は、その世界について語るための式です。
  \item \texttt{Semantics}は、式を状態上の命題へ変換します。
\end{itemize}

\texttt{refines}は、本章の規約では「左側の強い仕様が右側の弱い仕様を含意する」という関係を表します。
文献やツールによっては、精緻化順序の向きを逆に定める場合もあるため、記号だけで向きを判断してはいけません。
たとえば、「有効で、かつ残量がある」という仕様は、「有効である」という仕様よりも強い仕様です。
このように、仕様間の関係を命題として扱えるようになると、仕様 DSL は単なる設定ファイルではなく、検査可能な言語に近付きます。

DSL を設計するときは、構文を先に増やしすぎないほうがよいでしょう。
まず、「その式はどの状態に対して何を意味するのか」を定めます。
意味論が決まれば、テストで確認する部分、Lean で証明する部分、モデル検査で探索する部分を分けやすくなります。

\section{実務への読み替え}

本章の考え方は、すぐに「トポスを使って設計する」という話ではありません。
実務への読み替えは、次のような問いとして現れます。

\begin{itemize}
  \item 現在書いている仕様は、どの状態空間の上で評価されるのでしょうか。
  \item 仕様の構文と意味論は分かれているでしょうか。
  \item その仕様は\texttt{Bool}で検査するのでしょうか、それとも\texttt{Prop}として証明するのでしょうか。
  \item すべての入力について証明したいのでしょうか、それとも有限状態を探索したいのでしょうか。
  \item 仕様を強める、弱める、抽象化するという関係を明示できているでしょうか。
\end{itemize}

トポス、型理論、仕様言語を結ぶ共通の視点は、「ある世界の中で、対象・変換・論理を一緒に扱う」という点にあります。
Lean では、その最小形として、型、関数、命題、証明を扱えます。
仕様 DSL では、構文と意味論を分けることにより、ドメインの言葉を検査可能な命題へ接続できます。

\section{本章のまとめ}

本章では、elementary topos を「集合の圏に似た構造を持ち、内部で論理を語れる圏」として眺めました。
Lean の\texttt{Type}、\texttt{Prop}、証明は、対象、命題、証拠を一つの言語で扱います。
仕様 DSL は、構文を\texttt{Semantics}によって状態上の命題へ解釈し、仕様間の精緻化を含意として比較できます。
トポス、型理論、仕様言語には、「ある世界の中で対象・変換・論理を一緒に扱う」という共通の視点があります。
