\chapter{米田の補題：インターフェースから値を見る}
\label{chap:yoneda-interface}

\begin{chapterabstract}
本章では、米田の補題を本格的な定理証明としてではなく、「値はそれに対するすべての観測方法から特徴付けられる」という直感として導入します。
対象の内部表現を直接のぞくのではなく、その対象をどのように使えるか、どのような関数で観測できるかに注目します。
この見方は、API 設計、抽象データ型、テストダブル、観測可能性、インターフェースの安定性を考えるうえで役立ちます。

圏論側では、集合の圏における小さな影として、
\[
  \operatorname{Nat}(\operatorname{Hom}(A,-), \mathsf{Id}) \cong A
\]
を読みます。
Lean 側では、Mathlib を使った完全な圏論の形式化には進まず、値を関数で観測する小さな例を扱います。
\end{chapterabstract}

\begin{learninggoals}
\item 米田の補題を「観測方法と値の対応」という直感で説明できるようになります。
\item 代表可能関手と自然変換が、ソフトウェア上のインターフェースにどのように似ているかを説明できるようになります。
\item Lean で「すべての観測が一致するなら値が一致する」という小さな影を表せるようになります。
\item API 設計、テストダブル、観測可能性に対して、米田的な見方を安全な比喩として使えるようになります。
\end{learninggoals}

\section{この章を読む理由}

実務では、値の内部表現を直接見られないことが多くあります。
ライブラリの利用者は、公開されたメソッドだけを使います。
API の利用者は、レスポンスの公開フィールドだけを見ます。
テストコードは、本物のデータベースではなく、インターフェース越しの振る舞いを見ます。
つまり、多くの場面で私たちは「値そのもの」ではなく、「値をどのように観測できるか」を扱っています。

\begin{intuitionbox}
米田の補題を本章の言葉で読むと、次のようになります。

あるものが何であるかは、そのものをあらゆる方法で使ったときに、どのように振る舞うかによって決まります。
内部表現を開いて見る代わりに、すべての観測窓から見える振る舞いを集めると、そのものを復元できます。
\end{intuitionbox}

この表現は、まだ厳密な米田の補題ではありません。
しかし、ソフトウェアエンジニアが最初に持つ直感としては十分に有用です。
特に、隠蔽された実装、公開インターフェース、adapter、テストダブル、観測ログを考えるときに、この見方は「何を同じと見なすのか」を整理する助けになります。

\FigurePlaceholder{fig-ch38-yoneda-overview.png}{0.90}{米田的な観測}{fig:ch38-yoneda-overview}

図\ref{fig:ch38-yoneda-overview}は、中央にある値を直接のぞくのではなく、複数の観測関数を通して外から見る状況を示しています。
観測方法が十分に豊富であれば、外部から見える振る舞いだけで値を区別できます。

\section{本章では直感だけを扱う}

米田の補題は、圏論の中心的な定理です。
正確に扱うには、圏、関手、自然変換、代表可能関手、自然同型を組み合わせる必要があります。
本書ではすでに関手と自然変換を扱いましたが、本章では完全な証明までは行いません。

\begin{warningbox}
本章の目的は、米田の補題を使って高度な圏論計算をすることではありません。
目的は、「対象を、その対象に関するすべての使い方から見る」という発想を得ることです。

Lean でも、Mathlib の \texttt{CategoryTheory} を使った本格的な米田埋め込みや米田補題の形式化には進みません。
それらは、第40章以降の発展課題です。
\end{warningbox}

この境界を明確にしておくことは重要です。
比喩は理解を助けますが、比喩だけで定理を証明したことにはなりません。
本章では、比喩と最小モデルを足場にして、後で標準的な圏論教材へ進むための入口を作ります。

\section{観測としての関数}

まず、集合と関数の世界で考えます。
型 \texttt{A} の値 \texttt{x : A} があるとします。
この値を観測するとは、何らかの関数
\[
  f : A \to B
\]
を適用し、結果 \texttt{f x : B} を見ることだと考えます。

たとえば、\texttt{User} 型の値があるとき、次のような観測が考えられます。

\begin{center}
\begin{tabular}{lll}
\toprule
観測関数 & 返すもの & 実務上の意味 \\
\midrule
\texttt{userId : User -> Nat} & ID & 同一性確認に使う \\
\texttt{displayName : User -> String} & 表示名 & UIに出す \\
\texttt{isActive : User -> Bool} & 有効状態 & 権限制御に使う \\
\texttt{toJson : User -> String} & JSON表現 & API境界で使う \\
\bottomrule
\end{tabular}
\end{center}

\begin{softwarebox}
インターフェースは、値の内部を直接公開するものではありません。
むしろ、「利用者がその値に対して何をできるか」を列挙したものです。
米田の補題が扱うのはすべての射と自然性であり、有限個の公開メソッドが常に内部値を決定するという定理ではありません。
実務では、選んだ観測の範囲で何を同じと見なすかを決めるための見方として使います。
\end{softwarebox}

ここで重要なのは、一つの観測だけでは足りない場合があることです。
ID だけが同じユーザーは、名前や権限が異なるかもしれません。
名前だけが同じユーザーは、別人かもしれません。
観測が少なすぎると、外部から区別できない値が残ります。

\section{すべての観測方法が値を決める}

観測方法を一つだけでなく、すべて考えるとどうなるでしょうか。
型 \texttt{A} の値 \texttt{x} に対して、任意の型 \texttt{B} と任意の関数 \texttt{f : A -> B} を選び、\texttt{f x} を見ます。

二つの値 \texttt{x y : A} について、すべての \texttt{B} とすべての \texttt{f : A -> B} で
\[
  f(x) = f(y)
\]
が成り立つなら、\texttt{x} と \texttt{y} は同じ値です。
これは、観測関数として恒等関数 \texttt{id : A -> A} も選べるためです。
恒等関数で観測すれば、\texttt{id x = id y}、すなわち \texttt{x = y} が得られます。

\begin{definitionbox}
本章でいう「すべての観測方法」とは、値 \texttt{x : A} に適用できるすべての関数 \texttt{f : A -> B} を考えるという意味です。

この考え方は、集合の圏 \(\mathsf{Set}\) における米田の補題の非常に小さな影です。
一般の圏では、\texttt{A -> B} の代わりに射 \(A \to B\) を考えます。
\end{definitionbox}

この事実だけを見ると、恒等関数を使った当然の話に見えます。
しかし、米田の補題の力は、この事実を一般の圏、一般の関手、自然変換の言葉で成立させる点にあります。

\section{米田の補題を一文で読む}

米田の補題の標準形と variance は、Hom 関手からの自然変換と対象上の要素との自然な対応として扱われます~\cite{riehl-category-theory-in-context}。
本章後半の observer の例は、その定理自体ではなく、値分離の小さな影です。

圏 \(\mathcal{C}\) の対象 \(A\) を固定します。
各対象 \(X\) に対して、\(A\) から \(X\) への射全体 \(\mathcal{C}(A,X)\) を対応させます。
これは、\(A\) から外へ向かう使い方を集める関手です。

この関手を、\(\mathcal{C}(A,-)\) と書きます。
\(-\) は、「ここに任意の対象を入れる」という意味です。

\begin{definitionbox}
\textbf{代表可能関手}：ある対象 \(A\) からの射全体、またはある対象 \(A\) への射全体によって作られる関手です。
本章では、\(A\) から各対象への射を集める \(\mathcal{C}(A,-)\) を使います。
\end{definitionbox}

米田の補題の共変版は、概略として次の形で読めます。
\[
  \operatorname{Nat}(\mathcal{C}(A,-), F) \cong F(A)
\]
左辺は、\(A\) からのすべての使い方を、関手 \(F\) が与えるデータへ一貫して変換する方法の集まりです。
右辺は、\(F\) を \(A\) に適用して得られるデータそのものです。

\begin{intuitionbox}
米田の補題は、「\(A\) から始まるすべての使い方を一貫して解釈する方法」が、「\(A\) における値」を一つ選ぶことと同じであると述べています。
\end{intuitionbox}

ここでは、「一貫して」という条件が重要です。
対象 \(X\) ごとに、ばらばらな対応を選ぶのではありません。
射をさらに合成したときにも、対応が合成と干渉しない必要があります。
この一貫性が自然性です。

\FigurePlaceholder{fig-ch38-natural-observation.png}{0.92}{観測の自然性}{fig:ch38-natural-observation}

図\ref{fig:ch38-natural-observation}では、観測関数 \(f : A \to B\) と、その後処理 \(g : B \to C\) を考えます。
先に \(f\) で観測してから \(g\) を適用することと、最初から \(g \circ f\) で観測することは同じです。
この基本的な可換性が、自然性の最小限の姿です。

\section{集合の圏での小さな影}

\(\mathsf{Set}\) で \(F\) を恒等関手 \(\mathsf{Id}\) とすると、米田の形は次のようになります。
\[
  \operatorname{Nat}(\operatorname{Hom}(A,-), \mathsf{Id}) \cong A
\]
これは、\(A\) の値 \(x\) と、「任意の関数 \(f : A \to B\) に対して \(f(x)\) を返す仕組み」が対応するという意味で読めます。

値 \(x : A\) があれば、任意の \(B\) と \(f : A \to B\) に対して、\(f(x)\) を返せます。
逆に、そのような仕組みがあれば、\(B = A\)、\(f = \operatorname{id}\) を選ぶことで、元の値を取り出せます。

\begin{softwarebox}
これは、抽象データ型のインターフェースに似ています。
値そのものを直接受け取らなくても、「どのような関数で観測しても、その結果を返すもの」があれば、少なくとも十分に豊かな世界では、その値を持っている場合と同じ情報を持っています。
\end{softwarebox}

ただし、この議論では「すべての観測方法」を考えている点に注意が必要です。
現実の API では、公開された観測方法は有限であり、権限、性能、互換性などの都合で制限されます。
そのため、現実のインターフェースだけで、内部値が常に完全に決まるとは限りません。

\section{Lean 4で観測の影を見る}

Lean では、値 \texttt{x : A} から「任意の型 \texttt{B} と関数 \texttt{A -> B} に対して \texttt{B} の値を返すもの」を定義できます。
これは、値を観測関数に渡すだけの小さな関数です。

\begin{leanbox}
次のコードでは、\texttt{observeAll x} が「任意の観測関数 \texttt{f} に対して \texttt{f x} を返すもの」を表します。
\texttt{same\_by\_all\_observers} は、すべての観測が一致するなら値が一致するという小さな事実を示します。
\end{leanbox}

\begin{listing}[htbp]
\begin{minted}{lean4}
def observeAll {A : Type} (x : A) :
    (B : Type) -> (A -> B) -> B :=
  fun _ f => f x

theorem recover_by_id {A : Type} (x : A) :
    observeAll x A (fun a => a) = x :=
  rfl

theorem same_by_all_observers {A : Type} {x y : A}
    (h : forall (B : Type) (f : A -> B), f x = f y) :
    x = y :=
  h A (fun a => a)
\end{minted}
\caption{\texttt{recover\_by\_id}・\texttt{same\_by\_all\_observers}}
\label{lst:ch38_all_observers}
\end{listing}

\texttt{recover\_by\_id} は、恒等関数で観測すれば元の値が返ることを示しています。
\texttt{same\_by\_all\_observers} は、すべての観測関数で結果が同じなら、恒等関数で見ても同じであるため、値そのものが等しいことを示しています。
この二つの定理から分かるのは、\texttt{x} を \texttt{observeAll x} へ送る対応が、値を区別できることまでです。
表示した多相関数型だけでは自然性をフィールドとして要求していないため、その型の任意の項が値から生じるという米田同型の逆向きまでは形式化していません。

次に、観測してから後処理する場合と、後処理を含む観測を一度に行う場合とで、結果が一致することを確認します。

\begin{listing}[htbp]
\begin{minted}{lean4}
def post {A B C : Type} (g : B -> C) (f : A -> B) :
    A -> C :=
  fun a => g (f a)

theorem equal_values_same_observations
    {A B : Type} {x y : A}
    (h : x = y) (f : A -> B) :
    f x = f y := by
  cases h
  rfl

theorem observation_commutes {A B C : Type} (x : A)
    (f : A -> B) (g : B -> C) :
    g (observeAll x B f) =
      observeAll x C (post g f) :=
  rfl
\end{minted}
\caption{\texttt{observation\_commutes}}
\label{lst:ch38_observation_commutes}
\end{listing}

このコードは、一般圏論の自然変換構造を定義しているわけではありません。
\texttt{observation\_commutes} は、固定した値 \texttt{x} から作った観測族について、先に \texttt{f} で観測して \texttt{g} で後処理する場合と、\texttt{g} と \texttt{f} の合成を一度に使う場合とで結果が一致するという、自然性の等式を確認しています。

\section{部分的なインターフェースでは足りない}

現実の API やクラスでは、すべての関数 \texttt{A -> B} を公開することはありません。
公開するのは、ごく一部のメソッドやフィールドです。
この場合、「公開インターフェースから区別できない」ことと「内部値が同じである」ことは一致しません。

\FigurePlaceholder{fig-ch38-partial-interface.png}{0.88}{部分インターフェース}{fig:ch38-partial-interface}

図\ref{fig:ch38-partial-interface}は、二つの値が同じ ID を返す一方で、名前は異なる状況を示しています。
ID だけを公開するインターフェースから見ると、二つの値は同じに見えます。
しかし、名前も観測できる場合は、両者を区別できます。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure User where
  id : Nat
  name : String

structure UserView where
  id : Nat
  name : String

def toView (u : User) : UserView :=
  { id := u.id, name := u.name }

theorem user_eq_from_view_eq {u v : User}
    (h : toView u = toView v) : u = v := by
  cases u
  cases v
  cases h
  rfl

def byId (u : User) : Nat := u.id

def sameById (u v : User) : Prop :=
  byId u = byId v

def userA : User := { id := 1, name := "Alice" }
def userB : User := { id := 1, name := "Alicia" }

example : sameById userA userB :=
  rfl
\end{minted}
\caption{\texttt{user\_eq\_from\_view\_eq}・\texttt{sameById}}
\label{lst:ch38_partial_observer}
\end{listing}

\texttt{toView} は、\texttt{id} と \texttt{name} の両方を公開するビューです。
このビューが等しいなら、今回の単純な \texttt{User} では、値そのものも等しくなります。
一方、\texttt{byId} だけを見る \texttt{sameById} では、\texttt{userA} と \texttt{userB} を区別できません。

\begin{warningbox}
「インターフェースから見て同じ」と述べるときは、どのインターフェースから見るのかを明示する必要があります。
観測方法が少なければ、異なる値が同じに見える場合があります。
これは抽象化として有用な場合もありますが、完全な同一性の証明ではありません。
\end{warningbox}

\section{API設計への読み替え}

米田的な見方は、API 設計で次の問いを立てる助けになります。

\begin{itemize}
\item 利用者は、この値をどの観測方法で見るのでしょうか。
\item その観測方法は、区別すべき値を本当に区別できるのでしょうか。
\item 反対に、区別する必要のない内部差分を隠せているのでしょうか。
\item adapter を通しても、観測結果は一貫しているのでしょうか。
\end{itemize}

たとえば、\texttt{User} の内部表現を変更しても、公開 API が返す \texttt{id}、\texttt{name}、\texttt{role} がすべて保存されるなら、利用者から見た互換性は保たれるかもしれません。
これは、「公開された観測方法に関して同値である」という主張です。

\begin{softwarebox}
実務では、完全な内部同一性よりも、公開インターフェースに関する観測同値が重要な場合が多くあります。
API 互換性、SDK adapter、テストダブルは、内部表現ではなく、公開された使い方の一貫性によって評価されます。
\end{softwarebox}

この見方は、第31章で扱った自然性にもつながります。
adapter が業務ロジックと干渉しないとは、観測や変換の順序を入れ替えても結果が一致するということです。

\section{テストダブルと観測可能性}

テストダブルは、本物の実装と同じ内部構造を持つ必要はありません。
必要なのは、仕様上必要な観測の範囲で同じ振る舞いをすることです。
既存のテストが偶然観測する有限個のケースで一致するだけでは、すべての利用文脈で置き換えられるとは証明できません。

たとえば、本物のメール送信サービスは、外部の SMTP サーバーに接続するかもしれません。
しかし、テストダブルは、送信された宛先と件名をメモリに記録するだけで十分です。
テストが使う観測方法が「何が送信されたか」だけなら、その観測に関して同じ振る舞いをすれば十分です。

\begin{intuitionbox}
米田的な視点では、テストダブルの正しさを「内部が本物と同じか」ではなく、「必要な観測方法から見て本物と区別できないか」として考えられます。
\end{intuitionbox}

観測可能性についても同様です。
ログ、メトリクス、トレースは、実行中のシステムを外部から見る観測関数のように振る舞います。
観測方法が不十分であれば、障害の原因を区別できません。
観測方法が適切であれば、内部状態を直接見なくても問題を推定できます。

\section{表現可能性への橋渡し}

本章では、代表可能関手を「ある対象からの使い方をすべて集めるもの」として紹介しました。
これは、圏論の重要な用語である表現可能性への入口です。

ある関手 \(F\) が代表可能であるとは、ある対象 \(A\) が存在し、\(F\) が \(\mathcal{C}(A,-)\) と同じ情報を持つということです。
ソフトウェアの観点では、「ある仕様や振る舞い全体が、一つの代表的なデータ構造からの使い方として説明できる」と読めます。

\begin{definitionbox}
\textbf{表現可能性}：抽象的に与えられた振る舞いや問い合わせの集まりを、ある具体的な対象からの射の集まりとして表せることです。
本章では詳細には踏み込みませんが、米田の補題は、この表現可能性を扱うための基本的な道具になります。
\end{definitionbox}

表現可能性を本格的に扱うと、普遍性、随伴、極限、データベース的な問い合わせ、型理論的な意味論へつながります。
これらの話題には、標準的な圏論教材や Mathlib の \texttt{CategoryTheory} を読む段階で再び出会います。

\section{発展への道}

この章の先には、二つの学習ルートがあります。

一つ目は、圏論側のルートです。
代表可能関手、自然同型、米田埋め込み、前層、Kan 拡張へ進みます。
米田の補題は、これらの話題の中心にあります。

二つ目は、Lean 側のルートです。
Mathlib の \texttt{CategoryTheory} では、圏、関手、自然変換、同型が一般的な形で定義されています。
本章の自作ミニモデルをその API へ移すには、universe、型クラス、記法、既存定理の検索に慣れる必要があります。

本書では、第40章で Mathlib の \texttt{CategoryTheory} へ進むための地図を示します。
本章の目的は、その前に「このような抽象的な定理が、なぜインターフェースや観測の話と関係するのか」を直感的に理解することです。

\section{本章のまとめ}

本章では、米田の補題を「値や対象は、それに対するすべての観測方法から特徴付けられる」という直感として読みました。

集合の圏では、値 \texttt{x : A} から「任意の \texttt{f : A -> B} に対して \texttt{f x} を返すもの」を作れます。

すべての観測が一致するなら、恒等関数で観測することにより、値そのものの一致が得られます。

現実の API やインターフェースでは、観測方法の一部しか公開されないため、観測同値と内部同一性を区別する必要があります。

テストダブル、観測可能性、API 互換性は、内部表現ではなく、公開された観測方法に関する同値として考えられます。

本章の Lean コードは米田の補題の完全な証明ではなく、発展的な Mathlib 圏論へ進むための最小モデルです。
