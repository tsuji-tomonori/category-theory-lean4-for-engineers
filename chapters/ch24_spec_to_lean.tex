\chapter{仕様書を Lean に移す方法}
\label{chap:spec-to-lean}

\begin{chapterabstract}
この章では、自然言語で書かれた仕様書から、Lean で扱える小さな検証モデルを作る手順を学ぶ。目的は、仕様書の全文を機械翻訳することではない。入力、出力、状態、操作、事前条件、事後条件、不変条件に分け、証明したい核だけを型・関数・述語・定理として切り出すことである。題材として、口座からの出金仕様を使う。本章で作るモデルは、次章の口座操作ケーススタディへの準備になる。
\end{chapterabstract}

\begin{learninggoals}
\item 自然言語仕様の曖昧な表現を、検査可能な部品へ分解できる。
\item 仕様の部品を、Lean の型・関数・述語・定理へ対応づけられる。
\item 事前条件、事後条件、不変条件の違いを説明できる。
\item 証明する性質と証明しない性質を、設計文書上で明示できる。
\item 口座出金仕様を、小さな Lean モデルとして表現できる。
\end{learninggoals}

\section{この章を読む理由}

実務の仕様書は、人間どうしの合意を作るために書かれる。したがって、読みやすさや業務文脈が重視される。一方で、Lean が扱えるのは、型、関数、命題、証明として明示されたものだけである。この差を埋めるには、仕様書をそのまま Lean に移そうとするのではなく、検査したい性質を小さく切り出す必要がある。

\begin{intuitionbox}
仕様書を Lean に移す作業は、自然言語を別の言語へ一文ずつ翻訳する作業ではない。むしろ、仕様書から「値の形」「処理」「守るべき条件」「証明したい主張」を取り出して、検査できる小さな模型を作る作業である。
\end{intuitionbox}

たとえば、仕様書に次のような文があったとする。

\begin{quote}
利用者は、残高の範囲内で口座から出金できる。残高を超える金額を出金しようとした場合は、エラーとして扱い、残高は変更しない。
\end{quote}

人間ならこの文をだいたい理解できる。しかし Lean には、ここに出てくる「利用者」「口座」「残高」「範囲内」「出金できる」「エラー」「変更しない」が何なのか分からない。そこで、まず語を分ける。

\begin{itemize}
\item 口座は、少なくとも残高を持つデータである。
\item 出金額は自然数として扱う。
\item 出金可能とは、出金額が残高以下であることとする。
\item 出金に成功したら、残高は出金額だけ減る。
\item 出金に失敗したら、状態は変わらない。
\end{itemize}

このように分けると、自然言語の仕様は少しずつ Lean の部品に近づく。

\FigurePlaceholder{fig-ch24-spec-to-lean-pipeline.png}{0.92}{自然言語仕様から Lean の型・関数・述語・定理へ分解する流れ}{fig:ch24-spec-to-lean-pipeline}

図\ref{fig:ch24-spec-to-lean-pipeline}は、この章で使う分解の流れを示す。左側には自然言語の仕様がある。中央では、仕様を入力、状態、操作、条件に分ける。右側では、それぞれを Lean の型、関数、述語、定理として表す。下には、あえて証明しない範囲を置く。ここを明示することで、「証明済み」という言葉の意味が過大にならない。

\section{自然言語仕様の危うさ}

自然言語仕様が危うい理由は、自然言語が悪いからではない。自然言語は、業務担当者、開発者、運用者、利用者のあいだで意味を共有するために必要である。ただし、機械的な検査には向いていない表現がある。

\begin{softwarebox}
仕様書で危険なのは、単に文章が長いことではない。危険なのは、重要な条件が文章の外側に隠れていること、成功時と失敗時の状態変化が混ざっていること、何を保存すべきかが明示されていないことである。
\end{softwarebox}

典型的には、次のような表現が曖昧さを生む。

\begin{itemize}
\item 「正常に処理する」：何をもって正常とするのか。
\item 「適切なエラーを返す」：エラーの型、値、状態変化は何か。
\item 「一貫性を保つ」：どの値どうしの関係を保つのか。
\item 「残高を更新する」：成功時だけか、失敗時にも何か記録するのか。
\item 「不正な入力を拒否する」：拒否した後の状態は変わるのか。
\end{itemize}

Lean に移すためには、これらの表現を、より小さな質問へ分解する。

\begin{definitionbox}
本章では、仕様を次の六つの部品に分ける。
\begin{enumerate}
\item 入力：操作に渡される値。
\item 出力：操作の戻り値。
\item 状態：操作の前後で変化しうる値。
\item 事前条件：操作を正しく呼ぶため、または成功するために必要な条件。
\item 事後条件：操作後に満たすべき条件。
\item 不変条件：操作の前後を通じて壊れてはいけない条件。
\end{enumerate}
\end{definitionbox}

ここで大切なのは、すべての仕様がきれいにこの六つへ分かれるとは限らないことである。むしろ、分けようとしたときに曖昧さが見つかる。これは失敗ではない。Lean による形式化の最初の価値は、証明を書く前に、仕様の穴を見つけることにある。

\section{仕様を型・関数・述語・定理に分ける}

Lean へ移すときは、まず文を「名詞」「動詞」「条件」「主張」に分けるとよい。名詞は型や構造体になりやすい。動詞は関数になりやすい。条件は述語になりやすい。主張は定理になりやすい。

\begin{center}
\begin{tabular}{p{0.22\linewidth}p{0.34\linewidth}p{0.34\linewidth}}
\toprule
仕様書の要素 & 例 & Lean での表現 \\
\midrule
名詞 & 口座、残高、出金額 & 型、構造体、フィールド \\
動詞 & 出金する、更新する、拒否する & 関数 \\
条件 & 残高の範囲内、残高を超える & 述語、命題 \\
主張 & 成功時は残高が減る、失敗時は変わらない & 定理 \\
境界 & 外部決済、監査ログ、UI表示 & 本章では証明しない前提 \\
\bottomrule
\end{tabular}
\end{center}

この表は、厳密な文法規則ではない。設計レビューで使うための作業表である。たとえば「残高を超える金額を出金しようとした場合はエラー」という文は、「条件」と「出力」と「状態保存」の三つを含んでいる。これを一つの文章のまま扱うと曖昧だが、三つに分ければ Lean で扱いやすい。

\begin{leanbox}
Lean では、データの形を \texttt{structure} で表し、操作を \texttt{def} で表す。条件は \texttt{Prop} を返す定義として書ける。証明したい仕様は、\texttt{theorem} として名前を付ける。
\end{leanbox}

まず、口座と出金操作の最小モデルを作る。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter18

-- 本章では、金額を Nat で単純化する。
-- 現実の通貨、小数、丸めは後で境界として明示する。
structure Account where
  balance : Nat
  deriving Repr, DecidableEq

-- 出金が成功するための事前条件。
def CanWithdraw (a : Account) (amount : Nat) : Prop :=
  amount <= a.balance

-- 成功すれば更新後の口座を返す。
-- 失敗すれば none を返す。
def withdraw (a : Account) (amount : Nat) : Option Account :=
  if amount <= a.balance then
    some { balance := a.balance - amount }
  else
    none

end Chapter18
\end{minted}
\caption{口座出金仕様の最小モデル}
\label{lst:ch24-account-model}
\end{listing}

この段階では、まだ「証明」はほとんどない。しかし、仕様はすでにかなり明確になっている。\texttt{Account} は残高だけを持つ。\texttt{CanWithdraw} は出金可能条件を表す。\texttt{withdraw} は成功か失敗を \texttt{Option Account} で返す。

ここで、設計判断も一つ入っている。失敗時に元の口座を返すのではなく、\texttt{none} を返している。これは、失敗を戻り値で明示する設計である。後で「失敗時は状態が変わらない」という性質を言いたい場合、\texttt{withdraw} だけでは状態を返していないので、別の wrapper 関数を定義する必要がある。このような設計判断が見えることも、形式化の価値である。

\section{事前条件・事後条件・不変条件}

仕様を Lean に移すとき、事前条件、事後条件、不変条件を混同しないことが重要である。三つは似ているが、読むタイミングが違う。

\begin{definitionbox}
事前条件は、操作の前に仮定する条件である。事後条件は、操作の後に期待する条件である。不変条件は、操作の前後を通じて維持したい条件である。
\end{definitionbox}

口座出金の例では、次のように読める。

\begin{itemize}
\item 事前条件：出金額が残高以下である。
\item 成功時の事後条件：更新後の残高は、元の残高から出金額を引いた値である。
\item 失敗時の事後条件：状態を変えない、または失敗として返す。
\item 不変条件：残高は負にならない。
\end{itemize}

ただし、本章のモデルでは残高を \texttt{Nat} で表している。\texttt{Nat} は自然数なので、負の残高は型の時点で表せない。つまり「残高は負にならない」という条件の一部は、述語ではなく型によって担保されている。

\begin{warningbox}
型で排除できる不正状態と、定理として証明する不正状態を混同しない。\texttt{Nat} を使えば負の値は表せない。しかし、現実の金額型に丸めや符号付き整数を使うなら、同じ主張は自動では成り立たない。Lean のモデルが何を単純化しているかを、必ず書く。
\end{warningbox}

成功時の事後条件を、Lean の述語として書いてみる。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter18

-- 成功時の事後条件。
def PostSuccess
    (before : Account) (amount : Nat) (after : Account) : Prop :=
  after.balance = before.balance - amount

-- 事前条件があれば、withdraw は成功する。
theorem withdraw_success_when_allowed
    (a : Account) (amount : Nat)
    (h : CanWithdraw a amount) :
    withdraw a amount = some { balance := a.balance - amount } := by
  have hle : amount <= a.balance := by
    simpa [CanWithdraw] using h
  simp [withdraw, hle]

-- 成功時の戻り値は、PostSuccess を満たす。
theorem withdraw_success_post
    (a : Account) (amount : Nat)
    (h : CanWithdraw a amount) :
    Exists (fun b =>
      withdraw a amount = some b /\ PostSuccess a amount b) := by
  exact Exists.intro { balance := a.balance - amount }
    (And.intro (withdraw_success_when_allowed a amount h) rfl)

end Chapter18
\end{minted}
\caption{事後条件と小さな証明}
\label{lst:ch24-postcondition}
\end{listing}

\begin{syntaxbox}
\texttt{have} は、証明の途中で補助的な事実に名前を付ける構文である。
\begin{itemize}
\item \texttt{have h : P := ...} は「ここで \texttt{P} の証明を \texttt{h} として持っておく」と読む。
\item 長い証明を、小さな中間結果に分けて読みやすくするために使う。
\item 後続の行では \texttt{h} を根拠として利用できる。
\item \texttt{¬ P} は「\texttt{P} ではない」という否定命題である。
\end{itemize}
\end{syntaxbox}


\texttt{withdraw\_success\_when\_allowed} は、「事前条件が満たされているなら成功する」という定理である。\texttt{withdraw\_success\_post} は、「成功した結果は事後条件を満たす」という形に言い換えている。実務の仕様書では、この二つが一文にまとめて書かれることが多い。Lean では、分けて書くほうが読みやすい。

\section{証明すべき性質を選ぶ}

形式化で失敗しやすいのは、最初から仕様書全体を Lean に移そうとすることである。実務上は、壊れると困る性質を選ぶべきである。特に、次のような性質は Lean に向いている。

\begin{itemize}
\item すべての入力について成り立つべき等式。
\item 操作後にも保存される不変条件。
\item 成功時と失敗時で分かれる事後条件。
\item リファクタリング前後の意味保存。
\item マイグレーションの round-trip 性質。
\end{itemize}

一方で、Lean に向いていない、または本章ではまだ扱わない性質もある。

\begin{itemize}
\item 外部 API が仕様通りに応答すること。
\item DB のトランザクション分離レベルが実運用で満たされること。
\item UI 上で利用者が誤操作しないこと。
\item 監査ログが規制要件を完全に満たすこと。
\item 実装コードと Lean モデルが常に同期していること。
\end{itemize}

これらを扱えないという意味ではない。扱うには、より大きなモデル、外部システムの仮定、実装との対応付けが必要になる。本章では、まず小さな核に絞る。

\begin{softwarebox}
証明対象を選ぶ基準は、「重要かつ小さい」である。重要だが巨大な仕様は、まず小さな命題に分割する。小さいが重要でない仕様は、Lean で証明してもレビュー価値が低い。
\end{softwarebox}

失敗時の仕様も、Lean で明示できる。出金できない場合は \texttt{none} を返す、という性質を定理にする。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter18

-- 事前条件が成り立たなければ、withdraw は失敗する。
theorem withdraw_failure_when_not_allowed
    (a : Account) (amount : Nat)
    (h : Not (CanWithdraw a amount)) :
    withdraw a amount = none := by
  have hnot : ¬ amount <= a.balance := by
    simpa [CanWithdraw] using h
  simp [withdraw, hnot]

-- amount が 0 なら、どの口座でも出金後に同じ口座が返る。
theorem withdraw_zero (a : Account) :
    withdraw a 0 = some a := by
  cases a
  simp [withdraw]

end Chapter18
\end{minted}
\caption{失敗時仕様の定理}
\label{lst:ch24-failure-spec}
\end{listing}

\texttt{withdraw\_failure\_when\_not\_allowed} は、失敗ケースを仕様から漏らさないための定理である。\texttt{withdraw\_zero} は小さな境界条件である。出金額が \texttt{0} の場合に結果がどうなるかを明示している。

\section{証明しない性質も明示する}

証明した性質だけを書くと、読者はそれ以上の保証があるように誤解することがある。したがって、Lean の章や設計文書では、証明しない性質を明示する必要がある。

\begin{warningbox}
「Lean で証明した」と言ってよいのは、Lean に書いた型、関数、仮定、定理の範囲だけである。モデルに含めていない外部システム、並行性、永続化、丸め、権限、監査要件は、自動的には保証されない。
\end{warningbox}

本章の口座モデルについて、証明範囲と非範囲を整理すると次のようになる。

\begin{center}
\begin{tabular}{p{0.42\linewidth}p{0.48\linewidth}}
\toprule
本章で扱う範囲 & 本章で扱わない範囲 \\
\midrule
残高を \texttt{Nat} とする単純モデル & 通貨単位、小数、丸め、桁あふれ \\
出金成功条件と戻り値 & 認証、認可、口座所有者の確認 \\
成功時・失敗時の戻り値仕様 & DB 更新、ロック、トランザクション \\
小さな事後条件の証明 & 監査ログ、通知、外部決済連携 \\
Lean モデル上の性質 & 本番実装との完全一致 \\
\bottomrule
\end{tabular}
\end{center}

この表は、弱気な注意書きではない。むしろ、Lean の証明を実務で使うための前提である。どこまで保証したのかが明確なら、レビューでも運用でも使いやすい。

\section{サンプル仕様：口座からの出金}

ここまでの内容を、仕様書の形から Lean の形へ通して見てみる。まず、自然言語仕様を少し整える。

\begin{quote}
口座は残高を持つ。出金要求は金額を持つ。出金額が残高以下であれば、出金は成功し、残高は出金額だけ減る。出金額が残高を超える場合、出金は失敗する。失敗した場合、状態は変更しない。本章では、残高と出金額を自然数として扱い、通貨単位、丸め、認可、DB永続化、監査ログは扱わない。
\end{quote}

この仕様は、すでに Lean へ移しやすい形になっている。名詞、操作、条件、成功時、失敗時、非範囲が分かれているからである。

\FigurePlaceholder{fig-ch24-withdraw-scope.png}{0.92}{口座出金仕様で証明する範囲と証明しない範囲}{fig:ch24-withdraw-scope}

図\ref{fig:ch24-withdraw-scope}は、出金仕様のスコープを示す。中央に Lean で扱う小さなモデルを置き、外側に本章では証明しない実務要素を置く。中央を小さく保つほど、定理は読みやすくなる。外側を無視するのではなく、境界として明示する。

失敗時に「状態は変更しない」と言いたい場合、\texttt{withdraw} の戻り値だけでは少し言いにくい。そこで、失敗時には元の口座を返す wrapper を定義する。

\begin{listing}[htbp]
\begin{minted}{lean4}
namespace Chapter18

-- 出金に失敗した場合は、元の口座をそのまま返す。
def withdrawOrKeep (a : Account) (amount : Nat) : Account :=
  match withdraw a amount with
  | some b => b
  | none => a

-- 失敗条件では、状態が変わらない。
theorem withdrawOrKeep_failure_keeps
    (a : Account) (amount : Nat)
    (h : Not (CanWithdraw a amount)) :
    withdrawOrKeep a amount = a := by
  have hnot : ¬ amount <= a.balance := by
    simpa [CanWithdraw] using h
  simp [withdrawOrKeep, withdraw, hnot]

-- 成功条件では、残高が amount だけ減る。
theorem withdrawOrKeep_success_balance
    (a : Account) (amount : Nat)
    (h : CanWithdraw a amount) :
    (withdrawOrKeep a amount).balance = a.balance - amount := by
  have hle : amount <= a.balance := by
    simpa [CanWithdraw] using h
  simp [withdrawOrKeep, withdraw, hle]

end Chapter18
\end{minted}
\caption{失敗時に状態を保つ wrapper}
\label{lst:ch24-withdraw-or-keep}
\end{listing}

ここで \texttt{withdraw} と \texttt{withdrawOrKeep} は役割が違う。\texttt{withdraw} は成功か失敗かを戻り値で表す。\texttt{withdrawOrKeep} は、失敗時の状態保存を表すための補助関数である。仕様書の同じ文章から、複数の Lean 定義が出てくることは珍しくない。

\section{仕様レビューで使う形式}

実務では、Lean コードだけを見せても、仕様レビューとしては不十分な場合が多い。自然言語、対応する Lean 定義、証明済みの定理、非対象範囲を対応させると読みやすくなる。

\begin{center}
\begin{tabular}{p{0.32\linewidth}p{0.28\linewidth}p{0.30\linewidth}}
\toprule
仕様項目 & Lean 側の名前 & レビューで確認すること \\
\midrule
口座は残高を持つ & \texttt{Account.balance} & 残高以外の状態を省いてよいか \\
出金可能条件 & \texttt{CanWithdraw} & 条件は \texttt{amount <= balance} で十分か \\
出金操作 & \texttt{withdraw} & 失敗を \texttt{none} で表してよいか \\
成功時の残高 & \texttt{withdraw\_success\_post} & 事後条件は業務仕様と合うか \\
失敗時の状態保存 & \texttt{withdrawOrKeep\_failure\_keeps} & 失敗時に副作用を持たないモデルでよいか \\
非対象範囲 & 境界表 & DB、認可、監査ログを別途検査するか \\
\bottomrule
\end{tabular}
\end{center}

この表は、Lean と仕様書をつなぐインデックスである。Lean の定理名を仕様項目に対応させておくと、仕様変更時にどの定理を見直すべきかが分かりやすい。

\begin{leanbox}
Lean の定理名は、仕様項目名に近づけるとよい。\texttt{theorem t1} のような名前では、証明が通っても設計文書との対応が分からない。\texttt{withdrawOrKeep\_failure\_keeps} のように、対象、条件、結論が読める名前にする。
\end{leanbox}

\section{圏論の言葉で見直す}

本章の主役は Lean への移し方だが、圏論の見方も背景にある。仕様を型・関数・述語・定理へ分けると、ソフトウェア変更を「構造を保つ変換」として読みやすくなる。

\begin{itemize}
\item 型は、扱う対象の世界を決める。
\item 関数は、対象から対象への変換を表す。
\item 述語は、対象や変換に課す制約を表す。
\item 定理は、変換が制約や意味を保存することを表す。
\end{itemize}

出金操作は、口座と金額から結果を作る変換である。成功条件のもとでは、残高に関する事後条件を保存する。失敗条件のもとでは、状態を保存する。ここで見ているのは、単なる計算結果ではなく、「どの性質が変換後にも残るか」である。この保存性の見方が、後のマイグレーション、関手、自然変換、Pullback の章へつながる。

\section{本章のまとめ}

自然言語仕様を Lean に移すとは、全文を機械翻訳することではなく、証明したい核を型・関数・述語・定理に分けることである。

仕様は、入力、出力、状態、操作、事前条件、事後条件、不変条件に分けると扱いやすい。

事前条件は操作前の仮定、事後条件は操作後の期待、不変条件は前後を通じて壊れてはいけない性質である。

証明対象は「重要かつ小さい」性質から選ぶ。仕様書全体を最初から形式化しようとしない。

証明しない性質を明示することで、Lean の証明を実務レビューで使いやすくできる。

口座出金の例では、\texttt{Account}、\texttt{CanWithdraw}、\texttt{withdraw}、\texttt{PostSuccess}、\texttt{withdrawOrKeep} を使って、成功時・失敗時の仕様を小さく表現した。

次章では、この章で作った考え方をさらに拡張し、入金、出金、送金を持つ口座操作の仕様証明を一つのケーススタディとして扱う。
