本章では、モナドを「文脈を持つ処理を合成する形」として導入します。
モナドを抽象的な数学用語として暗記するのではなく、失敗する処理、エラー理由を返す処理、状態を読み書きする処理、非同期に完了する処理を、同じ合成パターンで扱うための道具として読みます。

最後に、失敗しうるユーザー移行を題材に、\texttt{bind} の法則が effectful code のリファクタリング規則として働くことを見ます。
\end{chapterabstract}

\begin{learninggoals}
\item \texttt{Option}、\texttt{Except}、状態付き計算を「文脈を持つ値」として説明できます。
\item \texttt{pure} と \texttt{bind} の役割を、実務コードの処理連結として読めます。
\item モナド則を、抽象法則ではなく effectful code のリファクタリング規則として説明できます。
\item Lean 4 で \texttt{Option} の小さなモナド則を証明できます。
\item 失敗しうるマイグレーションを \texttt{bind} で安全に合成する最小モデルを書けます。
\end{learninggoals}

\section{文脈を保った処理の合成}

実務コードでは、通常の関数だけでは表しにくい処理が頻繁に現れます。
たとえば、入力が欠けていれば失敗するバリデーション、失敗理由を返すマイグレーション、内部状態を更新する処理、外部 API の応答を待つ非同期処理です。
これらは、見た目には別の処理です。
しかし、どれも「値だけでなく、値が置かれている文脈も一緒に運ぶ処理」として見ると、同じ形で合成できます。

図\ref{fig:ch17-overview}は、通常の値を文脈へ入れる \texttt{pure} と、文脈を保ったまま後続処理へつなぐ \texttt{bind} を中心に、失敗、エラー、状態の例を並べています。

\FigurePlaceholder{fig-ch17-overview.png}{0.90}{モナドの全体像}{fig:ch17-overview}

本章で扱うモナドは、Mathlib の高度な圏論 API としてのモナドではありません。
本章では、初学者がコードとして追えるように、\texttt{Option}、\texttt{Except}、状態付き計算の最小モデルだけを使います。
圏論上、モナドは自己関手と、単位および乗法という二つの自然変換を持ち、対応する単位律と結合律を満たす構造として定義されます。
本章では、それに対応する \texttt{pure} と \texttt{bind} の表示を使い、操作がすべての中身の型に一様に与えられることと、三つの法則に焦点を絞ります。

\section{\texttt{Option}、\texttt{Except}、\texttt{State} を同じ形で見る}

最初に、三つの典型例を並べます。
ここでは、細部ではなく、「値をそのまま返すのではなく、何らかの文脈に入れて返す」という共通点に注目します。

\begin{center}
\begin{tabular}{p{0.22\linewidth}p{0.30\linewidth}p{0.38\linewidth}}
\toprule
型の形 & 文脈 & 実務での読み方 \\
\midrule
\texttt{Option A} & 値があるかもしれない & 欠損、検索失敗、バリデーション失敗 \\
\texttt{Except E A} & 失敗理由 \texttt{E} を持つ & エラーコード、移行失敗理由、API エラー \\
\texttt{S -> (A, S)} & 状態 \texttt{S} を読み書きする & カウンタ、セッション、更新履歴、疑似的な状態遷移 \\
非同期結果 & 後で値が得られる & タスク、Future、Promise、外部 I/O \\
\bottomrule
\end{tabular}
\end{center}

本章では、このような文脈を抽象的に \texttt{M A} と書きます。
\texttt{A} は中身の型であり、\texttt{M} は文脈を表します。
たとえば、\texttt{M A} が \texttt{Option A} なら「失敗するかもしれない \texttt{A}」です。
\texttt{M A} が \texttt{Except E A} なら「エラー \texttt{E} を出すかもしれない \texttt{A}」です。

\section{\texttt{pure} と \texttt{bind}}

モナドの操作として、まず \texttt{pure} と \texttt{bind} を見ます。
\texttt{pure} は、通常の値を文脈の中に入れます。
\texttt{bind} は、文脈付きの値を取り出せた場合だけ、次の文脈付き処理へ渡します。

本章では、モナドを次の二つの操作を持つものとして扱います。
\begin{itemize}
\item \texttt{pure : A -> M A}：通常の値を文脈 \texttt{M} に入れます。
\item \texttt{bind : M A -> (A -> M B) -> M B}：文脈付きの値を、次の文脈付き処理へつなぎます。
\end{itemize}
これらをすべての型 \texttt{A}、\texttt{B} に一様に与え、後で述べる三つのモナド則を要求します。

\texttt{Option} の場合、\texttt{pure} は \texttt{some} で値を包みます。
\texttt{bind} は、入力が \texttt{none} なら次の処理を実行せず、\texttt{none} を返します。
入力が \texttt{some a} なら、\texttt{a} を次の処理へ渡します。

ここでは Lean 標準の \texttt{Option.bind} を直接使わず、動作を \texttt{match} で追える \texttt{optPure} と \texttt{optBind} を自作します。

\begin{listing}[htbp]
\begin{minted}{lean4}
def optPure {α : Type} (a : α) : Option α :=
  some a

def optBind {α β : Type} (x : Option α) (f : α → Option β) : Option β :=
  match x with
  | none => none
  | some a => f a

#eval optBind (some 10) (fun n => some (n + 1))  -- => some 11
#eval optBind (none : Option Nat) (fun n => some (n + 1))  -- => none
\end{minted}
\caption{\texttt{optPure} と \texttt{optBind}}
\label{lst:ch17_option_bind}
\end{listing}

コード \ref{lst:ch17_option_bind} の一つ目の \texttt{\#eval} は、\texttt{some 10} の中身を取り出して \texttt{some 11} を作ります。
二つ目は最初から \texttt{none} なので、後続の関数を実行せずに \texttt{none} になります。
この「失敗を自動的に伝播する」振る舞いが、バリデーションやマイグレーションで重要になります。

図\ref{fig:ch17-bind-flow}の成功経路では \texttt{some a} の \texttt{a} が後続関数へ渡り、失敗経路では \texttt{none} がそのまま返ります。

\FigurePlaceholder{fig-ch17-bind-flow.png}{0.90}{\texttt{Option.bind} の分岐}{fig:ch17-bind-flow}

\section{モナド則をリファクタリング規則として読む}

モナドには三つの基本法則があります。
数学上、これらは \texttt{pure} と \texttt{bind} が一貫して振る舞うことを表します。
ソフトウェア上は、effectful code のリファクタリング規則として読めます。
具体的には、不要な包みを消してもよいこと、何もしない後続処理を消してもよいこと、括弧の付け方を変えてもよいことを保証します。

\begin{center}
\small
\setlength{\tabcolsep}{3pt}
\begin{tabular}{@{}p{0.18\linewidth}p{0.34\linewidth}p{0.42\linewidth}@{}}
\toprule
法則 & 形 & リファクタリングとしての読み方 \\
\midrule
左単位律 & \texttt{bind (pure a) f = f a} & 値を一度包んですぐ渡すだけなら、直接渡せます。 \\
右単位律 & \texttt{bind x pure = x} & 最後に包み直すだけの処理は消せます。 \\
結合律 & \texttt{bind (bind x f) g = bind x (fun a => bind (f a) g)} & 処理列の括弧の付け方を変えても意味は変わりません。 \\
\bottomrule
\end{tabular}
\end{center}

モナド則は、実行順序の交換を許す法則ではありません。
図\ref{fig:ch17-monad-laws}が示すように、同じ順序の処理列から不要な \texttt{pure} を除いたり、\texttt{bind} の括弧を付け替えたりしても意味が変わらないという法則です。

\FigurePlaceholder{fig-ch17-monad-laws.png}{0.92}{三つのモナド則}{fig:ch17-monad-laws}

\section{Lean で \texttt{Option} の小さな性質を証明する}

\texttt{Option} の三つの法則は、場合分けで証明できます。
\texttt{x : Option A} には \texttt{none} と \texttt{some a} の二通りしかありません。
Lean の \texttt{cases} は、この二通りを漏れなく分解します。

左単位律は、定義を展開すると \texttt{rfl} で閉じます。
右単位律と結合律では、\texttt{x} を \texttt{none} と \texttt{some} に分け、\texttt{<;>} で両方のゴールへ \texttt{rfl} を適用します。

\begin{listing}[htbp]
\begin{minted}{lean4}
theorem option_left_identity {α β : Type} (a : α) (f : α → Option β) :
    optBind (optPure a) f = f a := by
  rfl

theorem option_right_identity {α : Type} (x : Option α) :
    optBind x optPure = x := by
  cases x <;> rfl

theorem option_assoc {α β γ : Type}
    (x : Option α) (f : α → Option β) (g : β → Option γ) :
    optBind (optBind x f) g =
      optBind x (fun a => optBind (f a) g) := by
  cases x <;> rfl
\end{minted}
\caption{\texttt{Option} の三法則}
\label{lst:ch17_option_laws}
\end{listing}

ここで証明しているのは、\texttt{Option} の中身に関する特別な性質ではありません。
どのような型 \texttt{A}、\texttt{B}、\texttt{C} であっても、\texttt{Option} の失敗伝播の形そのものがこの法則に従うという主張です。
これは、特定の入力例だけでなく、\texttt{Option} の全パターンについて確認している点で、テストとは異なります。

\section{サンプル：失敗するマイグレーションを安全に合成する}

次に、失敗しうるマイグレーションを考えます。
旧データ \texttt{UserDraft} には、メールアドレスと年齢がある場合と、ない場合があります。
新データ \texttt{UserV2} では、メールアドレスと年齢を必須にします。
この場合、移行は常に成功するとは限りません。

これは、旧データの欠損によって新スキーマへ移行できない状況の最小モデルです。
失敗可能性を返り値の \texttt{Option UserV2} に明示するため、後続処理も成功と失敗の両方を扱う必要があります。

\begin{listing}[htbp]
\begin{minted}{lean4}
structure UserDraft where
  id : Nat
  emailOpt : Option String
  ageOpt : Option Nat
  deriving Repr, DecidableEq

structure UserWithEmail where
  id : Nat
  email : String
  ageOpt : Option Nat
  deriving Repr, DecidableEq

structure UserV2 where
  id : Nat
  email : String
  age : Nat
  deriving Repr, DecidableEq
\end{minted}
\caption{\texttt{UserDraft} と \texttt{UserV2}}
\label{lst:ch17_user_model}
\end{listing}

移行を一度に書かず、二段階に分けます。
まずメールアドレスを必須化し、次に年齢を必須化します。
どちらも失敗する可能性があるため、返り値は \texttt{Option} になります。

\begin{listing}[htbp]
\begin{minted}{lean4}
def requireEmail (u : UserDraft) : Option UserWithEmail :=
  match u.emailOpt with
  | none => none
  | some email => some { id := u.id, email := email, ageOpt := u.ageOpt }

def requireAge (u : UserWithEmail) : Option UserV2 :=
  match u.ageOpt with
  | none => none
  | some age => some { id := u.id, email := u.email, age := age }

def migrateUser (u : UserDraft) : Option UserV2 :=
  optBind (requireEmail u) requireAge
\end{minted}
\caption{\texttt{migrateUser}}
\label{lst:ch17_user_migration}
\end{listing}

\texttt{requireEmail u} が \texttt{none} なら、\texttt{requireAge} は実行されません。
\texttt{requireEmail u} が \texttt{some withEmail} なら、\texttt{withEmail} が \texttt{requireAge} へ渡されます。
これにより、失敗の検査を各ステップに閉じ込めながら、全体の移行を一本の処理として組み立てられます。

\begin{listing}[htbp]
\begin{minted}{lean4}
def goodDraft : UserDraft :=
  { id := 7, emailOpt := some "a@example.com", ageOpt := some 20 }

def missingEmailDraft : UserDraft :=
  { id := 8, emailOpt := none, ageOpt := some 20 }

def missingAgeDraft : UserDraft :=
  { id := 9, emailOpt := some "b@example.com", ageOpt := none }

#eval migrateUser goodDraft  -- => some { id := 7, email := "a@example.com", age := 20 }
#eval migrateUser missingEmailDraft  -- => none
#eval migrateUser missingAgeDraft  -- => none
\end{minted}
\caption{\texttt{migrateUser} の実行例}
\label{lst:ch17_user_examples}
\end{listing}

この例では、成功時だけ \texttt{some UserV2} が得られます。
メールアドレスが欠けている場合も、年齢が欠けている場合も、結果は \texttt{none} になります。
\texttt{Option} には失敗理由が残りません。
失敗理由も必要であれば、\texttt{Except} を使います。

図\ref{fig:ch17-migration-pipeline}では、\texttt{requireEmail} が成功したときだけ \texttt{requireAge} へ進み、どちらかが失敗すると全体が \texttt{none} になります。

\FigurePlaceholder{fig-ch17-migration-pipeline.png}{0.92}{ユーザー移行パイプライン}{fig:ch17-migration-pipeline}

\subsection{do 記法は bind の読みやすい書き方}

Lean では、\texttt{Option} や \texttt{Except} のようなモナド的な型に対して \texttt{do} 記法を使えます。
これは、\texttt{bind} を読みやすく書くための構文です。
次の二つの移行関数は、同じ意味を持ちます。

\begin{listing}[htbp]
\begin{minted}{lean4}
def migrateUserDo (u : UserDraft) : Option UserV2 := do
  let withEmail ← requireEmail u
  requireAge withEmail

theorem migrateUser_refactor (u : UserDraft) :
    migrateUser u = migrateUserDo u := by
  cases u with
  | mk id emailOpt ageOpt =>
      cases emailOpt <;> cases ageOpt <;> rfl
\end{minted}
\caption{\texttt{migrateUser\_refactor}}
\label{lst:ch17_do_notation}
\end{listing}

\texttt{let withEmail ← requireEmail u} は、成功値を \texttt{withEmail} として後続へ渡します。
\texttt{Option} の \texttt{do} ブロックは、途中で \texttt{none} になると停止するため、この記述は上の \texttt{optBind} と同じ制御フローになります。
証明では、\texttt{UserDraft} の中身を分解し、\texttt{emailOpt} と \texttt{ageOpt} の有無を場合分けしています。
これは、「すべての欠損パターンについて、\texttt{bind} 版と \texttt{do} 記法版が同じ結果になる」と確認していることを意味します。

\section{\texttt{Except}：失敗理由を残す}

\texttt{Option} は、失敗したかどうかだけを表します。
一方、実務では失敗理由が必要になることがよくあります。
欠損メール、欠損年齢、フォーマット不正などを区別したい場合です。
その場合は \texttt{Except E A} を使います。

\texttt{Except E A} は、成功時に \texttt{A} の値を返し、失敗時に \texttt{E} 型のエラーを返す型です。
\texttt{Option A} が「失敗するかもしれない \texttt{A}」なら、\texttt{Except E A} は「理由 \texttt{E} 付きで失敗するかもしれない \texttt{A}」です。

\begin{listing}[htbp]
\begin{minted}{lean4}
inductive MigrationError where
  | missingEmail
  | missingAge
  deriving Repr, DecidableEq

def requireEmailE (u : UserDraft) : Except MigrationError UserWithEmail :=
  match u.emailOpt with
  | none => Except.error MigrationError.missingEmail
  | some email => Except.ok { id := u.id, email := email, ageOpt := u.ageOpt }

def requireAgeE (u : UserWithEmail) : Except MigrationError UserV2 :=
  match u.ageOpt with
  | none => Except.error MigrationError.missingAge
  | some age => Except.ok { id := u.id, email := u.email, age := age }
\end{minted}
\caption{\texttt{MigrationError} と移行処理}
\label{lst:ch17_except}
\end{listing}

\texttt{Except} でも、成功時だけ次へ進み、失敗時はそのエラーを保ったまま止まります。
\texttt{Option} と \texttt{Except} の違いは、失敗時に保持する情報量です。
合成の形は同じです。

\begin{listing}[htbp]
\begin{minted}{lean4}
def exceptPure {ε α : Type} (a : α) : Except ε α :=
  Except.ok a

def exceptBind {ε α β : Type}
    (x : Except ε α) (f : α → Except ε β) : Except ε β :=
  match x with
  | Except.error e => Except.error e
  | Except.ok a => f a

theorem except_left_identity {ε α β : Type}
    (a : α) (f : α → Except ε β) :
    exceptBind (exceptPure a) f = f a := by
  rfl

theorem except_right_identity {ε α : Type} (x : Except ε α) :
    exceptBind x exceptPure = x := by
  cases x <;> rfl

theorem except_assoc {ε α β γ : Type}
    (x : Except ε α) (f : α → Except ε β) (g : β → Except ε γ) :
    exceptBind (exceptBind x f) g =
      exceptBind x (fun a => exceptBind (f a) g) := by
  cases x <;> rfl
\end{minted}
\caption{\texttt{Except} の三法則}
\label{lst:ch17_except_laws}
\end{listing}

このコードでは、\texttt{Except.error} なら後続処理へ進まず、\texttt{Except.ok} なら中身を関数 \texttt{f} へ渡します。
\texttt{Option} と同じく、左単位律、右単位律、結合律の三つを確認できます。

\section{\texttt{State}：値と一緒に状態を渡す}

状態付き計算も、同じ発想で扱えます。
状態 \texttt{S} を受け取り、値 \texttt{A} と新しい状態 \texttt{S} を返す関数を考えます。
これは \texttt{S -> (A, S)} という形になります。

本章では、状態付き計算を \texttt{SimpleState S A = S -> (A, S)} として表します。
これは「状態 \texttt{S} を受け取り、値 \texttt{A} と更新後の状態 \texttt{S} を返す処理」です。

\begin{listing}[htbp]
\begin{minted}{lean4}
abbrev SimpleState (σ α : Type) := σ → (α × σ)

def statePure {σ α : Type} (a : α) : SimpleState σ α :=
  fun s => (a, s)

def stateBind {σ α β : Type}
    (m : SimpleState σ α) (f : α → SimpleState σ β) : SimpleState σ β :=
  fun s =>
    let r := m s
    f r.fst r.snd
\end{minted}
\caption{\texttt{SimpleState}}
\label{lst:ch17_state}
\end{listing}

\texttt{stateBind} は、まず計算 \texttt{m} を現在の状態 \texttt{s} で実行します。
その結果として、値 \texttt{r.fst} と新しい状態 \texttt{r.snd} が得られます。
次に、値を後続処理 \texttt{f} へ渡し、更新後の状態でその処理を実行します。

\begin{listing}[htbp]
\begin{minted}{lean4}
def getState : SimpleState Nat Nat :=
  fun s => (s, s)

def putState (s : Nat) : SimpleState Nat Unit :=
  fun _ => ((), s)

def tick : SimpleState Nat Nat :=
  stateBind getState (fun n =>
    stateBind (putState (n + 1)) (fun _ =>
      statePure n))

#eval tick 10  -- => (10, 11)
\end{minted}
\caption{\texttt{tick}}
\label{lst:ch17_state_example}
\end{listing}

この例では、\texttt{tick 10} は現在の状態 \texttt{10} を値として返し、状態を \texttt{11} に更新します。
状態付き処理では、順序が特に重要です。
ただし、同じ順序の処理列について、\texttt{bind} の括弧を整理できることはリファクタリング上の意味を持ちます。

\section{非同期処理をどう読むか}

非同期処理も、モナド的な合成として読めます。
たとえば、\texttt{Task A} や \texttt{Future A} のような型は、「今すぐではないが、将来 \texttt{A} が得られる計算」を表します。
\texttt{pure} は、すでに手元にある値を完了済みの非同期結果として包みます。
\texttt{bind} は、前の非同期結果が完了してから、その値を使う次の非同期処理を開始します。

本章では、実際のスケジューリング、キャンセル、タイムアウト、並行実行、外部 I/O の意味論までは形式化しません。
非同期を扱う現実のシステムでは、順序、失敗、再試行、リソース解放などが別途重要になります。
ここでは、非同期処理も「文脈を持つ値を次の処理へ渡す」という形で読める、という範囲にとどめます。

\section{実務上の読み替え：合成則をレビューの言葉にする}

モナドを実務に持ち込むとき、最初から「これはモナドです」と表現する必要はありません。
次の問いに答えられることが重要です。

\begin{itemize}
\item この処理の返り値には、どのような文脈が付いていますか。
\item 失敗、エラー、状態、非同期性を、返り値の型に明示していますか。
\item 次の処理へ進む条件は \texttt{bind} の形で一箇所に閉じ込められていますか。
\item 不要な \texttt{pure} や括弧を整理しても、意味が変わらない形になっていますか。
\item 失敗理由が必要なのに \texttt{Option} だけで潰していませんか。
\end{itemize}

\section{本章のまとめ}

\texttt{Option A}、\texttt{Except E A}、\texttt{S -> (A, S)} は、いずれも文脈を持つ値として読めます。

\texttt{pure} は通常の値を文脈に入れ、\texttt{bind} は文脈付きの値を次の文脈付き処理へつなぎます。

モナド則は、effectful code の左単位律、右単位律、結合律というリファクタリング規則として読めます。

Lean では、\texttt{Option} の \texttt{none} と \texttt{some} を場合分けすることで、モナド則を小さく証明できます。

失敗しうるマイグレーションでは、失敗可能性を返り値の型に入れ、\texttt{bind} で段階的に合成すると、失敗ケースを漏らしにくくなります。

失敗理由が必要な場合は、\texttt{Option} ではなく \texttt{Except} を使います。

状態付き処理は、値と更新後の状態を次へ渡す同じ合成形を持ちます。
非同期処理も同様に読めますが、本章のモデルはスケジューリング、取消、外部 I/O の意味論までは扱いません。
