この部では、データ、API、DB スキーマ、設定ファイルなどのマイグレーションを、圏論と Lean の視点から扱います。
マイグレーションは、ソフトウェア開発で頻繁に発生します。
フィールドを追加します。
名前を変えます。
複数のカラムをまとめます。
一つのフィールドを分割します。
レスポンス形式を変更します。
旧バージョンのデータを新バージョンで読み込みます。
これらはすべて、ある形式から別の形式への変換です。

しかし、マイグレーションには種類があります。
完全に情報を保つ変更もあれば、旧形式から新形式へは安全に変換できても戻せない変更もあります。
情報を落とす変更も、ある条件を満たすデータだけなら戻せる変更もあります。
これらを区別しないまま「互換性がある」と述べると、レビューやテストで重要な欠落を見逃しやすくなります。

圏論では、情報を失わない相互変換を同型として表します。
Lean では、その同型性を round-trip law として書けます。
\texttt{rollback (migrate x) = x}、または \texttt{migrate (rollback y) = y} という形の命題です。
片方しか証明できないなら、それは完全同型ではなく、片方向互換です。
条件付きでしか証明できないなら、well-formed 条件を明示する必要があります。

この部では、\texttt{UserV1} から \texttt{UserV2} への lossless migration、\texttt{fullName} を \texttt{firstName} と \texttt{lastName} に分割する lossy migration、\texttt{List.map migrate} によるバッチ移行、API response wrapper の変更、DB スキーマ変更後のクエリ結果保存を扱います。
各ケースでは、情報保存、片方向互換、仕様弱化のどの保証を選ぶべきかを明確にします。

この部の目的は、マイグレーションを「作業手順」ではなく「変換と保存性の問題」として捉え直すことです。
Lean で小さなモデルを作ることで、移行仕様の曖昧さを減らし、レビュー可能な証拠を残せます。

\section*{この部で扱うこと}

\begin{itemize}
\item 完全同型、片方向互換、lossy migration の分類
\item \texttt{migrate} と \texttt{rollback} の round-trip 証明
\item 条件付き同型と well-formed データ
\item \texttt{List.map} によるバッチ移行
\item 関手則による構造保存
\item API adapter と自然性
\item DB schema migration とクエリ保存
\item マイグレーション証明チェックリスト
\end{itemize}
