この部では、Leanを使って仕様書の一部を証明する方法を扱う。ここで重要なのは、自然言語で書かれた仕様をすべてLeanに翻訳しようとしないことである。実務の仕様書には、曖昧な表現、運用上の判断、外部システムへの依存、パフォーマンス上の制約、例外的な業務ルールなどが含まれる。これらを最初から完全に形式化しようとすると、すぐに作業量が膨らみ、実務上の価値が見えにくくなる。

本書では、仕様書の中から「小さく、重要で、壊れると困る性質」を切り出す。たとえば、口座残高が負にならないこと、送金前後で総残高が保存されること、割引計算後の金額が期待した範囲に収まること、特定のリファクタリングで計算結果が変わらないことなどである。このような性質は、Leanの型、関数、述語、定理として表現しやすい。

まず、自然言語仕様を分解する。入力、出力、状態、操作、事前条件、事後条件、不変条件に分ける。次に、証明したい性質を選ぶ。すべてを証明するのではなく、証明すべき核を選ぶことが重要である。証明対象が大きすぎると、Leanのコードも仕様書も読みにくくなる。逆に、証明対象を適切に絞ると、仕様の曖昧な部分が明確になり、設計レビューにも使いやすくなる。

この部の中心例は、口座操作と料金計算である。口座操作では、残高、不変条件、入金、出金、送金を扱う。料金計算では、税、割引、手数料、丸めなど、順序によって結果が変わる可能性のある処理を扱う。これらは実務に近く、かつLeanで小さなモデルを作りやすい。

この部を読み終えるころには、読者は仕様書の中から証明可能な性質を抽出し、Leanで小さな検証モデルを作る手順を理解しているはずである。また、Leanで証明したことと、実システム全体について保証できることの境界も意識できるようになる。

\section*{この部で扱うこと}

\begin{itemize}
\item 自然言語仕様の分解
\item 型、関数、述語、定理への変換
\item 事前条件、事後条件、不変条件
\item 口座操作の仕様証明
\item 料金計算の仕様証明
\item テストと証明の役割分担
\item 証明した範囲と証明していない範囲の明示
\end{itemize}

\section*{この部で扱わないこと}

\begin{itemize}
\item 本番コード全体の完全検証
\item 外部APIやDBの実挙動の証明
\item 並行処理や分散トランザクションの完全形式化
\item 仕様書の全自動Lean変換
\end{itemize}

\section*{次の部への接続}

次の部では、仕様証明の考え方を、データ構造やAPIの変更に応用する。特に、マイグレーション時に「新旧形式が同じ情報を表しているか」「戻せるか」「どの条件なら安全か」をLeanで表す方法を扱う。
